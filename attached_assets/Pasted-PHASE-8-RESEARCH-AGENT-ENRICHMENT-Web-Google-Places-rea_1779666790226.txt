PHASE 8 — RESEARCH AGENT ENRICHMENT
(Web + Google Places real impl + Modal integration + DQ recalc)

CONTEXT
Phase 8 replaces the orchestrator's web search and Google Places 
stubs with real implementations, producing narrative, amenities, 
photos, reviews, sentiment, and recent events for properties.

CRITICAL FRAMING: This is the input pipeline for Cashflow Agent 
accuracy. The Cashflow Agent depends on amenity flags for rent 
premium adjustments, reviews/sentiment for F52 buy-signal detection, 
photos for visual quality grading, narrative for strategy arbitrage 
context, and recent events for risk adjustments. Sloppy or 
hallucinated outputs degrade Cashflow Agent accuracy. The data 
quality bar is structural reliability, not just presence.

SCOPE LOCKED:
- Phase 8 output fields render on the PER-PROPERTY PAGE only 
  (/archive/properties/:parcelId). The Asset Detail Modal stays 
  the same in terms of displayed fields.
- DQ score recalculated to include Phase 8 fields as point-
  contributing. User clicks AUTO-ENRICH on modal, DQ moves up 
  even though new fields are not visible in the modal itself.
- AUTO-ENRICH button is hybrid: synchronous for fast enrichment 
  (<5s wall time), async with status badge for longer runs.
- All resolvable properties (not just Atlanta). Archive backfill 
  prioritized (296 properties = Cashflow Agent validation cohort).

PRINCIPLES
═══════════════════════════════════════════════════════════════════

1. STRUCTURED OUTPUT, NOT FREE TEXT. Amenity flags are booleans, 
   not narrative mentions. Reviews are extracted entities with 
   sentiment scores, not raw text dumps. Narrative is the only 
   free-form field, and it's bounded.

2. CITATIONS MANDATORY. Every claim in the narrative is traceable 
   to a source URL. If no source supports a claim, the claim is 
   omitted. The LLM does not invent.

3. EXTRACTION OVER GENERATION. Where possible, extract structured 
   facts (amenities from Places types, reviews verbatim from 
   Places API). Synthesis is reserved for narrative only.

4. STANDING PRINCIPLES (from prior dispatches):
   a. Producer/reader contract: any write to property_descriptions 
      must update every field readers display (modal + per-property 
      page + DQ score calc)
   b. Schema errors surface; no .catch(logger.warn) swallowing 
      DB writes. Use the error-handling policy from the recent 
      column-rename fixes
   c. Column-rename sweep: this dispatch adds new columns — verify 
      no stale references creep in
   d. Paired-read verification before declaring complete; UI 
      verification includes filter toggles and interactions, not 
      just "page loads"

5. INVESTIGATION FIRST. The current stubs return not_yet_implemented. 
   The shape of the modal and the per-property page's data fetch 
   paths must be confirmed before integration code is written.

STEP 3 — GOOGLE PLACES ADAPTER
═══════════════════════════════════════════════════════════════════

Location: src/services/research/google-places/

A. Call sequence per property:
   1. Find Place by name + address → place_id
   2. Place Details lookup → name, formatted_address, 
      formatted_phone_number, website, types, rating, 
      user_ratings_total, opening_hours, business_status
   3. Place Reviews (within Details) → up to 5 most relevant + 
      most recent reviews
   4. Place Photos → up to 5 photo references

B. STRUCTURED EXTRACTION
   - Amenity flags: parse the `types` array + photo captions for 
     known amenity signals. Maintain a mapping table:
     Places type → amenity boolean field
     (e.g., 'gym' → fitness, 'park' → outdoor_space, etc.)
   - Reviews: each extracted as { author, rating, text, time, 
     sentiment_score, named_entities, hazard_mentions, 
     amenity_mentions }
   - Sentiment summary: aggregate review sentiment, weighted 
     by recency

C. NLP PASS ON REVIEWS
   Second pass extracts:
   - Named entities (named amenities, neighborhood landmarks)
   - Hazard mentions (crime, noise, pests, parking, management 
     complaints)
   - Amenity gaps (complaints about missing features)
   These feed the F52 sentiment-rent divergence formula 
   downstream.

D. ERROR HANDLING
   - API errors (rate limit, quota exceeded) → fail explicitly, 
     record in enrichment_log, do NOT swallow
   - No match found → success with empty result + enrichment_log 
     entry noting 'no_places_match'
   - Place ID retrieval failure for downstream calls → partial 
     success, log which sub-call failed

STEP 4 — WEB SEARCH ADAPTER
═══════════════════════════════════════════════════════════════════

Location: src/services/research/web-search/

A. Call sequence per property:
   1. "<property_name> <city> <state>"
   2. "<property_address> apartment"
   3. "<property_name> management OR ownership"
   4. "<property_name> news OR announcement" (last 2 years)

B. RESULT EXTRACTION (structured)
   - Property website URL
   - ApartmentRatings / ApartmentList / similar aggregator URLs
   - News articles in last 2 years (title, URL, summary, date)
   - Recent events: renovations, ownership changes, capex 
     announcements
   - Citation set built from result URLs

C. NARRATIVE GENERATION (single LLM call, bounded)
   - Input: property name, address, structured facts from Places, 
     web search results
   - Output: 2-3 paragraph description with inline citations
   - Constraints:
     * Length cap: 300 words max
     * Every claim cites a source URL from search results
     * No claims without supporting source
     * Tone: factual, no marketing language
   - LLM instruction: "Do not invent. Cite or omit."

D. ERROR HANDLING — same discipline as Places adapter

STEP 7 — MODAL: AUTO-ENRICH BUTTON WIRING (HYBRID)
═══════════════════════════════════════════════════════════════════

A. Make AUTO-ENRICH button invoke the Research Agent for the 
   current property:

   Endpoint: POST /api/v1/properties/by-parcel/:parcelId/enrich
   
   Behavior:
   1. Validate: property has address + city + state (button is 
      disabled otherwise — already enforced in UI)
   2. Trigger Research Agent enrichment chain for this property
   3. Set a soft timeout: 5 seconds wall time
   4. If completes within 5s: return enriched result inline 
      (synchronous mode)
   5. If exceeds 5s: detach, continue async, return job_id with 
      status='enriching' (asynchronous mode)

B. UI behavior — synchronous mode (<5s):
   - Button shows spinner immediately on click
   - On completion: modal refreshes, DQ score updates, 
     success toast
   - User sees updated values in modal where applicable
   - Per-property page (when navigated to) shows full new data

C. UI behavior — asynchronous mode (>5s):
   - Button changes to "Enriching..." badge with subtle pulse
   - Modal remains usable; user can edit other fields, close 
     modal
   - Backend continues enrichment in background
   - When job completes: 
     * If modal still open: refresh values + DQ score, show 
       success toast
     * If modal closed: next time it opens for this property, 
       show enriched data + "Recently enriched" indicator
   - WebSocket or polling (whichever pattern the codebase uses) 
     for status updates

D. ERROR STATES
   - API key missing: button shows "Configure Places API" with 
     link to settings
   - Quota exceeded: error toast "Enrichment quota reached today"
   - Property not found in Places: toast "No Places match found" 
     + log; web search may still have populated narrative/events
   - Network failure: retryable error with "Try again" action

STEP 9 — REVIEW-ON-ADD WORKFLOW
═══════════════════════════════════════════════════════════════════

When a new asset is added to the Data Library (via Apartment 
Locator scrape, Data Library upload, or manual entry), the modal 
should open in "review mode" showing the auto-generated profile 
for user review before commit.

A. Trigger points:
   - Apartment Locator daily scrape completes → for each newly 
     added property, the orchestrator runs Research Agent → 
     property is flagged for review
   - Data Library upload via the upload UI → after files attached 
     and orchestrator processes them, property flagged for review
   - User clicks "Add Asset" manually → modal opens; on save, 
     orchestrator triggered → modal shows results for review

B. Review mode UI state:
   - Modal banner: "Profile auto-generated. Review and edit 
     before committing."
   - All fields shown editable
   - "Commit Profile" button (replaces standard Save)
   - "Reject and Re-enrich" button (alternate action — clears 
     auto-generated values and re-runs enrichment)

C. Commit flow:
   - User edits any field → those edits write to 
     property_descriptions.<field>.layers.manual_override
   - Untouched fields remain in their source layers (orchestrator-
     written)
   - On Commit Profile: property marked as user-reviewed 
     (intake_jobs status updates from a new 'awaiting_review' 
     state to 'complete')
   - DQ score recalculated post-commit

D. Reject flow:
   - User clicks "Reject and Re-enrich"
   - Clears layers.web, layers.places (NOT manual_override)
   - Resets intake_job status to 'pending'
   - Orchestrator picks up on next poll, re-runs enrichment
   - User notified when re-enrichment completes

E. Skip-review option for power users:
   - Setting in user preferences: "Auto-commit new profiles" 
     (default: false — review-on-add is the default)
   - When enabled: modal doesn't open automatically; profiles 
     commit as-is from orchestrator output
