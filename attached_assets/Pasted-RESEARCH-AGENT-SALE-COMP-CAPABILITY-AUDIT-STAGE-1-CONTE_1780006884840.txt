CONTEXT

(Comp-Anchored, Per-Unit, Sales Comp PPU) returning INSUFFICIENT

Cost is poisoning reconciliation as the only non-Cap×NOI active 
method. Operator commit: activate research agent to populate 
sale comp inventory from municipal sources (Pattern A + Pattern 
B), with CoStar gap-fill retained for coverage holes per P10.

This dispatch is an AUDIT, not a build. Output is a one-page 
reality assessment that scopes what Stage 2 (build) needs to do.

2. market_sale_comps table exists (per Query 1 schema diagnostic 
   confirmed empty)

═══════════════════════════════════════════════════════════════════
INSPECTION SCOPE
═══════════════════════════════════════════════════════════════════

A. TOOL REGISTRY INSPECTION
   Read research agent tool registry (src/agents/research/ or 
   equivalent). Enumerate tools and classify each:
   
   - Sale-comp-relevant: anything that pulls recorded transactions, 
     sale records, county recorder data, parcel transaction history
   - Subject-research only: parcel detail, zoning, ownership 
     lookup, valuation tax data
   - Other: ArcGIS, Municode, RentCast, etc.
   
   Report: which tools exist that COULD populate market_sale_comps, 
   and whether each currently writes to that table or only returns 
   for subject context.

B. COUNTY INTEGRATION COVERAGE
   For each tool flagged as sale-comp-relevant, enumerate:
   - Which counties/jurisdictions the tool currently supports
   - Whether the integration is live or stubbed
   - API/data source per county (county recorder, appraisal district, 
     state record system, etc.)
   
   Required coverage for backtest:
   - Florida: Hillsborough, Orange, Miami-Dade, Duval (primary 
     markets)
   - Georgia: Fulton, DeKalb, Cobb, Gwinnett (Atlanta backtest)
   - Texas: Dallas Central Appraisal District (Dallas market)
   
   Report: coverage matrix per county — LIVE / STUBBED / NOT 
   STARTED.

C. INVOCATION PATTERN
   Determine how/when sale-comp tools are currently invoked:
   - Subject-triggered only (Pattern A): tools fire when a deal 
     is created at a specific address
   - Market sweep (Pattern B): scheduled/cron job pulls inventory 
     for covered submarkets
   - Both
   - Neither (tools exist but aren't called)
   
   Report: current invocation pattern + frequency if scheduled.

D. WRITE PATH
   Determine where sale-comp data lands today:
   - market_sale_comps directly
   - recorded_transactions or another table (and whether anything 
     ETLs to market_sale_comps)
   - DealContext JSON only (returned for subject reasoning, never 
     persisted to comp inventory)
   - Other table
   
   Report: write path per tool, and whether the writes populate 
   market_sale_comps in a form the Valuation Grid can consume.

E. HISTORICAL BACKFILL CAPABILITY
   Determine whether tools support `as_of` / date-range parameters 
   for pulling historical transactions:
   - Can the tool pull sale records from 2018-2022 for the 
     Jacksonville and Atlanta backtest deals?
   - If not, what's the earliest date the data sources expose?
   
   Report: backfill feasibility for the S1 backtest deals.
