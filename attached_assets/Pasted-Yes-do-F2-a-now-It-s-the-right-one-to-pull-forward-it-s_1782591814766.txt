Yes — do F2(a) now. It's the right one to pull forward: it's an open privilege-escalation hole, the fix is independent of both product questions, and leaving a known security gap open while product decisions get made is the wrong tradeoff. But before you write it, three things about the fix design, because "validate req.body.tier against UCB" has a couple of forks of its own that decide whether the fix is correct or just narrower.
First — what should the validation do on mismatch: reject, or override? Two options with different blast radii:

Reject: if req.body.tier exceeds the user's UCB tier, 400 the request. Clean, but it assumes the client is supposed to send tier at all — and if any legitimate flow sends a tier the user is entitled to, you now have a client trusted to self-report a value you could just look up.
Override (better): ignore req.body.tier entirely and derive deals.tier server-side from UCB at creation, the same COALESCE(ucb.subscription_tier,'scout') resolution the rest of the repoint uses. The client doesn't get a vote. This is stronger because it closes the hole by construction rather than by check — there's no body value to validate because the body value isn't trusted at all. It also makes deals.tier consistent with the canonical tier source, which quietly helps the F2(b) problem later.

I'd push hard for override, not reject. The whole A9-Finding-1 lesson was "don't trust a second source of truth for tier" — and req.body.tier is the most untrusted possible source. Derive it. The only reason to keep accepting a body value is if there's a legitimate case where a user creates a deal at a tier different from their subscription — and I can't think of one, but you'd know. If that case doesn't exist, override is unambiguously right.
Second — this fix touches the F2(b) coupling, so be deliberate about scope. If you derive deals.tier from UCB at creation, then new deals will carry canonical vocab (scout/operator/principal/institutional) and never enterprise/pro. That's good — but it means F2(a)'s fix changes the write path that F2(b)'s 16 legacy deals came through. Don't let that tempt you into also fixing the 16 existing deals here — that's F2(b), it's coupled to F3, and it belongs in the coordinated pass. The clean line: F2(a) fixes the write path going forward (new deals derive from UCB); the 16 existing enterprise/pro deals stay untouched until the F3+F2(b) coordinated pass. Fixing them now, before F3 raises automation_level, would be reconciling data whose gate behavior you haven't settled yet. Stay in your lane: F2(a) is the write path, nothing else.
Third — verify it the way this session learned to verify. The fix is a write-path change, so the S1-01-grade proof isn't "the code validates" — it's: create a deal as a scout-UCB user with {tier:"institutional"} in the body, then query the row and confirm deals.tier came out scout (override) or the request 400'd (reject), not institutional. Observe the actual stored value, not the validation logic. That's the direct check that the hole is closed. One seeded user, one POST, one SELECT.
So: derive deals.tier from UCB server-side, ignore the body value, leave the 16 legacy deals for the coordinated pass, and prove it with a POST-then-SELECT showing the stored tier is the UCB value not the body value. If you confirm there's no legitimate "create a deal at a tier different from my subscription" flow, that's the whole fix.
On the two product questions — I won't answer them for you, they're genuinely yours, but here's the consideration that should weight each, briefly:
Q1 (enabled_modules): the deciding fact is whether agent/module selection (the §G ecosystem feature you described — users choosing which agents run) is meant to be entitlement-gated or just a UI preference. If selection is gated by what a user's plan includes, enabled_modules is the natural enforcement column and F1 is missing-enforcement (High, build the gate). If selection is "pick any of the agents you want, tier only limits credits/automation," then enabled_modules is vestigial and tier does all the gating — strip it. The question to ask yourself: is there any agent or module a paying user of one tier is supposed to be unable to select at all? If yes → enforce. If every tier can select everything and tier only governs credits/automation → delete.
Q2 (automation_level): the deciding fact is whether event-driven automation is part of what the operator+ tier means or a separate capability a user turns on. If "operator tier = your deals get auto-analyzed on creation" is the value proposition, auto-raise — a user who paid for it and has to find a toggle to get it is a support ticket and a churn risk. If event-driven runs have cost/noise implications a user should consent to (they consume credits automatically, which a user might not want firing on every deal), opt-in is safer. Given it's credit-billed, there's a real argument for opt-in — auto-firing paid agent runs on every deal creation without explicit consent could surprise users with credit burn. But that's exactly the product judgment that's yours, not mine.
