STEP 1 — TAX MODULE LOCATION + INVENTORY
A. Find the Tax module:
   - grep for 'tax module', 'property_tax', 'tax_calculator', 
     'reassessment', 'millage' across backend/src/
   - Identify the module's primary file path
B. Document the module's surface:
   - Inputs: what fields it consumes
   - Outputs: what fields it produces
   - Math: what calculations it performs (high-level)
   - Trigger: what causes it to fire
   - Storage: where its outputs land

STEP 2 — TRIGGER LOGIC VERIFICATION
A. Identify how the module is invoked
B. Specifically check: does it fire on sale events / acquisition?
   - Sale event detection logic (closing date, transaction 
     creation, deal status change)
   - Module invocation chain on sale
C. Identify gaps:
   - Sale events that should trigger reassessment but don't
   - Trigger conditions that fire incorrectly (false positives)
   - Edge cases (deal reopens, sale falls through, etc.)

STEP 3 — JURISDICTION COVERAGE AUDIT
A. List all jurisdictions the platform supports (FL counties, GA, 
   TX, NC, TN; check PROFORMA_CALCULATION_TEMPLATE.md and adapter 
   inventory)
B. For each jurisdiction, verify the module has:
   - Reassessment rule (when does property reassess after sale?)
   - Millage rate (current and historical)
   - Exemption handling (homestead, agricultural, etc., where 
     applicable)
   - Cap rules (Save Our Homes for FL, others)
C. Report coverage matrix:
   - GREEN: full math implemented
   - YELLOW: partial implementation (some rules missing)
   - ORANGE: stub/placeholder math
   - RED: no jurisdiction-specific math (uses generic fallback)

STEP 4 — PER-ASSUMPTION DERIVATION LOGIC
For each tax-related assumption (property tax annual, property 
tax growth, reassessment uplift), document:

A. How the Tax module produces the value
B. State-conditional logic (acquisition vs hold vs post-reassessment)
C. Confidence calibration
D. Operator override path
E. Reference to Tax module rather than re-specification of math 
   (per P9)

STEP 5 — AGENT PROMPT ALIGNMENT (per P9)
A. Locate agent prompts that reference property tax derivation
B. Audit: does the prompt accurately describe how Tax module 
   produces values?
C. Identify any drift between prompt and module behavior
D. Update prompts in same dispatch if drift found

STEP 6 — GAP REMEDIATION PLAN
For each gap identified:
A. Severity (BLOCKING / IMPORTANT / INFORMATIONAL)
B. Effort estimate
C. Sequencing recommendation (immediate, post-Batch 2, Phase 3)
