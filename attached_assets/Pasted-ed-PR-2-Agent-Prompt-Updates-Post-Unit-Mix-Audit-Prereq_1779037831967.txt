## What PR 2 does

Updates the Cash Flow Agent's prompt to use the new tool and fields PR 1 made available. Two prompt edits:

1. Update floor-plan grid sourcing instructions in `cashflow/system.ts` so the agent calls `fetch_unit_mix` for per-floor-plan data (not `fetch_rent_roll`, which returns property-wide aggregates only)
2. Update Other Income Method 3 prompt section to reference the now-hydrated `otherIncomeMonthly` field and add the fallback path for sparse data

Estimated effort: 30-60 minutes including verification.

## Edit 1 — Floor-plan grid sourcing instructions

**File:** `backend/src/agents/configs/cashflow/system.ts`
**Section:** Floor-plan grid sourcing (approximately line 938 per the audit reference)

### Find the existing text (likely contains something like)

```
For per-floor-plan unit data (current rents, unit counts, floor plan groupings),
call fetch_rent_roll.
```

### Replace with

```
**Per-floor-plan unit data sourcing:**

For per-floor-plan unit data (unit counts, current market rents, in-place rents,
floor plan groupings), call `fetch_unit_mix`. This is the canonical source —
reads from `deal_assumptions.unit_mix` with sponsor overrides applied via
`unit_mix_overrides`.

DO NOT call `fetch_rent_roll` for per-floor-plan data. `fetch_rent_roll` returns
property-wide aggregates only (total units, average rent, total monthly income,
lease expiration counts, market rent spread). It does NOT return floor-plan
breakdown. Calling `fetch_rent_roll` and expecting floor-plan detail will
produce undefined fields and smooth (incorrect) underwriting.

**Tool selection for floor-plan grid output:**

When producing `proforma.revenue.gpr.unit_mix[]`, assemble each `UnitMixEntry`
from multiple tool outputs:

- `fetch_unit_mix` → unit_count, floor_plan_label, current_market_rent (per floor plan)
- `fetch_peer_comp_noi_metrics` + `fetch_data_library_comps` → comp_ceiling P25/P50/P75 per floor plan
- `fetch_owned_asset_actuals` → capture_rate baseline (if available, otherwise platform default 0.78)
- `fetch_data_matrix.extractedData.capex.capex_schedule` → renovation_cost per floor plan and renovation_scope

Combine these into the per-floor-plan grid output per the canonical UnitMixEntry shape.

**Sponsor override behavior:**

When sponsor edits the Unit Mix tab (PATCH /api/v1/deals/:dealId/financials/override),
the overrides land in `deal_assumptions.unit_mix_overrides`. The next call to
`fetch_unit_mix` returns the overridden values automatically. You do not need to
read overrides separately — they are pre-merged by the tool.
```

## Edit 2 — Other Income Method 3 prompt section

**File:** `backend/src/agents/configs/cashflow/system.ts`
**Section:** Other Income reasoning method (Section A.5 if structured per v3 prompt patch)

### Find the existing text (likely contains something like)

```
Method 3 (Rent Roll + Fee Schedule + T-12 Reconciliation):
Establish current per-category state from rent roll detail.
```

### Replace with

```
**Method 3 — Rent Roll + Fee Schedule + T-12 Reconciliation:**

Establish current per-category state from rent roll detail.

**Source for per-category rent roll detail:** Call `fetch_data_matrix` and read
`extractedData.rentRoll.otherIncomeMonthly` for per-category ancillary income
(parking, pet, storage, washer/dryer, RUBS, fees, insurance admin, cable, other).
This field returns per-category monthly dollars when the source rent roll
included per-category detail.

**Fallback when otherIncomeMonthly is sparse or null:**

If `otherIncomeMonthly` returns fewer than 3 categories itemized, OR is null
entirely, apply this hybrid logic:

1. For categories present in `otherIncomeMonthly`: standard Method 3 logic
   (current state × sponsor fee plan × adoption)

2. For categories not in `otherIncomeMonthly` but present in archive cohort
   norm: source per-category values from CIE findings (when CIE is available)
   or from `fetch_archive_assumption_distribution` filtered to comparable
   class/vintage/submarket deals. Each cohort-sourced category carries
   `source: 'archive_cohort'` in its evidence narrative.

3. For new categories the sponsor is introducing (not in rent roll, not in
   cohort): Method 1 logic (fee schedule × applicable units × adoption rate
   from owned-portfolio or archive). Each carries
   `source: 'fee_schedule_projection'` in evidence narrative.

The output remains a per-category breakdown. Each category's evidence
narrative names its source. The math engine's hierarchical handling reconciles
the aggregate against T-12 unchanged.

**Critical unit convention:**

All numeric values for Other Income are total annual dollars, NEVER
per-unit-per-month. Emit `proforma.revenue.other_income.parking` as $8,648
annual (not $37 per unit per month). The platform's math engine and write-back
expect annual dollars across the Other Income field family.

This convention applies to:
- `proforma.revenue.other_income` (aggregate)
- `proforma.revenue.other_income.parking`, `.pet`, `.storage`, etc. (per category)
- All three methods (Method 1, Method 2, Method 3)
```
