What's genuinely missing
1. intake_jobs schema gaps

Current columns: id, file_id, parcel_id, state, block_reason, user_input, enrichment_log, created_at, updated_at, source_data, source_type

Missing vs. dispatch requirements:

Column	Dispatch requirement	Impact
raw_input JSONB	Store original scrape record verbatim	Audit/reprocessing impossible without it
source_record_id text	Dedup key: (source_type, source_record_id)	Currently source_data.apartment_locator_id has this but no unique index guards against duplicates
attempts integer	Max 3 retries on failed status	Currently jobs fail with no retry
ignored state	Manual mark-as-ignore flow	State machine incomplete per Principle 3
2. property_descriptions write-back (Principle 2 / Step 5) — missing entirely

The orchestrator logs municipal data to enrichment_log but never writes to property_descriptions. When a job resolves complete, the parcel ID is updated on intake_jobs but the resolved owner, assessed value, land acres, county, etc. are trapped in the JSON log — not surfaced to any property page.

property_descriptions currently has 296 rows populated through a different path. The paired-read verification described in Step 5 would fail today: GET /api/v1/properties/:parcelId/summary cannot show orchestrator enrichment results because the write never happens.

3. Apartment Locator source adapter as a proper routing layer

The intake_jobs rows exist and have source_type='apartment_locator', but there's no src/services/intake-sources/apartment-locator/ directory. The dispatch wants a formal adapter class that:

Accepts a scrape record and normalizes it to an IntakeJob payload
Guards idempotency via (source_type, source_record_id) unique index
Stores raw_input verbatim
Currently it appears jobs were bulk-created via a migration or script rather than through a typed adapter. This means new Apartment Locator scrape records have no clear path into intake_jobs — sync-table.service.ts only pumps data into apartment_locator_properties, not intake_jobs.

4. Max retry logic — not implemented

processJob catches errors and sets state to failed but there's no attempts counter and no retry loop. A job that hits a transient DB failure goes permanently to failed with no recovery.

5. Closing doc — not written

docs/operations/PHASE_1_1_1_2_CLOSING.md doesn't exist.
