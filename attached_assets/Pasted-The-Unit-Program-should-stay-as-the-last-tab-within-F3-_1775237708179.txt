The Unit Program should stay as the last tab within F3 — not a separate page. Here's why that's the right position and what it means for the workflow:
The narrative arc of F3's tabs matters. The tab order tells a story: Demand ("is there demand, and for what type?") → Comps ("what's the competitive landscape?") → Trends ("where is it heading?") → Program ("so here's what we should build"). Each tab builds the case for the next. By the time the user reaches Program, they've absorbed the market evidence that justifies the mix decisions. If you pull Program out to its own page, you break that narrative flow and force the user to mentally carry context across navigation boundaries.
The convergence problem is real but solvable without a new page. The Unit Program needs data from two independent chains — the market intelligence chain (F6→F3) and the physical constraint chain (F5 Zoning→Dev Capacity→M03 Massing). The solution is to surface the zoning envelope within the Program tab as a constraint strip (which is what the redesign I just built does — the unit and SF envelope gauges at the top). The user doesn't need to flip to F5 to see their ceiling; it's embedded in the builder. The zoning data flows via dealStore the same way comp data flows.
For the deal type question specifically:
For development deals, the user workflow is: set up the deal → run zoning (F5, which populates dev capacity) → discover comps (F6) → analyze market (F3 Demand/Comps/Trends) → build unit program (F3 Program tab, constrained by zoning envelope). The Program tab is where the two chains meet. The AI Optimize button should take both chains as input — "given these zoning constraints and this market data, here's the optimal mix."
For redevelopment deals, the existing unit mix auto-populates from the Deal Capsule. The Program tab shows that baseline with an editable delta layer on top. The question it answers shifts from "what should we build?" to "what should we add or convert?" The zoning envelope still matters because adding units or SF requires checking remaining capacity. The three modes you identified — add units, add SF, or both — should be a toggle within the Program tab, not separate views.
For existing deals, the Program tab becomes read-only with a gap analysis overlay: "here's your actual mix vs. what the market data suggests is optimal." No builder, no zoning constraints. It answers "how well does our existing product fit this market?" and feeds the as-is mix straight to ProForma.
The one thing the Replit agent missed: Their analysis focuses on the F6→F3 data pipeline (comp set curation → comp analysis), which is correct. But they don't address the fact that the Program tab's mock data problem is worse than the Comps tab's mock data problem, because Program doesn't just need comp data — it needs the synthesized outputs of all three F3 tabs plus zoning. Wiring F3 to consume F6's comp set (their recommendation) is necessary but not sufficient. The Program tab also needs the zoning envelope to flow in from zoningModuleStore, and it needs F3's own Demand/Comps/Trends analysis outputs as typed inputs, not just proximity on the same page.
