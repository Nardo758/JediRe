BACKEND

  own the parent deal OR be on the workspace for the parent submarket.

  Storage cleanup queued separately, not in this PR.
- If submarketDocuments.routes.ts missing: create it. Path prefix

  delete. Reuse the same service layer as deal documents — pass scope discriminator

- If schema needs submarket_id: write the next migration in
  jedire/backend/migrations/. Nullable column (existing rows have no submarket).
  Index it.

FRONTEND
- Add typed methods to api.client.ts:
    api.deals.files.download(fileId): Promise<{url: string}>
    api.deals.files.delete(fileId): Promise<void>
    api.submarkets.documents.list(submarketId)
    api.submarkets.documents.upload(submarketId, file)
    api.submarkets.documents.download(fileId)
    api.submarkets.documents.delete(fileId)
- Extract shared <DocumentList> component at
  frontend/src/components/shared/DocumentList.tsx. Props:
    { files, onDownload(fileId), onDelete(fileId), scope: 'deal'|'submarket', isLoading }
  Both FilesSection.tsx and the Market Documents component consume it.
  Identical row UI — no two copies of the row markup.
- Row buttons (Bloomberg dark, JetBrains Mono, T tokens):
    Download icon button — T.text.cyan, hover T.text.cyanBright
    Delete icon button — T.text.red, hover T.text.redBright
  9px font minimum. No shadows. 2px max radius. Icons inline-aligned right.
- Download handler: call API → receive { url } → window.open(url, '_blank').
  Don't fetch bytes through the SPA.
- Delete handler:
    1. Confirm dialog: "Delete {filename}? This cannot be undone."
       Use existing confirm pattern in the codebase; don't introduce a new one.
    2. Optimistic remove from local state
    3. Call API
    4. On success: refetch list to confirm canonical state
    5. On failure: restore the row + show error toast
- Per-row loading state during delete. Don't block the whole list.
- For the Market Documents surface: small label under the upload zone reading
  "Visible to all deals in {submarket_name}" — operators need to know this isn't
  a deal-private file.

TESTS
- Vitest per handler:
    - download success → window.open called with returned URL
    - delete success → row removed
    - delete failure → row restored, toast shown
    - delete 403 → row restored, "permission denied" toast
- Backend integration test per new endpoint (in-memory DB):
    - download returns { url } for authorized caller
    - download returns 403 for unauthorized
    - delete soft-deletes (deleted_at populated, row not returned by list)
    - delete 403 for unauthorized
