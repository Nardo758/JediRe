## Context

You are implementing the Tax Service — a deterministic, jurisdiction-aware tax math engine that powers the F9 Taxes tab (Sections A, B, C, D) across all four sections of the JEDI RE platform. The F9 UI already exists; you are populating it, not redesigning it.

This is **Task 6.5** in `AGENT_TASK_INVENTORY.md`, classified as **C** (deterministic code). It is not an agent. It is a service that agents call as a tool.

The launch jurisdictions are: **FL** (with Miami-Dade, Broward, Palm Beach county overlays), **GA** (with Fulton county overlay), **TX** (with Harris county overlay), and a **federal** ruleset that always applies. Plus a **default** state ruleset for jurisdictions not yet modeled.

## Ground Rules — Non-Negotiable

These are derived from the existing `REPLIT_AGENT_IMPLEMENTATION_PROMPT.md` ground rules and apply unchanged here:

1. **No `if (state === 'X')` outside `src/services/tax/rulesets/X/`.** This is verified by a CI grep check. If you find yourself wanting to write state-conditional logic anywhere else, stop and architect it as a ruleset method.

2. **Three-layer separation is sacred.** Rule mechanics → TypeScript code. Annual rates → JSON config. Per-deal data → runtime fetch. Never embed a 2026 millage rate in `fl.ruleset.ts`. Never put cap-formula math in a JSON file. Mixing layers silently corrupts future-year deals.

3. **Every output wraps in `LayeredValue<T>`.** No raw numbers escape the service. Provenance is always preserved.

4. **Composition is mandatory.** Every property resolves three stacked rulesets (federal + state + county). The composer walks the stack. Do not bypass it.

5. **Rate sheets are validated at boot.** Use a JSON schema validator (`zod` or `ajv`) on every rate sheet during service initialization. Invalid sheets fail loud, not silent.

6. **Deterministic outputs.** Same inputs ⇒ byte-identical outputs. Test this explicitly.

7. **No LLM in the hot path.** Agents have a narrow role (§11 of the spec) and operate out-of-band. The runtime computation is pure code.

8. **Cache aggressively, invalidate by fiscal year.** Jurisdiction cache TTL = end of fiscal year. Parcel cache TTL = end of fiscal year OR on tax bill upload, whichever comes first.

9. **All writes are audited.** Per platform convention; rate sheet promotions, cache invalidations, and override events all hit `audit_log`.

10. **No UI changes.** F9 tab structure already exists per the latest design. You populate cells; you do not restructure the tab.

11. **Audit before building.** Before Phase 1, run a codebase audit per the "audit before building" principle in CLAUDE.md. Document findings in `SPEC_QUESTIONS.md`. Do not duplicate existing types or services.

12. **JSX rendering rules apply** (even though this is mostly backend). Any UI integration code (e.g. F9 cell wiring) follows: space after `return` before JSX; no `React.Fragment` with keys inside CSS grids; no IIFE patterns inside JSX; named function components with explicit returns.

**Deliverables:**

2. `src/services/tax/rateSheets/federal-2026.json` per spec Appendix B
3. Unit tests:
   - Bonus dep returns 0.20 for 2026, 0.40 for 2025, 0 for 2027
   - Depreciation life: 27.5 for multifamily, 39 for office
   - Cost seg eligible for all property types
   - Federal income tax rate by entity + bracket
4. Section C composer stub (`computeSectionC`) that takes `RulesetStack` with only `federal` populated and returns the federal-only result

**Checkpoint:** Federal ruleset passes tests. Hand off summary.

## What NOT to Do

- ❌ Do NOT redesign the F9 Taxes tab UI — it exists, you populate it
- ❌ Do NOT make tax math agent-driven; this is a deterministic service
- ❌ Do NOT embed rates (millage, doc stamp, bonus dep %) inside ruleset TypeScript files — rates go in JSON
- ❌ Do NOT embed rule logic (10% cap formula, bonus dep schedule structure) in JSON — rules go in TypeScript
- ❌ Do NOT add `if (state === 'FL')` anywhere outside `rulesets/fl/`
- ❌ Do NOT skip the Phase 0 audit — duplicating existing types/services is the highest-cost failure mode
- ❌ Do NOT bypass `LayeredValue` wrapping — every output field carries provenance
- ❌ Do NOT introduce new dependencies (Redis, alternate HTTP clients, etc.) without flagging
- ❌ Do NOT write a Miami-Dade-specific scraper in v1 — use ATTOM. County adapters are deferred until cost/staleness justifies them
- ❌ Do NOT model TX franchise tax in v1 — it's flagged out of scope
- ❌ Do NOT model 1031 exchanges, opportunity zones, or AMT — also out of scope
- ❌ Do NOT silently work around ground rules — flag conflicts in writing
