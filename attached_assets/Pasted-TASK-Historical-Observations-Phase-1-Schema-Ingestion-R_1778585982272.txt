TASK — Historical Observations Phase 1: Schema + Ingestion + Reminder

Per docs/architecture/HISTORICAL_OBSERVATIONS_SPEC.md §11
Phase 1, §3 (schema), §6 (ingestion architecture), §7 (property
performance ingestion), and §9 (reminder system).

This is the foundation chapter for the platform's empirical
calibration substrate. Once Phase 1 lands, every existing parse
operation also creates corpus rows, the operator can see data
coverage per deal, and weekly reminders surface gaps.

  REQUIRED:

    parallel work, doesn't block Phase 1)

  NOT REQUIRED:
  - CoStar parser / dual-write (Phase 4 work)
  - LODES / QCEW / Veraset ingestion (Phase 4)
  - M22 variance analysis surface (separate spec, future)
  - M38 reliability ledger (separate, future)

SCOPE

In scope:
  - Schema migration: historical_observations table with all
    columns per HISTORICAL_OBSERVATIONS_SPEC.md §3
  - Transformer service: per-parser-type writes to corpus
    inline with existing data-router.ts route functions
  - Realized-outputs backfill service: nightly job that walks
    rows hitting T+3, T+12, T+24 anniversaries and populates
    realized_*_t* columns from later observations
  - CorpusQueryService: query API per HISTORICAL_OBSERVATIONS_SPEC.md §8
    (skeleton with stub implementations is acceptable for
    Phase 1; full empirical refit logic is Phase 5)
  - Lifecycle transition service: on deals.status →
    'owned'/'portfolio'/'closed', auto-create Data Library
    Deal Folder (via the deal_id FK from the prerequisite
    ticket) and bootstrap initial historical_observations row
  - Reminder system: 4 NotificationType values added, weekly
    Monday 09:00 local-time Inngest cron, Data Coverage Panel
    component on deal page header
  - Weekly digest: aggregate notification + email per
    HISTORICAL_OBSERVATIONS_SPEC.md §9.4

Out of scope:
  - CoStar / market survey / REIT 10-K / comp parsers (Phase
    4-6; spec'd but not built here)
  - LODES + QCEW + FRED ingestion (Phase 4)
  - Empirical refits (M35 playbook re-derivation, M07 baseline
    fitting, M36 Σ Phase B — all Phase 5+)
  - M38 reliability stats (waits for owned-and-held cycle per
    §1.2)
  - Subject backfill of Jacksonville + 2 Atlanta historical
    performance (separate one-sprint Replit session, parallel
    to this work)

backend/src/services/historical-observations/document-to-corpus.ts (NEW):

export async function writeT12ToCorpus(
  pool: Pool,
  dealId: string,
  parsedT12: T12ParsedResult,
  reportPeriod: Date,
): Promise<void>;

export async function writeRentRollToCorpus(
  pool: Pool,
  dealId: string,
  parsedRentRoll: RentRollParsedResult,
  snapshotDate: Date,
): Promise<void>;

export async function writeOMToCorpus(
  pool: Pool,
  dealId: string,
  parsedOM: OMParsedResult,
): Promise<void>;

export async function writeTaxBillToCorpus(
  pool: Pool,
  dealId: string,
  parsedTaxBill: TaxBillParsedResult,
  taxYear: number,
): Promise<void>;

Each function:
  1. Looks up the deal's parcel_id (creating if needed)
  2. Determines observation_date from the parsed result's
     reporting period (NOT the upload timestamp)
  3. Computes the row fields per HISTORICAL_OBSERVATIONS_SPEC.md §7.6 table
  4. Looks for an existing row at
     (parcel_id, observation_date, 'monthly')
  5. INSERTs new row OR UPDATEs existing row with new
     source_signals appended and tier promoted if appropriate
     (e.g., S2 → S1 if T12 + RentRoll both present within
     a 14-day window)
  6. Returns void; throws on error so the outer transaction
     rolls back

INTEGRATION

backend/src/services/document-extraction/data-router.ts —
add corpus write calls at the end of each existing route

  case 'T12': {
    const inserted = await routeT12(pool, data, propertyId, dealId, sourceRef, sourceDate);
    await writeT12ToCorpus(pool, dealId, data, reportPeriodFromData(data));
    return inserted;
  }
  case 'RENT_ROLL': {

    await writeRentRollToCorpus(pool, dealId, data, snapshotDateFromData(data));
    return inserted;
  }
  // ... similar for OM, TAX_BILL

The corpus write is INLINE TRANSACTIONAL — same connection, same
transaction. If the corpus write throws, the entire upload rolls
back. This is intentional (per HISTORICAL_OBSERVATIONS_SPEC.md
§7.5 architectural decision; events-without-subscribers is a
known failure mode per audit TE-03).

REALIZED OUTPUTS BACKFILL SERVICE

backend/src/services/historical-observations/realized-outputs.service.ts (NEW):

export class RealizedOutputsService {
  /**
   * Walk every row where realization_complete = FALSE.
   * For each, compute realized_*_t* values from later
   * observations at the same geography_level + geography_id.
   *
   * A row's realization is "complete" when all three windows
   * (T+3, T+12, T+24) are either populated or definitively
   * unfillable (no observation exists at the required offset
   * after sufficient time has passed).
   */
  async backfillPendingRows(asOfDate: Date): Promise<{
    rowsProcessed: number;
    rowsCompleted: number;
    rowsStillPending: number;
  }>;
}

backend/src/inngest/functions/historicalObservationsBackfillCron.ts (NEW):

export const historicalObservationsBackfillCron =
  inngest.createFunction(
    { id: 'historical-observations-backfill-weekly',
      name: 'Historical Observations Realized Outputs Backfill' },
    { cron: '0 3 * * 1' },  // 03:00 UTC every Monday
    async ({ step }) => {
      const svc = new RealizedOutputsService(getPool());
      return step.run('backfill', () => svc.backfillPendingRows(new Date()));
    },
  );

LIFECYCLE TRANSITION SERVICE

backend/src/services/portfolio/lifecycle-transition.service.ts (NEW):

When deals.status transitions to 'owned' / 'closed' / 'portfolio':
  1. Add to Portfolio (already automatic via existing query)
  2. Mark the deal's Data Library Folder as "Portfolio" (the
     dependency ticket creates the deal_id FK; this work uses it)
  3. Bootstrap historical_observations row at status transition
     date with whatever subject state exists (from existing
     parsed OM, T12, rent roll data)

Trigger via a Kafka event 'deal.status.changed' emitted by the
deals service at status transitions. Subscribe to it; on
relevant transitions, run the three steps.
