### Task 6.5 — Tax Math Service Design

Replaces the Florida-specific mechanics with a pluggable architecture. Florida is one ruleset; Texas (DFW), Georgia (ATL), and others are additional rulesets loaded at runtime based on the deal's jurisdiction.

```
src/services/tax/
├── taxService.ts                    # Main entry: computes tax forecast for a deal
├── rulesets/
│   ├── fl.ruleset.ts                # FL: 10% non-homestead cap, doc stamps,
│   │                                #     intangible tax, reassessment on sale,
│   │                                #     truth-in-millage
│   ├── tx.ruleset.ts                # TX: no state income tax, Chapter 313
│   │                                #     abatements, Harris/Dallas county
│   │                                #     protest windows, 10% homestead cap
│   ├── ga.ruleset.ts                # GA: Fulton/DeKalb millage, homestead
│   │                                #     exemptions, conservation use, annual
│   │                                #     assessment freeze rules
│   ├── ny.ruleset.ts                # Future: NYC 421-a, J-51, ICAP
│   ├── ca.ruleset.ts                # Future: Prop 13, Mello-Roos
│   └── default.ruleset.ts           # Generic fallback: ad valorem + transfer tax
├── types.ts                         # TaxRuleset interface, TaxForecast type
└── resolver.ts                      # Deal → jurisdiction → ruleset lookup
```

**Interface contract every ruleset implements:**

```typescript
interface TaxRuleset {
  jurisdiction: string;               // e.g. 'FL', 'TX-Harris', 'GA-Fulton'

  // Annual property tax at year N given assessed value + millage + exemptions
  annualPropertyTax(ctx: TaxContext, year: number): number;

  // Transfer taxes due at acquisition (doc stamps, deed recording, etc.)
  acquisitionTransferTax(purchasePrice: number): number;

  // Transfer taxes due at sale (may differ from acquisition)
  dispositionTransferTax(salePrice: number): number;

  // Reassessment trigger: what happens to assessed value on sale?
  reassessmentOnSale(ctx: TaxContext): 'full' | 'capped' | 'none';

  // Annual assessment cap (e.g. FL 10% non-homestead, TX 10% homestead)
  annualAssessmentCap(): number | null;

  // Special taxes (intangible tax, mortgage recording tax, etc.)
  specialTaxes(ctx: TaxContext): SpecialTax[];

  // Abatement/exemption eligibility check
  abatementEligibility(ctx: TaxContext): AbatementProgram[];

  // Metadata
  requiresInputs(): string[];         // e.g. ['millage', 'exemptions', 'assessed_value']
  dataSourceHints(): string[];        // e.g. ['county_assessor_api', 'truth_in_millage_pdf']
}
```

**Resolver logic:**

```typescript
function resolveRuleset(deal: Deal): TaxRuleset {
  const key = `${deal.state}-${deal.county}`;
  if (rulesets[key]) return rulesets[key];       // most specific
  if (rulesets[deal.state]) return rulesets[deal.state];  // state-level
  return rulesets.default;                        // generic fallback
}
```

**Adding a new jurisdiction is a new file in `rulesets/`, not a code change elsewhere.** The CashFlow Agent doesn't know or care which ruleset applies — it calls `taxService.forecast(deal, assumptions)` and gets back a typed `TaxForecast`.

**Launch rulesets (matches your current market coverage):**
- `fl.ruleset.ts` — full implementation (existing logic migrates here)
- `tx.ruleset.ts` — Dallas/Houston baseline
- `ga.ruleset.ts` — Atlanta baseline
- `default.ruleset.ts` — everywhere else, conservative fallback

**Jurisdictions added post-launch** become data projects, not engineering projects — you gather the rules, a domain expert encodes them into a new ruleset file, the ruleset registers itself via a convention-over-configuration pattern.

## Category 19 — Web Search as an Agent Capability

Agents need the ability to look up information that isn't in structured data sources. This is a **fallback capability, not a primary research path.** Agents reach for web search only when the relevant data isn't available via a dedicated tool (FRED, CoStar, county GIS, RentCast, etc.).

### When web search helps

- **Jurisdiction rules that aren't in APIs** — county millage rates, municipal ordinances, recently-passed legislation, TIF/abatement program specifics. These are often buried in PDFs and local government sites that don't have API access.
- **Regulatory changes between scheduled data refreshes** — a state insurance regulator just changed admitted-market rules; your structured source updates monthly but the change happened yesterday.
- **Trade press intelligence** — cap rate benchmarks, market commentary, deal gossip from GlobeSt, Bisnow, Multi-Housing News, CoStar News that hasn't propagated to structured data.
- **Fresh news that hasn't hit the ingestion pipeline** — the gap between a news event firing and Category 11 processing it.
- **Stale Municode lookups** — zoning amendments where Municode's last update is older than the amendment date.

### When web search is the wrong answer

- Anything in FRED / BLS / Census — your direct API is cleaner, faster, cheaper, and more reliable.
- Anything in CoStar, RentCast, county GIS — structured sources are authoritative.
- Property-specific data (parcel, ownership, tax history) — public records APIs win.
- Computation — agents compute; they don't Google the answer.
- General knowledge the model already has — no search needed.

### Task list

| # | Task | Classification | Notes |
|---|------|----------------|-------|
| 19.1 | Provision web search tool with scoped access | **T** | New tool: `web_search` — Tavily recommended, Brave as cheaper alternative |
| 19.2 | Add `fetch_webpage` tool for deep reads of a specific URL | **T** | Complements search — search to find URLs, fetch to read them |

| 19.8 | Search result provenance in LayeredValue | **C** | Fields sourced from web search get `source: 'agent:research'` + metadata flag `derived_from_search: true` |

### Recommendation: Tavily over Brave

**Tavily** — purpose-built for AI agents. Returns cleaner, pre-summarized results optimized for LLM consumption. Handles the "find information and cite sources" pattern natively. Pricing: ~$0.01-0.02 per search. Better quality, especially for research tasks.

**Brave** — general-purpose search API. Cheaper (~$0.003 per search at volume). Raw results, more like Google SERP. Fine for simple lookups but requires more work from the agent to synthesize.

**SerpAPI / Serper** — proxies Google results. Best quality, most expensive ($0.01-0.05 per search), legal gray area for some use cases. Overkill for this.

**Exa** — semantic/vector search. Excellent for research where query intent matters more than keyword match. Worth considering specifically for Commentary Agent.

**Recommendation:** Start with Tavily for both Research and Commentary agents. Measure usage. If search costs become significant (>5% of total Claude spend), evaluate migrating Research Agent to Brave for the volume and keeping Tavily on Commentary for quality-critical narrative work.

### Guardrails

- **Budget cap applies.** Web search counts against per-run cost cap. An agent that tries to search 50 times fails safely.
- **Citations are mandatory.** Any agent output containing information from a web search must include the source URL and retrieval timestamp. Enforced at the output schema level.
- **Not a primary source.** If a tool exists for the data (e.g. `fetch_fred_indicators`), the agent should use it first. Web search is fallback. Enforce in the system prompt: "Use structured data tools first. Only use web_search when structured tools cannot answer the question."
- **Stale-data tolerance.** Cached search results may be up to 24h old. The agent should not claim real-time certainty based on cached search.
