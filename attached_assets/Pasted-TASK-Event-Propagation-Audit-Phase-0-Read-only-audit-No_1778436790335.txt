PHASE 0a — EVENT SUBTYPE × MODULE MATRIX

Build the core artifact: a matrix with event subtypes as rows and
consuming modules as columns. Each cell gets a classification:

  ✓ WIRED              — Event subtype flows through primary channel,
                          mutates assumption/state programmatically,
                          closed-loop (event change → state change
                          without manual intervention).
  ⚠ PARTIALLY_WIRED    — Event surfaces in display/attribution but
                          does not mutate underlying LayeredValue
                          or assumption layer. Read-only effect.
                          (Example today: getM35ProformaAttribution
                           reads event_forecasts but doesn't apply
                           to FinancialContext.assumptions.)
  ⚙ AGENT_ONLY         — Event surfaced only to agent prompts as
                          text/JSON. Agent decides whether/how to
                          apply. No deterministic write path.
  ✗ NOT_WIRED          — Event subtype declared for this channel
                          in EVENT_SUBTYPE_REGISTRY but no code
                          consumes it from the spec'd module.
  ⚡ DUPLICATE_INJECTION — Event applied through multiple paths
                          (channel routing violation per Causal
                          Discipline Invariant A).
  ⤴ MISCLASSIFIED      — Event going through different channel
                          than primary_channel declares.
  ? SPEC_GAP           — Event subtype seen in code but not in
                          EVENT_SUBTYPE_REGISTRY.

For each cell: cite file:line evidence both for what IS wired and
what is NOT. If the cell is NOT_WIRED, the evidence is the absence
of any import/read of m35TrafficApiService / event_forecasts /
event_playbooks / key_events in the relevant module file. Document
the absence concretely:

  Example:
  multifamily_delivery × M07_traffic:
    Expected: trafficCalibrationJob imports m35TrafficApiService,
              calls getActiveEvents() to filter calibration window
              per spec §2.3 Mechanism A.
    Found:    backend/src/jobs/trafficCalibrationJob.ts contains
              zero references to m35TrafficApi (grep -n
              "m35TrafficApi\|getActiveEvents" returns no matches).
              Calibration runs event-blind.
    Status:   NOT_WIRED.

PHASE 0b — END-TO-END TRACE ON A LIVE EVENT

Pick one canonical example to trace fully through the system:
multifamily_delivery, 300 units, submarket = Bishop's submarket
(if known) or a synthetic test submarket.

  1. Event ingestion — what process writes the row into key_events?
  2. Playbook lookup — m35-playbook.service.ts: does a playbook

     value? If no (insufficient analogs), what's the fallback?
  3. Forecast generation — m35-forecast.service.ts: which metric_keys
     get event_forecasts rows generated? Confidence values?
  4. M07 consumption path:
     a. Does trafficCalibrationJob exclude the event window from

  5. M09 consumption path:

        cap_rate value to FinancialContext.assumptions.exitCapRate
        platform layer?
     b. Or does it only show eventAttribution to the user (read-only)?
  6. M08 consumption: does causal-discipline-engine compute
     expected_supply_pressure using the event subtype?
  7. M25 consumption: does jedi-score.service.ts pipeline_units

  8. Exit cap consumption: does trajectory-engine.ts read M35
     cap_rate forecasts, or only use the hardcoded
     exit_cap_trajectory: -0.0025 baseline at line 84?

For each step, report: WIRED / PARTIALLY_WIRED / AGENT_ONLY / NOT_WIRED
with the actual file:line evidence. Where the path is broken, identify
where it breaks.

PHASE 0c — DUPLICATE INJECTION CHECK

For each event subtype, search for ALL consumers (not just the
primary channel). Identify any case where the same event is read
by more than one channel-injection path:

  - employer_expansion: should ONLY drive M07. If anything in M09
    or M14 reads employer_expansion events directly and applies
    them to assumptions/risk, that's DUPLICATE_INJECTION.
  - multifamily_delivery: should ONLY drive M07. If JEDI Score
    reads pipeline_units AND M07 reads multifamily_delivery
    events AND ProForma reads vacancy_rate from event_forecasts,
    that's three injections of the same supply signal.
  - Etc. for every subtype.

This check directly tests Causal Discipline Invariant A. Report
violations as DUPLICATE_INJECTION findings with all the redundant
read paths cited.

Note: M03 reading multifamily_permit for dev feasibility is NOT
duplicate injection (per spec §3.3, that's allowed read-only access
to event data for a different purpose — developable supply, not
present-period market). The test for DUPLICATE_INJECTION is:
multiple modules applying the same event to overlapping
assumptions/state.

PHASE 0d — CAUSE/SYMPTOM SEPARATION CHECK

Causal Discipline Invariant B says strategy scores must compose
multiplicatively: Score = Expected × Validation, not additively.

Search the M08 strategy scoring code path:
  - backend/src/services/m08-strategies.service.ts
  - backend/src/services/sigma/causal-discipline-engine.ts
    (specifically computeSubScore method)

Verify:
  - Causes (employer events, supply events) feed Expected component
  - Symptoms (search momentum, surge index, days_to_lease) feed
    Validation component
  - The two are combined multiplicatively, with validation_factor
    bounded [V_FLOOR, V_CEILING] (0.7 to 1.3 per current code)
  - Causes and symptoms are NOT added as independent evidence

Report any place where the additive failure mode is present (cause
and symptom of the same event both pumping a sub-score independently).

PHASE 0f — SUPPLY/DEMAND FORMULA AUDIT

Specifically catalog the formulas the platform uses to translate
event magnitude into assumption deltas:

  M35 playbook (m35-playbook.service.ts:608-625):
    Jobs-based (employer events):
      rent_growth_pp_baseline = jobs × 0.0002 × wageMult × msaAtten
      absorption_pp_baseline  = jobs × 0.0001 × wageMult × msaAtten
    Non-jobs (supply, transit, etc.):
      scaled_median = playbook_median × msaAtten × wageMult

  Backtest service heuristic (backtest.service.ts:746-781):
    supply_delivery base: -1.5% (rent change)
    magnitude multipliers: 0.5 / 1.0 / 1.5 / 2.0
    jobs factor: 1 + min(|jobs|/1000, 2) × 0.2

  JEDI Score supply step-down (jedi-score.service.ts:286-291):
    0 units → 60, <200 → 55, <500 → 50

  Trajectory engine exit cap (trajectory-engine.ts:84):
    Hardcoded baseline -0.25%/yr, no event input

Report: which formulas are EVENT-AWARE (consume event data) vs
EVENT-BLIND (use only hardcoded constants or market snapshots).
Identify any inconsistencies (same magnitude unit producing
different effects in different places).

For the 300-unit example, walk through each formula:
  - Playbook non-jobs path: scaled by MSA tier (×0.7/1.0/1.4)
    and wage premium (×1.0/1.4). What does the actual playbook
    median value say for multifamily_delivery × vacancy_rate?
    × rent_growth_yoy? × cap_rate?
  - Backtest: -1.5% base × magnitude multiplier. What's the
    expected magnitude for 300 units?
  - JEDI Score: 300 units lands in the <500 bucket → -10 points
    on supply sub-score. This applies regardless of submarket
    size — is that correct?

PHASE 0g — CORRELATION ENGINE OVERLAP

The Correlation Engine (correlationEngine.service.ts, 30 CORs) is
a separate signal system from M35 playbooks. Verify:
  - COR-06 (Pipeline % vs Rent Growth) uses snapshot supply_pressure
    (categorical: low/moderate/high) — not event-level
  - COR-19, COR-20 are reserved/unused
  - No COR currently consumes event_forecasts or event_playbooks

If any correlation IS reading event data, document as DUPLICATE_
INJECTION (event signal entering through both the M35 attribution
path and the correlation adjustment path).

Also document any CORs whose adjustment delta_pct would be amplified
or contradicted by an active M35 event in the same submarket. These
aren't necessarily bugs — but they're places where the two systems
should be reconciled, not running in parallel.
