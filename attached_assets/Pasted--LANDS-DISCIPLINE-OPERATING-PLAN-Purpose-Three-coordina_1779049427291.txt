### Sequential gate to Scenario Management Phase 1

Scenario Management Phase 1 begins ONLY when Priorities 1, 2, 3 are all at "done done."

This is not negotiable in the lands-discipline approach. The case for sequential:

- Scenario Management touches Layer 2 (deal_scenarios table, year1 wrapping). If F-010 layer contamination still exists when scenarios ship, it gets replicated across every scenario.
- Ticket B's override pattern fix is structurally identical to what scenarios need. Coordinating them as one piece of work is cheaper than doing them separately.
- The agent prompt's reasoning depth (Priority 3) determines whether scenarios are populated by genuinely intelligent agent runs or by superficial ones. Better to fix reasoning before scenarios capture state.

Estimated total time: 3-4 weeks for sequential closure. Then 4-6 weeks for Scenario Management. Total: 7-10 weeks from now to Scenario Management v1.0 in production.

## PART 2 — WEEKLY STATUS RITUAL

A 10-15 minute weekly ritual that catches drift early. Designed to be a written status check, not a meeting. Worth doing every Monday or Friday consistently.

### Status format template

```markdown
# WEEKLY STATUS — [DATE]

## In-flight items

| Item | State | Last activity | Blocked? | Next action |
|---|---|---|---|---|
| Priority 1 (F-009) | <state> | <date> | <yes/no/why> | <action> |
| Priority 2 (F-010/Ticket B) | <state> | <date> | <yes/no/why> | <action> |
| Priority 3 (prompt iteration) | <state> | <date> | <yes/no/why> | <action> |
| ... | | | | |

States:
- not_started — never picked up
- in_progress — being actively worked
- waiting_review — PR open, awaiting review
- waiting_verification — merged, awaiting done-done verification
- done_done — verified complete per criteria
- blocked — cannot progress, reason documented

## Recently completed (since last status)

For each item moved to done_done this week:
- Item name + closing note link
- Verification result summary
- Any open follow-ups discovered during verification

## Supposedly done, verify

Items that were marked "done" at some point in the past but haven't been verified end-to-end against their done-done criteria.

For each item:
- What was claimed done
- What verification has happened
- What verification is still needed
- Action to close the gap

This section directly catches the Task #824 / F-009 failure mode. Items here are not yet "done done" even if the team or a Replit task says otherwise.

## Drafted but not yet active

Items where specs or plans exist but no execution is in flight. Documents the queue without forcing it to be active.

For each:
- Item name + spec doc
- Why it's not active yet (dependency, priority, bandwidth)
- Trigger condition for moving it to active

## Spec inventory (delta from last week)

New specs authored or revised this week:
- New: <list>
- Revised: <list>

This section catches the "spec authoring outpaces execution" pattern. If new specs accumulate weekly while in-flight items don't move to done_done, the operating discipline isn't working.

## Open questions / decisions needed

Things waiting on a decision from Leon to unblock progress.

For each:
- Question
- Context / options
- Date raised

## Risks surfaced this week

New risks identified, or existing risks that escalated.

For each:
- Risk
- Mitigation status
- Owner
```

### How to use it

**The first run is the baseline.** Run the format against the current state right now. Every item drafted or in flight gets a row. The "supposedly done, verify" section gets populated with items where the actual state is unclear (M36 plausibility, source residual convention integration, various spec applications).

**Subsequent runs are delta + verification.** Each week's status moves items between states and surfaces gaps. Items shouldn't sit in `waiting_verification` for more than two weeks without explicit action.

**The "supposedly done, verify" section is the most important.** It's the institutional memory that catches Task #824-style failures. An item moves OUT of this section only when verification confirms done-done. If verification surfaces gaps, the item moves back to in_progress, not silently to done.

**The ritual takes 10-15 minutes per week.** Less than that and it's not real. More than that and it won't get done consistently. The format above is sized for that window.

### Where to keep it

A markdown file in the repo: `docs/operations/weekly-status.md`. Each week's status is appended to the top (most recent first). The file grows over time as the platform's working history.

Alternative: a Notion page, a Linear cycle review, whatever the team's tooling supports. Format matters more than location.

Each phase has a specific verification workflow that must complete before

## PART 5 — WHY THIS APPROACH

The pattern across the past three weeks is that spec authoring has outpaced
verification. Most specs are good; several "completed" items haven't been
verified end-to-end. The 464 Bishop diagnostic was the first systematic
verification of the platform's state, and it surfaced multiple items
that had drifted from "supposedly done" to "actually incomplete."

The three mechanisms in this plan address that pattern at different layers:

- **Sequential closure** prevents new architecture from being built on top of unverified prior work
- **Weekly status ritual** maintains accurate state of every drafted/in-flight item, catches drift within one week instead of three
- **Verifiable acceptance criteria + Phase Verification Protocol** make "done" claims structurally harder to make falsely

This is a slower path than parallel execution. The case for it: the platform's positioning depends on trust in the underwriting it produces. Every silent failure surfaced (F-009, F-010, the agent reasoning gaps) was an erosion of that trust. Spending three to four weeks closing the trust deficit and establishing operational discipline is cheaper than shipping Scenario Management on top of an unverified foundation and having to refactor when the next diagnostic surfaces more issues.

The cost is real: 7-10 weeks to Scenario Management v1.0 instead of 4-6 weeks with parallel work. The benefit is real: every step is verified, every component is durable, the platform's trust foundation tightens with each completed priority.

This is the high-conviction path. Worth the slower start.
