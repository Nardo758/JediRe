# F9 Data Flow Wiring — Audit & Reconciliation

## Context

The F9 Financial Engine has four critical data flows that need to be
verified end-to-end. Multiple prior audits have surfaced partial breaks
(unit_mix wipe, purchase price dual-source, projections array not
surfaced on response). This task is the first comprehensive audit of
all four flows together — not just per-row, but per-flow integrity.

Existing TODO files to cross-reference (do not duplicate items already
logged there):
- `TODO_F9_SIDE_DEBT.md`
- `TODO_DEAL_TERMS_FOLLOWUP.md`
- `TODO_OPERATOR_STANCE_PHASE_3.md` (if it exists yet)

## The four flows

```
Flow 1: Unit Mix tab          ──┐
                                ├──▶  Pro Forma  ──┐
Flow 2: Other Income (INPUTS) ──┘                  │
                                                   │
Flow 3: INPUTS (Leasing + ProForma Numbers)  ──────┤
        + Pro Forma user overrides           ──────┤──▶  Projections
                                                   │
Flow 4: Deal cost / Partnership / Closing  ────────┴──▶  Capital + Returns
```

## Working principles (read first)

1. **Phase 1 audit only — no fixes.** Trace each flow with file:line
   evidence. Classify breaks. Stop and report.

2. **Use real data, not source-code defaults.** The prior session
   showed why this matters — query a live deal (suggest `3f32276f-
   aacd-4da3-b306-317c5109b403`, the 464 Bishop deal we've been working
   with) and verify the actual values flowing through, not the
   theoretical defaults.

3. **Cross-reference existing TODO files.** If a break is already
   logged, note it and move on. Don't re-surface known-knowns.

4. **Distinguish four break classes:**
   - `DATA_NOT_FLOWING` — source has data, consumer doesn't read it
   - `DATA_STALE` — consumer reads but cached/seeded value diverges
     from current source
   - `EVENTS_NOT_EMITTED` — change at source doesn't trigger consumer
     re-fetch
   - `CONSUMER_NOT_SUBSCRIBING` — events fire but consumer ignores

## Phase 1 — Audit (Plan mode, read-only)

For each flow below, produce a findings table with this format:

| Step | Source File:Line | Consumer File:Line | Break Class | Notes |
|---|---|---|---|---|

Then a paragraph summarizing the flow's overall health: green (works
end-to-end), yellow (works but with known limitations), red (broken).

### Flow 1: Unit Mix → Pro Forma

Trace the path of a unit_mix change to a Pro Forma render:

1. Where is `unit_mix` stored on the deal? (`deal_assumptions.unit_mix`
   JSONB confirmed from the wipe-bug hotfix)
2. Where does the composer compute GPR from unit_mix? Or does the
   composer use a standalone `gpr` field that's been pre-aggregated?
   If the latter, what computes the standalone gpr — and is it kept
   in sync with unit_mix on edit?
3. When the user edits a unit_mix row, does the GPR row in Pro Forma
   update? Trace the event chain: UnitMixTab edit → API write →
   what event fires → does ProFormaSummaryTab re-fetch?
4. If unit_mix changes and gpr is a stored standalone field, is there
   a service that re-derives gpr on unit_mix change? Or does it
   require a manual reparse?

Specifically check: for 464 Bishop, is the rendered GPR ($4,901,400)
equal to `sum(unit_mix[].count × unit_mix[].rent × 12)`? If the math
doesn't tie, that's a `DATA_STALE` finding.

### Flow 2: Other Income → Pro Forma

The earlier missing-rows debugging surfaced that `other_income_per_unit`
gets dollarized to `other_income` via the `toDollarRow` helper at
`proforma-adjustment.service.ts:1817-1900` (now landed). So the basic
read flow exists. Audit:

1. Does `other_income` in the Pro Forma response equal
   `other_income_per_unit × totalUnits × 12`? Verify on 464 Bishop.
2. Are there other "other income" buckets (parking, RUBS, ancillary,
   pet rent, application fees, late fees, vending) that exist in
   T-12 / rent roll but never make it into the Pro Forma's
   `other_income` line? Specifically check the year1 JSONB for fields
   matching `parking_*`, `rubs_*`, `ancillary_*`, `pet_*`, `late_*`,
   `vending_*`, `app_fee_*`. If any are populated but not consumed
   by the composer, that's `DATA_NOT_FLOWING`.
3. Are user edits to other-income components scattered across surfaces
   (one tab edits `parking`, another edits `pet_rent`)? Or
   consolidated in INPUTS / Other Income section?
4. When other_income changes, does NRI / EGI / NOI re-derive in real
   time, or only on full re-fetch?

### Flow 3: INPUTS + Pro Forma overrides → Projections

This is the most complex flow. The prior DEAL TERMS audit surfaced
that `buildProjectionsForExport(data, holdYears)` is called at
`deal-assumptions.routes.ts:504` but the resulting `projs` array is
local to the handler and not surfaced on the F9DealFinancials response.
Audit:

1. Where does the projections engine live? (`buildProjectionsForExport`
   per the prior audit — confirm or correct.) What inputs does it
   consume?
2. Does it read from year1 LV with overrides applied (resolved values),
   or from raw seed values pre-override?
3. Does it consume Leasing Assumptions (rent growth, vacancy growth,
   concession burn-off) from INPUTS? File:line of those reads.
4. Does it consume ProForma Numbers (OpEx growth assumptions,
   inflation, expense escalators) from INPUTS? File:line.
5. Are projections cached, or recomputed on every fetch? If cached,
   what invalidates the cache?
6. Critically: is there a path from the Projections tab UI back to
   the projections engine, or does the Projections tab only render
   what's in the static F9 response? If the latter, hold-period
   changes from DEAL TERMS won't surface in Projections without a
   refetch — which we already flagged as `Soft` in the prior PR review.
7. The user-override → projection chain: trace one specific override
   (e.g., user edits `payroll` to $400K). Does the projections array
   reflect Year 2 = $400K × (1 + escalator)? Or stuck at platform-
   default $250K × (1 + escalator)?

### Flow 4: Deal cost / Partnership / Closing → Capital + Returns

Per DEAL TERMS audit, this flow has known gaps:
- Purchase Price dual-source divergence (logged P1)
- 5 closing-cost sub-rows are `PERSISTENCE_GAP` (only aggregate
  `su:closingCosts` writes)
- Partnership/LP-GP split — never wired in DEAL TERMS

Audit what's actually wired today:

1. Purchase Price → All-in Basis → Equity Required: trace the path
   from DEAL TERMS Purchase Price to Capital tab Sources & Uses to
   Returns tab Equity Required. Where does the Capital tab read
   purchase price from — `deal_data.purchase_price`, `deals.budget`,
   `capitalStack.purchasePrice`, or somewhere else? Confirm whether
   the dual-source bug bleeds into Capital tab today.
2. Closing Costs → S&U: the aggregate `su:closingCosts` writes via
   `/financials/override`. Does the Capital tab S&U section render
   that single aggregate, or does it expect the 5 sub-rows
   (broker fee / legal / lender / reserves / contingency) and show
   "Not Provided" for them today?
3. Partnership / LP-GP split: locate where this lives in the schema
   and which tab reads it. The Returns tab's `lpNetIrr` and
   `lpEquityMultiple` calculations imply a split exists somewhere —
   trace the read source. If hardcoded to 90/10, that's a
   `PERSISTENCE_GAP` worth flagging.
4. Loan/debt terms → Capital tab: out of scope of DEAL TERMS, but
   verify the existing F9 Capital tab reads debt assumptions
   correctly. The prior reviewer noted `proforma_assumptions.user_exit_cap`
   exists in capsule-bridge as a separate write path — confirm Capital
   tab doesn't accidentally read from there too.
