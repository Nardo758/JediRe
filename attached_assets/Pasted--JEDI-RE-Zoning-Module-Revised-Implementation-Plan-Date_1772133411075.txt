## Architecture Reality (What We're Actually Working With)

| Layer | Status | Detail |
|-------|--------|--------|
| **Data** | ✅ Stored | PostgreSQL structured tables — district codes, setbacks, height, FAR, parking, etc. |
| **Source References** | ⚠️ Partial | Section numbers stored per rule, but no full Municode URLs |
| **Agent** | ✅ Functional | Queries stored data, returns structured results per zoning district |
| **Interpretation** | ⚠️ Split | Agent returns raw values; UI currently does the property-specific math/comparison |
| **Source Links** | ✅ On detection card | T002 delivered Planning & Zoning + Municode + Search buttons on the detected zoning card |
| **Source Links** | ❌ Not on interpreted results | Individual rules (setback, FAR, height) don't trace back to their code section |

## What This Plan Solves

**Problem 1 — Source URLs don't exist yet.** Section numbers are stored but can't link anywhere. Users need to click through to the exact Municode section or planning department page to verify.

**Problem 2 — Interpretation lives in the wrong place.** The UI is doing math (lot depth minus setbacks = buildable depth) that the Agent should own. The Agent should return *contextualized* results: "Your buildable depth is 70ft" not just "rear setback is 20ft."

**Problem 3 — Source tracing stops at the detection card.** T002 gives you three link buttons when zoning is first detected. But as the user moves through the 7 tabs (Dev Capacity, Regulatory Risk, Comparator, etc.), every interpreted rule should still trace back to its source.

## Phase 1: Municode URL Construction

**Goal:** Turn stored section numbers into clickable Municode URLs  
**Effort:** 2–3 days  
**Prerequisite for:** Everything else — source tracing depends on resolvable URLs

### How Municode URLs Work

Municode URLs follow a predictable pattern:

```
https://library.municode.com/{state}/{municipality}/codes/{code_name}?nodeId={node_id}
```

Examples:
```
# Atlanta Zoning - Chapter 16
https://library.municode.com/ga/atlanta/codes/code_of_ordinances?nodeId=PTIIICOO(), AM_PT16ZO

# Specific section - 16-06.007 (MRC district dimensional requirements)
https://library.municode.com/ga/atlanta/codes/code_of_ordinances?nodeId=PTIIICOOAM_PT16ZO_CH6MIDEDI_S16-06.007DIMIRE
```

The `nodeId` is the key. It's derivable from the section number but isn't always a clean transformation — some municipalities use hierarchical IDs, others use encoded section numbers.

### Task T003: Build URL Resolver Service

Create a service that maps stored section numbers to full Municode URLs.

**Approach A — Lookup table (recommended for MVP):**

```sql
-- Add to existing schema or create new table
CREATE TABLE municode_section_urls (
    id SERIAL PRIMARY KEY,
    municipality_id INTEGER REFERENCES municipalities(id),
    section_number VARCHAR(50) NOT NULL,       -- e.g., "16-06.007"
    chapter VARCHAR(100),                       -- e.g., "Chapter 6 - Mixed Use Districts"
    municode_url TEXT,                          -- Full resolved URL
    municode_node_id VARCHAR(200),             -- nodeId parameter
    title VARCHAR(500),                         -- Section title for display
    last_verified TIMESTAMP,                   -- When URL was last confirmed working
    UNIQUE(municipality_id, section_number)
);
```

Populate for Atlanta first — the zoning ordinance has a finite number of sections. This can be done by scraping the Municode table of contents for Atlanta once, extracting section numbers and their corresponding nodeIds.

**Approach B — Algorithmic resolution (future):**

Build a URL generator that constructs the nodeId from section number patterns per municipality. This scales better but requires understanding each municipality's URL scheme.

**Recommendation:** Start with Approach A for Atlanta metro. You only need a few hundred rows (one per zoning code section). Scrape once, verify, store. Fall back to the web search URL (from T002) when a section number doesn't have a resolved URL yet.

### Task T005: Fallback URL Strategy

When a section number doesn't have a resolved Municode URL in the lookup table:

1. **First fallback:** Generate the `webSearchUrl` (already built in T001) using the section number as the search term
2. **Second fallback:** Link to the municipality's `planningUrl` (already built in T001)
3. **Display:** Show the section number as text with a "Search on Municode" link instead of a direct link

```typescript
function getSourceUrl(rule: ZoningRule, municipality: Municipality): SourceLink {
  if (rule.sourceUrl) {
    // Direct Municode link available
    return { url: rule.sourceUrl, label: "View on Municode", type: "direct" };
  }
  if (rule.sectionNumber) {
    // Generate search URL
    const searchUrl = `https://www.google.com/search?q=${municipality.name}+${municipality.state}+zoning+section+${rule.sectionNumber}+site:municode.com`;
    return { url: searchUrl, label: `Search §${rule.sectionNumber}`, type: "search" };
  }
  if (municipality.planningUrl) {
    // Fall back to planning department
    return { url: municipality.planningUrl, label: "Planning & Zoning", type: "general" };
  }
  return null;
}
```

## Phase 2: Move Interpretation Into the Agent

**Goal:** Agent returns property-contextualized insights, not just raw district rules  
**Effort:** 3–4 days  
**Why this matters:** The Agent should be the brain. The UI should display conclusions, not compute them.

### The Interpretation Gap

**What happens now:**
```
Agent: "MRC-3 rear setback = 20ft"
UI:    (user enters lot depth 90ft)
UI:    (calculates 90 - 20 = 70ft buildable depth)
UI:    displays "Buildable depth: 70ft"
```

**What should happen:**
```
Agent receives: district MRC-3 + lot dimensions (90ft depth, 60ft width, 5,400 sqft)
Agent returns:  "Buildable depth: 70ft (90ft lot - 20ft rear setback per §16-06.007)"
                "Buildable width: 50ft (60ft lot - 0ft front + 10ft side per §16-06.007)"  
                "Buildable area: 3,500 sqft (70ft × 50ft)"
                "Maximum units: 13 (5,400 sqft × 109 units/acre density per §16-06.005)"
```

The Agent does the math. The Agent cites the source. The UI just displays.

### Task T006: Property-Aware Agent Query

Update the Agent to accept property-specific inputs alongside the zoning district lookup.

**Current flow:**
```
Frontend → POST /zoning/query { districtCode: "MRC-3" }
Agent    → SELECT * FROM zoning_rules WHERE district = 'MRC-3'
Agent    → return raw rules
```

**New flow:**
```
Frontend → POST /zoning/interpret {
  districtCode: "MRC-3",
  property: {
    lotSizeSqft: 5400,
    lotWidthFt: 60,
    lotDepthFt: 90,
    currentUse: "commercial",
    isCornerLot: false,
    yearBuilt: 1965
  }
}
Agent    → SELECT * FROM zoning_rules WHERE district = 'MRC-3'
Agent    → JOIN municode_section_urls ON section_number
Agent    → apply property dimensions to rules
Agent    → return interpreted results with source citations
```

### Task T007: Interpretation Engine

Build the interpretation logic inside the Agent (or as a service the Agent calls). This is the math that currently lives scattered across the frontend.

```python
class ZoningInterpreter:
    """
    Takes raw zoning rules + property dimensions
    Returns contextualized, source-cited insights
    """
    
    def interpret(self, rules: dict, property: dict) -> list[Interpretation]:
        interpretations = []
        
        # Buildable envelope
        if property.get('lotDepthFt') and rules.get('rearSetback'):
            buildable_depth = property['lotDepthFt'] - rules['rearSetback']['value'] - rules['frontSetback']['value']
            interpretations.append({
                'category': 'envelope',
                'field': 'buildableDepth',
                'value': buildable_depth,
                'unit': 'ft',
                'calculation': f"{property['lotDepthFt']}ft lot - {rules['frontSetback']['value']}ft front - {rules['rearSetback']['value']}ft rear",
                'sources': [rules['frontSetback']['source'], rules['rearSetback']['source']],
                'insight': self._assess_envelope_quality(buildable_depth, property)
            })
        
        # Density / max units
        if property.get('lotSizeSqft') and rules.get('maxDensity'):
            lot_acres = property['lotSizeSqft'] / 43560
            max_units = int(lot_acres * rules['maxDensity']['value'])
            interpretations.append({
                'category': 'density',
                'field': 'maxUnits',
                'value': max_units,
                'unit': 'units',
                'calculation': f"{lot_acres:.3f} acres × {rules['maxDensity']['value']} units/acre",
                'sources': [rules['maxDensity']['source']],
                'insight': f"By-right maximum of {max_units} units on this parcel"
            })
        
        # FAR-based max
        if property.get('lotSizeSqft') and rules.get('maxFAR'):
            max_gfa = property['lotSizeSqft'] * rules['maxFAR']['value']
            interpretations.append({
                'category': 'density',
                'field': 'maxGFA',
                'value': max_gfa,
                'unit': 'sqft',
                'calculation': f"{property['lotSizeSqft']:,} sqft lot × {rules['maxFAR']['value']} FAR",
                'sources': [rules['maxFAR']['source']],
                'insight': f"Maximum gross floor area: {max_gfa:,.0f} sqft"
            })
        
        # Parking requirements
        if rules.get('parkingRatio') and 'maxUnits' in [i['field'] for i in interpretations]:
            max_units = next(i['value'] for i in interpretations if i['field'] == 'maxUnits')
            parking_needed = int(max_units * rules['parkingRatio']['value'])
            interpretations.append({
                'category': 'parking',
                'field': 'parkingRequired',
                'value': parking_needed,
                'unit': 'spaces',
                'calculation': f"{max_units} units × {rules['parkingRatio']['value']} spaces/unit",
                'sources': [rules['parkingRatio']['source']],
                'insight': f"{parking_needed} spaces required at {rules['parkingRatio']['value']} per unit"
            })
        
        # Controlling constraint
        controlling = self._find_controlling_constraint(interpretations, rules, property)
        interpretations.append({
            'category': 'summary',
            'field': 'controllingConstraint',
            'value': controlling['factor'],
            'insight': controlling['explanation'],
            'sources': controlling['sources']
        })
        
        return interpretations
```

### Task T008: Insight Generation (the "So What")

Beyond math, the Agent should generate a one-line insight for each category:

| Category | Raw Data | Interpreted Insight |
|----------|----------|-------------------|
| Envelope | Setbacks: 0/10/20 | "Your buildable footprint is 3,500 sqft — 65% of lot area. Above average for MRC districts." |
| Density | Max 109 units/acre | "By-right maximum: 13 units. Current zoning allows 3× the existing 4-unit structure." |
| Height | Max 225ft | "Height limit of 225ft (≈20 stories) is not the constraint here — density and lot size are." |
| Parking | 1.0 spaces/unit | "13 parking spaces needed. Surface parking would consume ~3,900 sqft — consider structured." |
| Summary | — | "Density is the controlling factor. Variance for additional units would require SAP review (§16-26.003)." |

This is where the Agent earns its value — it doesn't just regurgitate rules, it tells the user what matters for *their specific property*.

**Implementation:** This can be rule-based for common patterns (no LLM needed for standard calculations). Reserve LLM calls for edge cases: overlay districts, conditional uses, variance likelihood.

### Task T009: Source Citation Component

Build a reusable component that displays a zoning rule with its source link. This component appears everywhere — every tab, every rule, every calculation.

```tsx
// SourceCitedValue.tsx
// Reusable across all 7 tabs

interface SourceCitedValueProps {
  label: string;            // "Rear Setback"
  value: string | number;   // 20
  unit?: string;            // "ft"
  calculation?: string;     // "90ft lot - 20ft setback"
  insight?: string;         // "Buildable depth: 70ft"
  source: {
    sectionNumber: string;  // "16-06.007"
    sectionTitle?: string;  // "Dimensional Requirements"
    url?: string;           // Full Municode URL or search fallback
    sourceType: 'municode' | 'search' | 'planning';
  };
}

function SourceCitedValue({ label, value, unit, calculation, insight, source }: SourceCitedValueProps) {
  return (
    <div className="flex justify-between items-start py-2 border-b border-gray-100">
      <div>
        <span className="text-sm text-gray-500">{label}</span>
        <div className="font-semibold">{value}{unit && ` ${unit}`}</div>
        {calculation && (
          <span className="text-xs text-gray-400">{calculation}</span>
        )}
        {insight && (
          <p className="text-sm text-blue-700 mt-1">{insight}</p>
        )}
      </div>
      <SourceLink source={source} />
    </div>
  );
}

function SourceLink({ source }: { source: SourceCitedValueProps['source'] }) {
  if (!source.url && !source.sectionNumber) return null;
  
  const icon = source.sourceType === 'municode' ? BookOpen 
             : source.sourceType === 'planning' ? MapPin 
             : Search;
  
  const colors = source.sourceType === 'municode' ? 'text-violet-600 hover:text-violet-800'
               : source.sourceType === 'planning' ? 'text-emerald-600 hover:text-emerald-800'
               : 'text-gray-500 hover:text-gray-700';

  return (
    <a 
      href={source.url} 
      target="_blank" 
      rel="noopener noreferrer"
      className={`flex items-center gap-1 text-xs ${colors} whitespace-nowrap`}
      title={source.sectionTitle || `Section ${source.sectionNumber}`}
    >
      <Icon component={icon} size={12} />
      §{source.sectionNumber}
    </a>
  );
}
```

**Design principle:** The source link is always visible but unobtrusive — a small `§16-06.007` link in violet next to each value. Not a full button like T002's detection card — that level of prominence is right for the initial confirmation but too heavy for every field in the detailed tabs.

interface DevCapacityOutput {

### Task T014: Strategy Arbitrage Consumes Dev Capacity

When M08 calculates the BTS strategy score:

1. Check for DevCapacity output in the deal store
2. If present: use `maxUnits`, `estimatedTotalDevCost`, `developmentProbability` for real BTS scoring
3. If absent: flag BTS as "unscored" with prompt "Run Zoning Analysis to unlock Build-to-Sell scoring"
4. The `controllingConstraint` and `variancePotential` inform the arbitrage narrative: *"Zoning allows 3× current density (§16-06.005). BTS outscores Rental by 15 points."*

## What This Looks Like for the User

**Before (current):**
User opens Zoning tab → sees "MRC-3" with raw dimensional values → has to manually look up what the code says → does their own math to figure out what they can build → no way to verify the numbers without leaving the platform.

**After (this plan):**
User opens Zoning tab → sees "MRC-3" with applied calculations specific to their lot → every value has a `§16-06.007` link they can click to verify on Municode → the Agent explains what matters ("density is your constraint, not height") → Dev Capacity flows into Strategy Arbitrage so BTS scoring uses real zoning data → all of this traces back to specific code sections the user can independently verify.

The verification-first philosophy stays intact: the platform interprets the code, but the user can always trace any answer back to its authoritative source.
