# JEDI RE — ZONING AI AGENT
## Specialist Architecture: Learning, Reasoning & Application

---

## THE CORE PROBLEM

Zoning codes are the single most complex, fragmented, and misinterpreted data layer in real estate. Every municipality writes them differently. Cross-references are buried. Exceptions reference other exceptions. Overlay districts contradict base districts. And the people who understand them — zoning attorneys and experienced developers — charge $300-500/hour.

The current wireframe treats zoning as a data lookup. That's a mistake. Zoning isn't a lookup problem — it's a **reasoning problem**. The Zoning AI Agent needs to do what a senior zoning attorney does: read the code, understand the context, apply it to a specific parcel's geometry and location, identify edge cases, and recommend the optimal path.

This document redesigns the Zoning AI Agent as a true specialist that learns, reasons, and improves.

---

### The Three Capabilities

1. **LEARN** — Ingest, structure, and continuously update zoning code knowledge
2. **REASON** — Apply codes to specific parcels using multi-step logic chains
3. **IMPROVE** — Get smarter from corrections, outcomes, and professional feedback

### Step 2: The Dual-Layer Knowledge System

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  DUAL-LAYER KNOWLEDGE ARCHITECTURE                                                      │
│                                                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────────────────┐ │
│  │  LAYER A: STRUCTURED RULES (PostgreSQL + JSONB)                                     │ │
│  │  ════════════════════════════════════════════                                       │ │
│  │                                                                                     │ │
│  │  • ZoningDistrictProfile objects (as shown above)                                   │ │
│  │  • Deterministic lookups: "What's the FAR in MRC-3?" → 3.2 (instant, 100%)         │ │
│  │  • Parking calculations: structured formula evaluation                              │ │
│  │  • Setback computations: rule-based geometry                                        │ │
│  │  • Use permission checks: by-right / conditional / prohibited                       │ │
│  │                                                                                     │ │
│  │  WHEN TO USE: Any question that has a single, deterministic answer                  │ │
│  │  in the code. No LLM call needed. Fast. Cheap. Always accurate.                     │ │
│  │                                                                                     │ │
│  └─────────────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────────────────┐ │
│  │  LAYER B: REASONING ENGINE (RAG + LLM)                                              │ │
│  │  ═════════════════════════════════════                                              │ │
│  │                                                                                     │ │
│  │  • Vector-embedded full code text (Pinecone / pgvector)                              │ │
│  │  • Cross-reference resolution: "Does the BeltLine overlay override MRC-3 setbacks?" │ │
│  │  • Ambiguity interpretation: "The code says 'may' — is this discretionary?"         │ │
│  │  • Hypothetical analysis: "Could I get a variance for 320 units?"                   │ │
│  │  • Code comparison: "How does Atlanta's MRC-3 compare to Nashville's MU-D?"         │ │
│  │  • Exception identification: "Are there any conditions that would block this use?"  │ │
│  │                                                                                     │ │
│  │  WHEN TO USE: Multi-step reasoning, cross-references, ambiguity,                    │ │
│  │  hypotheticals, or anything that requires INTERPRETATION not just lookup.            │ │
│  │                                                                                     │ │
│  └─────────────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                         │
│  ┌─────────────────────────────────────────────────────────────────────────────────────┐ │
│  │  QUERY ROUTER — Decides Which Layer Handles Each Question                           │ │
│  │                                                                                     │ │
│  │  User Query                                                                         │ │
│  │      │                                                                              │ │
│  │      ▼                                                                              │ │
│  │  Intent Classifier                                                                  │ │
│  │      │                                                                              │ │
│  │      ├── LOOKUP intent → Layer A (structured rules)                                 │ │
│  │      │   "What's the height limit in MRC-3?"                                        │ │
│  │      │   "How much parking for 320 units?"                                          │ │

│  │      │   Response: immediate, deterministic, with code citation                     │ │
│  │      │                                                                              │ │
│  │      ├── CALCULATION intent → Layer A + geometry engine                              │ │

│  │      │   "What's the buildable envelope after setbacks?"                            │ │

│  │      │   Response: structured data as context → LLM reasons over it                 │ │
│  │      │                                                                              │ │

│  │          "What can I build at 847 Peachtree?"                                       │ │

│  │          Response: complete ZoningAnalysis object with all dimensions,               │ │
│  │          uses, parking, incentives, overlays, risks, and recommendations             │ │
│  │                                                                                     │ │
│  └─────────────────────────────────────────────────────────────────────────────────────┘ │
│                                                                                         │
│  OPTIMIZATION IMPACT:                                                                   │
│  • 60-70% of queries hit Layer A only (fast, cheap, deterministic)                      │
│  • 20-25% need Layer A + B (medium cost, high accuracy)                                 │
│  • 5-10% need full APPLICATION pipeline (higher cost, maximum intelligence)             │
│  • Average response time drops from 3-5 sec → 200ms-1.5 sec                            │
│  • LLM token cost drops 70% (most queries don't need it)                                │
│                                                                                         │

```

### Step-by-Step Application Logic

```
┌─────────────────────────────────────────────────────────────────────────────────────────┐
│  STEP 1: IDENTIFY ALL RULES THAT APPLY                                                  │
│  ══════════════════════════════════════                                                  │
│                                                                                         │
│  For parcel at 847 Peachtree St NE:                                                     │
│                                                                                         │
│  a) Base zoning district → MRC-3 (from GIS spatial query)                               │
│  b) Overlay districts → SPI-16 Midtown (spatial intersection)                           │
│  c) Special districts → None (check historic, conservation, etc.)                       │
│  d) Proximity triggers:                                                                 │
│     → 0.3 mi from MARTA station → parking reduction eligible                           │
│     → Not within BeltLine overlay → no BeltLine requirements                            │
│     → Not in floodplain → no FEMA restrictions                                          │
│     → Not in historic district → no preservation requirements                           │
│  e) Recent amendments → Check if any MRC-3 rules changed since last extraction          │
│                                                                                         │
│  RESULT: Rule stack = [MRC-3 base] + [SPI-16 overlay] + [transit proximity]             │
│                                                                                         │
│  STEP 2: APPLY BASE DISTRICT RULES TO PARCEL GEOMETRY                                   │
│  ═══════════════════════════════════════════════════                                     │
│                                                                                         │
│  Parcel: 2.25 acres (98,010 SF), irregular polygon                                      │
│                                                                                         │
│  a) Max density: 109 units/acre × 2.25 acres = 245 units (by-right)                    │
│  b) Max height: 225 ft                                                                  │
│  c) Max FAR: 3.2 × 98,010 SF = 313,632 SF gross floor area                             │
│  d) Max lot coverage: 85% × 98,010 SF = 83,309 SF footprint                            │
│  e) Setbacks applied to parcel polygon:                                                 │
│     → Front (0 ft): full frontage buildable                                             │
│     → Side (10 ft): reduced by 20 ft total                                              │
│     → Rear (20 ft): reduced by 20 ft                                                   │
│     → Net buildable footprint: 78,240 SF                                                │
│  f) Cross-check: adjacent to R-5 district on east side?                                 │
│     → YES → side setback increases to 25 ft on that side                                │
│     → Revised buildable footprint: 76,890 SF                                            │
│     → ⚠️ This is a cross-reference catch most users would miss                          │
│                                                                                         │
│  STEP 3: APPLY OVERLAYS + EXCEPTIONS                                                    │
│  ════════════════════════════════════                                                    │
│                                                                                         │
│  SPI-16 Midtown overlay adds:                                                           │
│  a) Min 60% of ground floor must be "active use" (retail, restaurant)                   │
│  b) Streetscape requirements: 20ft clear zone along Peachtree                           │
│  c) Design review committee approval required (adds ~2 months)                          │
│  d) No blank walls > 30 ft on street-facing facades                                     │
│                                                                                         │
│  Does SPI-16 OVERRIDE any MRC-3 rules?                                                  │
│  → Yes: front setback explicitly 0 ft (confirms MRC-3)                                  │
│  → No height override (MRC-3's 225 ft applies)                                          │
│  → Additional: ground-floor transparency requirement (60% glazing)                      │
│                                                                                         │
│  STEP 4: CALCULATE BUILDABLE CAPACITY                                                   │
│  ════════════════════════════════════                                                    │
│                                                                                         │
│  Three scenarios computed:                                                               │
│                                                                                         │
│  BY-RIGHT:                                                                              │
│  → Units: min(245 by density, 313632/900sf avg = 348 by GFA) = 245 units               │
│  → Density is the binding constraint, not FAR                                           │
│  → Parking: 245 × 1.0 + 245 × 0.25 = 306 spaces (before reductions)                   │
│  → Transit reduction: 306 × 0.80 = 245 spaces required                                 │
│  → Parking structure: ~82,000 SF (at 330 SF/space)                                      │
│                                                                                         │
│  VARIANCE (SAP):                                                                        │
│  → Target: 320 units (30% above by-right)                                               │
│  → FAR check: 320 × 900 = 288,000 SF < 313,632 SF max → FAR OK                        │
│  → Parking: 320 × 1.0 + 320 × 0.25 × 0.80 = 384 spaces                               │
│  → Height check: 320u in 225ft = feasible at 16 units/floor × 20 floors                │
│                                                                                         │
│  REZONE (MRC-4):                                                                        │
│  → Units: 185/acre × 2.25 = 416 units                                                  │
│  → Height: 300 ft                                                                       │
│  → FAR: 4.8 × 98,010 = 470,448 SF                                                     │
│  → Parking: 416 × 1.0 + 416 × 0.20 × 0.80 = 483 spaces                               │
│                                                                                         │
│  STEP 5: IDENTIFY INCENTIVE PROGRAMS                                                    │
│  ═══════════════════════════════════                                                     │
│                                                                                         │
│  Applicable to this parcel:                                                              │
│  ✅ Density bonus: +15% if 10% units at 80% AMI                                         │
│     → By-right: 245 → 281 units (+36)                                                   │
│     → Cost: 24.5 affordable units at reduced rent = ~$180k/yr NOI reduction             │
│     → Benefit: 36 additional market-rate units = ~$10.1M value                           │
│     → NET BENEFIT: +$8.2M (massive positive)                                            │
│  ✅ Transit parking reduction: -20% parking (already applied above)                      │
│  ❌ LEED height bonus: not applicable in MRC-3 (only MRC-4+)                             │
│  ❌ Historic tax credits: not in historic district                                       │
│                                                                                         │
│  STEP 6: ASSESS ENTITLEMENT PATHS + RISK                                                │
│  ════════════════════════════════════════                                                │

## UI OPTIMIZATIONS (What Changes in the Wireframe)

### With a specialist agent, the UI simplifies:

```
BEFORE (Tab 1: Zoning Lookup — dumb data display):
  User sees: static cards showing district parameters
  User does: reads, interprets, cross-references manually

AFTER (Tab 1: Zoning Intelligence — agent-powered):
  User sees: conversational analysis with confidence badges
  Agent does: all 8 application steps automatically
  User gets: "Here's what you can build, here's the best path, here's the risk"

BEFORE (Tab 2: Entitlement Tracker — manual Kanban):
  User does: manually moves cards, updates timelines
  
AFTER (Tab 2: Entitlement Tracker — agent-monitored):
  Agent does: scrapes permit portal for status changes
  Agent does: detects hearing agenda items via News Agent
  Agent does: auto-updates timeline based on precedent library
  Agent does: alerts user when predicted timeline diverges from actual

BEFORE (Tab 3: Development Capacity — three static scenarios):
  User sees: pre-computed by-right / variance / rezone numbers
  
AFTER (Tab 3: Development Capacity — dynamic optimization):
  Agent does: identifies the density bonus program most users miss
  Agent does: computes the OPTIMAL path (not just three scenarios)
  Agent does: sequences entitlement steps for maximum risk-adjusted value
  Agent says: "Don't rezone. Use density bonus + SAP. 80% of the value, 1/3 risk."

BEFORE (Tab 6: Time-to-Shovel — generic municipal averages):
  User sees: P25/P50/P75 based on all applications in the database
  
AFTER (Tab 6: Time-to-Shovel — precedent-calibrated):
  Agent does: filters to comparable projects (same district, same scale, same type)
  Agent does: adjusts for trend ("approvals slowed 18% since new engagement req")
  Agent does: identifies the specific decision-maker patterns for this district
  Agent says: "Your SAP will be heard by the same BZA panel that approved 4 of 5
              similar requests in 2025. But they added affordable unit conditions
              to all 4. Budget for that."
