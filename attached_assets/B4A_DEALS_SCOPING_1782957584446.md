# DISPATCH — B4a: DEALS SCOPING TO ORG (READ ISOLATION)

**Second-to-last of the finish arc (B3 ✅ → B4a → B4b → B5). B4a is the tractable half of B4:**
deals already have `org_id`, so this applies org-scoping to the deal-read sites — extending the
cross-org isolation pattern B2c just PROVED (the `requireOrgRoleForOrg` membership bind, B2c #5) from
activity endpoints to the core data a member works with. B4b (properties — the structural half with
the market-data split) is separate and next.
**What B4a does:** a member sees only THEIR org's deals. Every deal read/list/access filters by the
caller's org. No cross-org deal leakage.
**Repo:** `Nardo758/JediRe.git` — branch `claude/b4a-deals-scoping`.
**Verification rule (S1-01):** proven LIVE — a member of org A cannot read org B's deals (list OR
by-id), and a member sees exactly their org's deals. Tenant boundary on core data.
**Report to:** `docs/audits/B4A_DEALS_SCOPING_VERDICT.md`

## PHASE 1 — ENUMERATE THE DEAL-READ SITES (READ-ONLY, STOP)

**1. Every deal read/list/access site.** Grep the deal routes/services for reads:
```
grep -rniE "from deals|join deals|deals\b.*(select|where)|getDeal|listDeal|findDeal|dealService" backend/src --include=*.ts
```
Classify each: LIST (returns many deals — must filter by org), BY-ID (returns one deal — must verify
the deal's org matches caller's org), WRITE (create/update — must set/verify org). `file:line` each.
The prior work said ~20 sites — confirm the real count.

**2. The current scoping (or lack of it).** For each read site: does it filter by `user_id` today,
by nothing, or already by `org_id`? B2c found `requireOrgRoleForOrg` exists and binds caller→org.
Confirm whether deal reads currently use ANY org filter or are user-scoped / unscoped. `file:line`.

**3. The resolution to reuse.** B4a scopes deals to the caller's ACTIVE org — which is
`session_active_org ?? default_org_id` (the B2b resolution, `resolveOrgForUser`/orgCreditService).
Confirm the helper to call so every deal query scopes to the same org the entitlement gate uses
(consistency: a member's deals and their credit pool resolve to the SAME org). `file:line`.

**4. The two read shapes:**
   - **LIST** → `WHERE org_id = <caller's active org>` appended. Returns only the org's deals.
   - **BY-ID** → after fetching the deal, verify `deal.org_id === caller's active org`; 403/404 if
     not (a member requesting another org's deal by guessing the id must be rejected). This is the
     per-record isolation, mirroring B2c's cross-org check.
   Design both patterns; confirm which sites are which.

## PHASE 2 — APPLY ORG SCOPING (after design approved)

1. **LIST sites** → add `WHERE org_id = <caller active org>`. Every deal-list returns only the
   caller's org's deals. `file:line` each.
2. **BY-ID sites** → after fetch, verify the deal's `org_id` matches the caller's active org; reject
   (403/404) if not. `file:line` each.
3. **WRITE sites** → create sets `org_id` = caller's active org (already partially done — B2b/inline-
   deals sets org on create; confirm and align); update verifies the deal belongs to the caller's org
   before mutating. `file:line`.
4. **Reuse the resolution** — every site scopes to the SAME active-org the entitlement gate uses
   (`resolveOrgForUser`). Do NOT introduce a second org-resolution path. `file:line`.
5. **Consistency with B2c isolation** — where `requireOrgRoleForOrg` or an equivalent org-membership
   check already exists (org-scoped routes), reuse it rather than re-implementing the boundary.
