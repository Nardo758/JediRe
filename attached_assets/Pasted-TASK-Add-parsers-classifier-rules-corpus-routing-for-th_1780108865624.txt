STRATEGIC CONTEXT — READ BEFORE STARTING
═══════════════════════════════════════════════════════════════════════════

This dispatch operationalizes the operator-upload Layer 2 ingestion path
per CLAUDE.md P10 (Data Sourcing Hierarchy). The platform's broader
sale-comp acquisition strategy decision (state-level PT-61/GSCCCA vs
commercial aggregator vs operator-upload-primary) is deferred per separate
operator review.

This dispatch's work is strategy-independent: regardless of which acquisition
strategy is eventually selected, operator-uploaded CoStar files must be
parsed and ingested. Do not treat this dispatch as committing the platform
to operator-upload-primary as the canonical strategy.

PROPERTY REFACTOR COUPLING

Output flows to the historical_observations corpus per §7.8, NOT directly
to market_sale_comps or market_rent_comps. This is intentional — the
corpus is the long-term destination and survives the property plumbing
refactor cleanly.

Two known gaps this dispatch does NOT solve:

1. CoStar PID ↔ JEDI property_id mapping is deferred. Until mapping exists,
   corpus rows have comp_for_subject_pid (raw CoStar PID) but cannot be
   linked to JEDI properties.id in the unified model.

2. Whether corpus rows ALSO flow through to property_sales /
   property_operating_data in the property refactor's Phase 2 backfills
   is an open architectural question. Surface to operator if the audit
   reveals this matters for Phase 2 implementation.

Neither blocks this dispatch from completing. Flag both as findings if
encountered.

PHASE 2 — IMPLEMENTATION (AFTER HUMAN APPROVAL)
═══════════════════════════════════════════════════════════════════════════

2A. EXTEND THE DOCUMENT TYPE ENUM

   Add to the DocumentType enum (per spec §7.3, split as noted in this dispatch):
     COSTAR_SUBMARKET_EXPORT   — DataTable.xlsx pattern
     COSTAR_SALE_COMPS         — Near_By_Sales_PID_*.xlsx pattern
     COSTAR_RENT_COMPS         — Rent_Comp_Properties_PID_*.xlsx pattern

2B. EXTEND THE CLASSIFIER

   Two-pass classification — filename is cheap, columns are definitive.

   Pass 1 — filename pattern:
     /^DataTable\.xlsx$/i              → COSTAR_SUBMARKET_EXPORT (provisional)
     /^Near_?By_?Sales_PID_\d+\.xlsx$/i → COSTAR_SALE_COMPS
     /^Rent_?Comp_?Properties_PID_\d+\.xlsx$/i → COSTAR_RENT_COMPS

   Pass 2 — column header signature (confirm provisional, or classify when
   filename was renamed by operator). Read row 1 of sheet 1:
     COSTAR_SUBMARKET_EXPORT — must contain ALL of:
       Period, Vacancy Rate, Market Asking Rent/Unit, Market Cap Rate
     COSTAR_SALE_COMPS — must contain ALL of:
       Address, Sale Date, Sale Price, Price/Unit, Dist (mi)
     COSTAR_RENT_COMPS — must contain ALL of:
       Property Id, % Overlap with Subject Property, Rent/SF, Occ %, Concess %

   If filename matches but columns don't, the file is malformed — return
   UNKNOWN and surface a user-facing error. Don't silently misclassify.

   Operator override: the upload UI already supports doctype override per
   spec §7.3. Wire these three new types into the override dropdown.

2C. PARSER 1 — parseCoStarSubmarket()

   Input: Buffer | filepath. Output: array of M1-tier rows ready for corpus.

   Per-row logic for DataTable.xlsx:
   - Parse Period column. Three shapes to recognize:
       "2024 Q3"        → historical (load)
       "2026 Q2 EST"    → forecast (SKIP — these are projections, not observations)
       "2026 Q2 QTD"    → partial quarter (SKIP — incomplete data)
     Only historical complete-quarter rows become corpus observations. EST/QTD
     rows are noted in the parse log but not written.
   - Convert "-" in numeric columns to NULL, not 0. Affects: Under Constr,
     12 Mo Sales Vol, 12 Mo Sales Vol Growth, Cap Rate.
   - Growth rate columns are already decimals (0.0875 = 8.75%). Do not
     re-divide by 100.
   - Submarket identity: this export carries no submarket name in the data.
     The upload UI must capture submarket_id at upload time. Parser signature
     takes submarketId as a required parameter; throw if missing.
   - Output row shape (per spec §7.8):
       { observation_date: YYYY-MM-DD (quarter-end),
         geography_level: 'submarket',
         submarket_id,
         source_tier: 'M1',
         source: 'CoStar',
         redistribution_restricted: true,
         payload: { vacancy_rate, asking_rent_per_unit, ... } }

2D. PARSER 2 — parseCoStarSaleComps()

   Per-row logic for Near_By_Sales_PID_<pid>.xlsx:
   - Subject PID parsed from filename (regex group). All comps in this file
     are comps FOR this subject PID.
   - Sale Date format: "Jun 2024" — parse with %b %Y; day defaults to
     first of month. If empty, skip row with warning.
   - Sale Price and Price/Unit can have commas; strip before parsing.
   - Cap Rate often blank — load as NULL.
   - Yr Blt/Renov format: split on "/". Examples:
       "2017"        → year_built=2017, year_renovated=NULL
       "2017/-"      → year_built=2017, year_renovated=NULL
       "1925/2024"   → year_built=1925, year_renovated=2024
   - Submarket column populated per row — use directly.
   - Output row shape:
       { observation_date: <sale date>,
         geography_level: 'parcel',
         event_subtype: 'sale',
         comp_for_subject_pid: 9600360,
         source_tier: 'C1',
         source: 'CoStar',
         redistribution_restricted: true,
         payload: { address, name, units, sale_price, price_per_unit,
                    price_per_sf, cap_rate, distance_mi, year_built,
                    year_renovated, submarket } }

2E. PARSER 3 — parseCoStarRentComps()

   Per-row logic for Rent_Comp_Properties_PID_<pid>.xlsx:
   - Row 1 (where Map # = "-") is the SUBJECT property. Property Id in row 1
     matches the PID from the filename. Subject row is parsed but flagged:
       payload.is_subject = true
   - Subsequent rows are comps. Map # is the CoStar-assigned comp number.
   - Per-bedroom rent columns (Studio, 1 Beds, 2 Beds, 3 Beds) often have "-"
     when a property doesn't have that bedroom type. Treat as NULL.
   - Unit counts per bedroom type (#Studio, #1 Beds, #2 Beds, #3 Beds): when
     zero, write 0 (not NULL — zero means the property has no studios, which
     is information).
   - "% Overlap with Subject Property" — preserve as comp_weight (already
     0–1 scale; row 1 self-overlap = 1).
   - Occ % and Concess % — already decimals (0.914, 0.115). Do not re-divide.
   - Output row shape:
       { observation_date: <upload date>,  // snapshot date of the export
         geography_level: 'parcel',
         comp_for_subject_pid: 9600360,
         comp_weight: <% Overlap>,
         source_tier: 'C1',
         source: 'CoStar',
         redistribution_restricted: true,
         payload: { property_id, building_name, address, units, stories,
                    year_built, year_renovated, avg_sf, distance_mi,
                    rent_per_sf, rent_per_unit, rent_studio, rent_1br,
                    rent_2br, rent_3br, occ_pct, concession_pct,
                    unit_count_studio, unit_count_1br, unit_count_2br,
                    unit_count_3br, ad_level, views_60d, overlap_pct,
                    neighborhood, is_subject } }

2F. ROUTE FROM CLASSIFIER → PARSER → CORPUS

   Edit backend/src/services/document-extraction/data-router.ts:

═══════════════════════════════════════════════════════════════════════════
GENERAL RULE FOR CLAUDE GOING FORWARD — HOW TO HANDLE OPERATOR-UPLOADED
FILES OF THIS GENERAL SHAPE
═══════════════════════════════════════════════════════════════════════════

When an operator uploads any vendor-exported Excel/CSV that isn't yet in the
classifier:

1. Inspect first. Read sheet names + column headers + first 10 data rows
   before writing any parser. Don't pattern-match to "looks like a T12" and
   write code; verify the columns.

2. Identify three things explicitly:
   - The vendor and file family (CoStar submarket, CoStar property, Yardi
     Matrix, Berkadia survey, JLL, CBRE, internal rent roll, etc.)
   - The redistribution license posture (CoStar/Yardi = restricted;
     operator-internal = unrestricted)
   - The corpus tier per §7.3 (M1/M2 for market data, C1/C2 for comps)

3. Look for "subject vs comp" markers in the file. CoStar uses Map # = "-"
   for subject; other vendors use first-row tagging or filename PID. Don't
   collapse subject and comps into the same shape.

4. Distinguish historical / forecast / partial. Submarket time-series files
   often interleave actuals with projections (EST) and incomplete current
   periods (QTD). Only actuals become corpus observations; forecasts and
   partials are noted but not written.

5. Preserve vendor-specific signal columns. CoStar's % Overlap, Apartments.com
   Ad Level, Common Views 60d are not noise — they're the vendor's
   proprietary index values and should ride through to the payload, not be
   dropped.

6. Empty cells: "-" or "" in vendor exports almost always means NULL, not
   zero. Conversion is destructive in one direction (NULL → 0 implies
   knowledge that doesn't exist).

7. File naming patterns are useful but not authoritative. Operators rename
   files. Always fall through to column-signature validation.

8. Sample files for new vendor doctypes go into
   jedire/backend/test/fixtures/<vendor>/ as gold-standard. Each parser
   gets at least one fixture-based test asserting exact row count and
   redistribution flag.
