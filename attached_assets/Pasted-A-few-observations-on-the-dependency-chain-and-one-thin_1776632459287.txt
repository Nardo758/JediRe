A few observations on the dependency chain and one thing worth flagging before Replit starts pulling from this backlog:
On #242 dependencies — #240 is the right prerequisite. Tax and insurance rulesets must exist before CashFlow underwriting can produce defensible numbers. Without them, the agent either hardcodes FL math (violating ground rule #10) or generates tax figures without jurisdiction awareness (producing wrong numbers for TX and GA deals). #240 must fully ship before #242 touches code.
On #242 depending on #237 (Phase 5 Hardening) — worth a second look. Phase 5 is the foundation hardening for the 5-agent runtime. CashFlow Agent underwriting is an extension of Phase 4C, not something that waits for Phase 5 polish. The risk of making #242 wait for Phase 5 is you defer the most valuable institutional-grade capability behind ops work. Consider whether #242 actually needs #237 or just needs Phase 4C (CashFlow Agent baseline) + #240. If Phase 4C is already done when you get here, you might be able to start #242 before Phase 5 ships, in parallel with hardening work.
Missing prerequisite on #242: deal_monthly_actuals table. Tier 2 evidence (owned-portfolio actuals) is a core pillar of the underwriting spec, and that table is still listed as a blocker in CLAUDE.md. Without it, the CashFlow Agent falls back to Tier 3 for everything, which erases the institutional edge. Recommend adding that migration as an explicit dependency — either a sub-task of #242 or its own small task that blocks #242.
#239 (Deal Intake from Email) depending on #240 and #241 feels right. The email workflow needs document parsing (which benefits from the tax ruleset system when deals come in from non-FL markets) and scored-fit-against-profile logic (which works better when backed by archive benchmarks). But verify that #239 doesn't actually need the full underwriting capability — it should be able to create a deal draft and trigger DealContext assembly without full underwriting. If it's only doing intake + fit-score + draft, #239 can ship before #242. The underwriting happens downstream when the user reviews the draft.
One scoping question on #242: the spec includes both the evidence system AND the archive feedback loop (archive_assumption_benchmarks table, nightly aggregation job, archive query tools). These are technically separable. The evidence system is what makes individual underwriting defensible. The archive loop is what makes the platform get smarter over time. Both matter, but they have different urgency — evidence is immediate user value, archive is compounding-over-months value.
If #242 feels too large to ship as one task, consider splitting:

#242a — Evidence System: underwriting_evidence table, UnderwritingValue<T>, per-field reasoning, F9 evidence panel UX, collision reports. Shippable without archive.
#242b — Archive Feedback Loop: archive_assumption_benchmarks table, nightly aggregation, fetch_archive_* tools, systematic bias correction. Depends on #242a being live for a few weeks to start accumulating meaningful aggregates.

That split lets you ship user-visible value on #242a while #242b bakes in the background. Your call on whether that fits your roadmap.
