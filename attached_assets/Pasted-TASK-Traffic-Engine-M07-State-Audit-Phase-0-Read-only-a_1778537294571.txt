PHASE 1a — SERVICE INVENTORY WITH ROLE CLASSIFICATION

For every traffic-related service file, classify its role:

  Role                         Description
  ─────────────────────────    ─────────────────────────────────
  CALIBRATION                  Updates coefficients from evidence
                                (e.g., trafficCalibrationJob)
  PREDICTION                   Produces forward projections
                                (e.g., trafficPredictionEngine)
  EXTRACTION                   Pulls signals from source documents
                                (e.g., rent roll → velocity)
  CONSUMPTION_BRIDGE           Applies M07 outputs to downstream
                                modules (e.g., trafficToProForma)
  CORRELATION                  Maps traffic signals to outcomes
                                (e.g., traffic-correlation)
  DATA_SOURCE                  External data fetching
                                (e.g., traffic-data-sources)
  ANALYTICS                    UI / API support layer
  LEARNING                     Subject-history accumulation
  M35_BRIDGE                   Event integration contract
                                (out of scope per boundary above —
                                see #715)
  ROUTE                        HTTP endpoints

Cite the line ranges where the role function lives. Note whether
each service is INVOKED (called from somewhere live) or ORPHANED
(defined but no caller). Orphaned services are spec'd-but-unwired
signals.

PHASE 1b — CALIBRATION STATE

The spec calls for three-layer Bayesian resolution:
Baseline → Platform → Deal. For each layer:

  Baseline layer: BASELINE_COEFFICIENTS in trafficCalibrationJob.ts
    Document each coefficient (visibility_capture_rate,
    apartment_seeker_pct, stop_probability, walkin_to_tour,
    tour_to_app, app_to_signed) — its baseline value, its
    interpretation, and whether the Bayesian update path uses it
    as prior.

  Platform layer: traffic_calibration_factors table
    Document the schema (msa, submarket, class, vintage_band
    bucket structure). Verify nightly job writes rows with
    Bayesian posterior = (prior × n_prior + evidence × n_evidence)
    / (n_prior + n_evidence). Sample-size thresholds:
    MIN_PROPERTIES_TTM=5, MIN_PROPERTIES_SPARSE=2.

  Deal layer: Subject-First Calibration rule
    Per CLAUDE.md, subject history is promoted above peer when
    sufficient. Find the implementation. What's the threshold
    for promotion (months of subject history? n snapshots?).
    Is it actually firing for properties with multi-snapshot
    rent rolls? Cite the function + line.

For each layer, classify:
  ✓ FULLY_IMPLEMENTED  — code exists, invoked, tested
  ⚠ PARTIAL            — code exists, partial invocation
  ✗ SPEC_ONLY          — described in spec, no code
  ❓ UNVERIFIABLE       — code present but no way to confirm it
                          runs without live DB introspection

PHASE 1d — EXTRACTOR INVENTORY

Per CLAUDE.md, single-snapshot extractors should produce four
outputs from one rent roll: velocity, seasonality, waterfall,
term mix. The userMemory notes "Single-snapshot rent roll
extractors validated and built."

Verify by finding each extractor:
  - Velocity extractor: lease signing rate from lease_start_date
    fields in a single rent roll
  - Seasonality extractor: distribution of lease_start_dates
    across the calendar year
  - Waterfall extractor: lease term × signing date distribution
  - Term mix extractor: distribution of lease term lengths

For each extractor:
  - File and function name
  - Input shape (rent roll snapshot format)
  - Output shape (which T-N signal it produces)
  - Where the output is consumed (M07 prediction? Display only?)
  - Test coverage status (test file exists or not — don't run tests)

Separately, classify diff-extractors (require 2+ snapshots):
existence and trigger conditions.

PHASE 1e — OPERATIONAL MODES

Three modes spec'd: Stabilized, Lease-Up, Redevelopment. For each
mode:
  - Where is mode detected? (logic that selects which mode applies)
  - Which formulas/coefficients change per mode?
  - Where does the mode-selection code live (file:line)?

If mode selection is absent (all properties processed identically),
that's a SPEC_GAP finding. If it exists but is hardcoded to one
mode, that's PARTIAL.
