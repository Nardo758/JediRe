PHASE 8 — RESEARCH AGENT ENRICHMENT (Web + Google Places, real impl)

CONTEXT
Phase 8 replaces the orchestrator's web search and Google Places 
stubs with real implementations. Output feeds property_descriptions: 
narrative description, amenity flags, photos, reviews, sentiment 
signals.

CRITICAL FRAMING: This is the input pipeline for Cashflow Agent 
accuracy. The Cashflow Agent depends on:
- Amenity flags for rent premium adjustments  
- Reviews + sentiment for F52 buy-signal detection
- Photos for visual quality grading
- Narrative for strategy arbitrage context
- Recent news for risk adjustments

Sloppy or hallucinated outputs degrade Cashflow Agent accuracy. 
The data quality bar is structural reliability, not just presence.

Scope: ALL resolvable properties. Archive backfill prioritized 
(296 properties are the Cashflow Agent validation cohort).

═══════════════════════════════════════════════════════════════════
PRINCIPLES
═══════════════════════════════════════════════════════════════════

1. STRUCTURED OUTPUT, NOT FREE TEXT. Every Research Agent output 
   field has a defined schema. Amenity flags are booleans, not 
   narrative mentions. Reviews are extracted entities with sentiment 
   scores, not raw text. Narrative is the only free-form field, 
   and it's bounded (length cap, citation requirement).

2. CITATIONS MANDATORY. Every claim in the narrative is traceable 
   to a source URL. No claim without citation. The web search 
   results are the citation set; if no source supports a claim, 
   the claim is omitted.

3. EXTRACTION OVER GENERATION. Where possible, extract structured 
   facts (amenities, reviews, photos) directly rather than letting 
   the LLM synthesize. Synthesis is reserved for narrative only.

4. STANDING PRINCIPLES FROM PRIOR DISPATCHES APPLY:
   - Producer/reader contract: writes to property_descriptions 
     must update every field readers display
   - Schema errors surface (no .catch(logger.warn) swallowing)
   - Paired-read verification before complete

5. INVESTIGATION FIRST. The current stubs return 
   not_yet_implemented. The shape of what replaces them depends 
   on what each data source actually returns. Step 1 establishes 
   the real shape.

STEP 3 — GOOGLE PLACES ADAPTER
═══════════════════════════════════════════════════════════════════

Location: src/services/research/google-places/

A. Call sequence per property:
   1. Find Place by name + address → place_id
   2. Place Details lookup → name, formatted_address, 
      formatted_phone_number, website, types, rating, user_ratings_
      total, opening_hours
   3. Place Reviews → up to 5 most relevant + most recent reviews 
      with sentiment
   4. Place Photos → up to 5 photo references

B. STRUCTURED EXTRACTION
   - Amenity flags: parse the `types` array + photo captions for 
     known amenity signals. Maintain a mapping table of Places 
     types → amenity boolean fields.
   - Reviews: each review extracted as { author, rating, text, 
     time, sentiment_score, named_entities, hazard_mentions, 
     amenity_mentions }
   - Sentiment summary: aggregate review sentiment, weighted by 
     recency

C. NLP PASS ON REVIEWS
   - Second NLP pass on review text extracts:
     * Named entities (named amenities, neighborhood landmarks)
     * Hazard mentions (crime, noise, pests, parking issues, 
       management complaints)
     * Amenity gaps (complaints about missing features)
   - These feed F52 sentiment-rent divergence detection

D. ERROR HANDLING
   - API errors (rate limit, quota exceeded) → fail the step 
     explicitly, record in enrichment_log, do NOT swallow
   - No match found → success with empty result + 
     enrichment_log entry noting "no_places_match"
   - Place ID retrieval failure for downstream calls → partial 
     success, log which sub-call failed

STEP 4 — WEB SEARCH ADAPTER
═══════════════════════════════════════════════════════════════════

Location: src/services/research/web-search/

A. Call sequence per property:
   1. Search query: "<property_name> <city> <state>"
   2. Search query: "<property_address> apartment"
   3. Search query: "<property_name> management OR ownership"
   4. Search query: "<property_name> news OR announcement"

B. RESULT EXTRACTION (structured, not free-text)
   - Property website URL (if found)
   - ApartmentRatings / ApartmentList / similar review aggregator 
     URLs
   - News articles in last 2 years (title, URL, summary, date)
   - Recent events (renovations, ownership changes, capex 
     announcements)
   - Citation set built from result URLs

C. NARRATIVE GENERATION (single LLM call, bounded)
   - Input: property name, address, structured facts from Places, 
     web search results
   - Output: 2-3 paragraph description with inline citations
   - Constraints:
     * Length cap: 300 words max
     * Every claim cites a source URL from the search results
     * No claims without supporting source
     * Tone: factual, no marketing language
   - The LLM is given strict instructions: "Do not invent. Cite 
     or omit."

D. SAME ERROR HANDLING DISCIPLINE AS PLACES
