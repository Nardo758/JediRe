Why the dispatch is shaped this way
Four explicit guardrails against patterns we've seen go sideways:
1. Investigation gate before code. The bot's recent dispatches have benefited from "stop and report before patching" structure. Step 1 enforces this: backlog count, data shape, schema gaps — reported before any code.
2. Stub-correctness requirement. Stubs that return null or fake success silently corrupt downstream work. Stubs that return { status: 'not_yet_implemented' } make the gap visible. Same pattern that would have prevented the year_built ghost-row bug.
3. Paired-read verification. The producer-writes/reader-reads-elsewhere pattern has now bitten 6+ times. Step 5 makes the paired-read check mandatory before marking the dispatch complete. Not optional, not "I'll verify later."
4. Explicit DO NOT list. Bot has sometimes accumulated workarounds or "while I'm here" improvements. The list scopes the work tightly so the build stays surgical and the next dispatches' scope stays clean.
