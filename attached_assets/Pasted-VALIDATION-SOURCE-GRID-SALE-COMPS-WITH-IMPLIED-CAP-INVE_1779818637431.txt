STEP 2 — IMPLIED CAP RATE FROM SALE COMPS
═══════════════════════════════════════════════════════════════════

This is the sophisticated underwriting validation method. 
Investigation focus:

A. Source data for sale comps:
   - What sale comp tables exist? (per inventory: there are sale 
     comp paths but coverage may vary)
   - What fields per comp (sale price, sale date, observed cap 
     rate disclosed, gross income, unit count, sqft, MSA, 
     submarket, asset class, vintage)
   - How fresh is the data?

B. Platform expense norms application:
   - line_item_benchmarks already exists with per-unit OpEx 
     distributions (per Day 4 of last week's data quality 
     recovery)
   - Apply platform expense norms to each comp:
     comp_implied_NOI = comp_GPR_estimate - 
                         platform_OPEX_at_comp_class_geo
   - Where comp_GPR_estimate comes from comp's reported avg 
     rent × unit count, or other available data
   - implied_cap = comp_implied_NOI / comp_sale_price

C. Triangulation:
   - For each comp, compute both observed_cap (when disclosed) 
     and implied_cap (using platform norms)
   - Divergence between them = signal about:
     * Buyer underwrote different OpEx than platform suggests 
       (possibly more aggressive)
     * Buyer paid for upside / value-add potential
     * Disclosed cap is post-tax-adjustment vs implied is 
       pre-adjustment
   - Surface the divergence with explanation

D. Distribution from N comps:
   - For the subject deal's submarket × asset class × vintage 
     bucket, gather N comps
   - Distribution of implied caps (P25/P50/P75)
   - Surface this distribution alongside the subject's going-in 
     cap assumption
   - Operator can immediately see if subject is being 
     underwritten aggressively (cap below P25) or 
     conservatively (above P75)

STEP 4 — VALIDATION GRID DESIGN
═══════════════════════════════════════════════════════════════════

A. Per-assumption grid layout:
   - Rows: assumptions (the ~22-40 multifamily existing 
     assumptions from Phase 2)
   - Columns: validation methods (Trailing / Broker / Platform / 
     Sale Comps / Archive Benchmarks / LVE / Manual Override)
   - Cells: the value each method produces, with confidence 
     score and evidence link
   - Color coding: green when methods converge, amber when 
     soft divergence, red when hard divergence

B. Per-cell information:
   - Method name
   - Computed value (with units)
   - Confidence score (0-1)
   - Evidence reference (link to underwriting_evidence row)
   - "Why" explanation (which sources / methods produced this)

C. Reasoning visibility:
   - For the subject's resolved value (the value actually used 
     in the model), show which method was chosen and why
   - Show the state-conditional rule that selected the source 
     ("New Construction → Platform stabilized cohort primary")
   - Allow operator to override and see the impact

D. Sigma checks:
   - For each method's value, show the position within the 
     benchmark distribution (P25 / P50 / P75)
   - Flag outliers explicitly
   - Allow drill-down to see the underlying benchmark data
