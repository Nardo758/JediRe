PPV PHASE 1.3 — DATA LIBRARY UPLOAD UI

CONTEXT
Apartment Locator adapter is live as one upstream source. Data Library 
uploads are the second planned source. The orchestrator's source-
adapter pattern is already established (Task B). This dispatch adds:

1. Browser drag-drop upload UI at /archive/upload  
2. Direct-to-R2 upload via signed URLs (no backend body bottleneck)
3. File registration endpoint (POST /api/v1/archive/files/register)  
4. Data Library source adapter for the orchestrator

After this dispatch, users can drag a property folder onto a web 
page and watch files flow through the same orchestrator that 
Apartment Locator scrapes go through.

═══════════════════════════════════════════════════════════════════
PRINCIPLES
═══════════════════════════════════════════════════════════════════

1. SAME ORCHESTRATOR, NEW ADAPTER. Don't build a parallel ingest 
   path. The Data Library adapter normalizes to IntakeJob payload, 
   the orchestrator handles the rest. Same idempotency, same retry, 
   same paired-read flow.

2. DIRECT-TO-R2 UPLOAD. Files don't pass through the backend's 
   request body. Backend issues a signed PUT URL, browser uploads 
   directly to R2, then notifies backend via the registration 
   endpoint. Reasons:
   - Replit body size limits would block large OMs (10-28MB)
   - Latency / memory pressure on backend reduced
   - Standard pattern for R2/S3 uploads

3. SOURCE ADAPTER PATTERN SHIPPED IN TASK B. Reuse the structure. 
   New adapter in src/services/intake-sources/data-library-upload/. 
   Same upsertJob() conventions, same source_record_id usage (use 
   sha256 of file contents as source_record_id for idempotency — 
   re-uploading the same file produces the same hash and UPSERTs 
   cleanly).

4. PAIRED-READ VERIFICATION IS THE GATE. End-to-end: drag file 
   onto page → see it appear in Intake Inbox → orchestrator picks 
   up → see data in property_descriptions if address resolves → 
   visible at /archive/properties/:parcelId.

STEP 2 — FILE REGISTRATION ENDPOINT
═══════════════════════════════════════════════════════════════════

POST /api/v1/archive/files/register

Body:
  {
    parcel_id: string | null,        // from folder name, or null
    sha256: string,                   // computed client-side before upload
    original_filename: string,
    mime_type: string,
    size_bytes: number,
    storage_key: string,              // where it landed in R2
    document_type: string | null      // from classifier, or null if unsure
  }

Response:
  {
    file_id: uuid,
    status: 'registered' | 'duplicate',
    intake_job_id: uuid,
    linked_observations: number
  }

Backend:
1. INSERT into data_library_files with ON CONFLICT (sha256) DO 
   NOTHING semantics (idempotent — re-uploading same file returns 
   'duplicate' status with existing file_id)
2. Create intake_jobs row with status='pending', source_type=
   'data_library_upload', source_record_id=sha256, file_id=<new>
3. ON CONFLICT (source_type, source_record_id): UPDATE raw_input, 
   refresh updated_at, do NOT touch status (same idempotency 
   pattern as Apartment Locator adapter)
4. Backfill: if data_library_files row was newly inserted, link 
   to existing historical_observations rows where 
   (parcel_id, observation_date, document_type) match
5. Return file_id, intake_job_id, linked_observations count

═══════════════════════════════════════════════════════════════════
STEP 3 — SIGNED URL ENDPOINT
═══════════════════════════════════════════════════════════════════

POST /api/v1/archive/files/signed-upload-url

Body:
  {
    proposed_storage_key: string,    // 'parcel/.../sha256.ext'
    mime_type: string,
    size_bytes: number
  }

Response:
  {
    signed_url: string,             // presigned PUT URL, 15-min expiry
    storage_key: string,            // confirmed final storage_key
    upload_method: 'PUT',
    expires_at: timestamp
  }

Backend:
1. Validate size (cap at 100MB for now; flag larger for review)
2. Generate signed PUT URL via R2 SDK
3. Return signed URL to client
4. Client uploads file directly to R2 via PUT
5. Client then calls /files/register to record metadata + create 
   intake_job

STEP 4 — DATA LIBRARY SOURCE ADAPTER
═══════════════════════════════════════════════════════════════════

Location: src/services/intake-sources/data-library-upload/

Function: registerUploadedFile(metadata) — called by the 
registration endpoint internally. Encapsulates:
1. Idempotency check (sha256 dedup)
2. data_library_files INSERT  
3. intake_jobs UPSERT
4. backfill of historical_observations.source_file_ids

Pattern mirrors Task B's apartment-locator adapter.

│ ...                                                             │
└────────────────────────────────────────────────────────────────┘

Implementation:
1. Use File System Access API for folder selection (showOpenFilePicker 
   or webkitdirectory fallback)
2. For each file:
   a. Compute sha256 client-side (use Web Crypto API)
   b. Request signed URL via /files/signed-upload-url
   c. PUT file directly to R2 via the signed URL
   d. On success, POST to /files/register
   e. Show progress per file (uploading → uploaded → processing → 
      complete/blocked)
3. Recent uploads list polls intake_jobs status every 5 seconds for 
   files visible in the current session
4. Click on a recent upload → navigate to /archive/properties/:parcelId 
   (when known) or /archive/intake (when blocked needs identifier)

Stack: match existing frontend conventions (whatever React patterns 
ArchiveLibraryPage and ArchivePropertyPage use). No new framework 
dependencies.

      for each file

   1. Upload one 20MB OM PDF
   2. Confirm direct-to-R2 path works (backend doesn't choke on 
      the body)
   3. Confirm OM appears in property's source documents

If any of A/B/C fail, STOP. Don't mark complete.
