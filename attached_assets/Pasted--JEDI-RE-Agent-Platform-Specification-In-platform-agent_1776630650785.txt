# JEDI RE — Agent Platform Specification

> In-platform agent runtime. Agents are service-account users that consume the
> same API humans use, write to the same database humans write to, and surface
> in the same UI humans see. No parallel storage. No private backdoors.
>
> Companion to CLAUDE.md. Drop at repo root.

---

## Core Principles

1. **Agents are employees, not services.** Each agent is a service-account user with its own credentials, RBAC permissions, and audit trail. No god-mode database access.
2. **Dogfooding.** Agents write through the same API humans use. One exception: DealContext assembly writes can go direct-to-service for cache efficiency (DealContext is not user-editable in the same way module outputs are).
3. **One source of truth.** Agent outputs land in existing domain tables, wrapped in `LayeredValue<T>` with agent-specific `source` tags. No parallel agent data tables.
4. **Tools scale, agents don't.** Prefer adding a tool to an existing agent over spinning up a new agent. Specialization is earned via usage data, not declared at design time.
5. **Never sleeps = durable workflows + event-driven + cron.** Not a thread pool. Not a polling loop. Inngest (or equivalent durable workflow engine) is the runtime.
6. **Budget caps are non-optional.** Every agent run has a per-run, per-deal, and per-user cost ceiling enforced at the metering layer.

## Agent Roster (Launch)

Ship with four agents + the AI Coordinator. Do not build more until usage data justifies specialization.

| Agent ID | Role | Primary Output | Tools (initial) |
|----------|------|----------------|-----------------|
| `research` | Assembles DealContext JSON from all platform APIs | `deal_context` record | `fetch_parcel`, `fetch_costar_metrics`, `fetch_tax_bill`, `fetch_comps`, `fetch_ownership`, `write_dealcontext` |
| `zoning` | Analyzes zoning constraints, entitlements, buildable envelope | `zoning_analysis` record per deal | `fetch_zoning_code`, `fetch_municode`, `fetch_parcel`, `compute_envelope`, `write_zoning_analysis` |
| `supply` | Quantifies pipeline supply + delivery timing per submarket | `supply_analysis` record per deal | `fetch_permits`, `fetch_costar_pipeline`, `fetch_submarket_deliveries`, `write_supply_analysis` |
| `cashflow` | Runs F9 ProForma math, writes projection + returns | `cashflow_projection` record per deal | `fetch_t12`, `fetch_rent_roll`, `fetch_assumptions`, `compute_proforma`, `write_projection` |
| `coordinator` | NLU, intent extraction, orchestration, synthesis | Chat responses + agent dispatches | Dispatch tools for all four above |

**Note on Coordinator:** The Coordinator runs in the user's session and does **not** have a service account — it executes on behalf of the current user. The four workers above each have their own service account.

### What is NOT an agent

Do not build these as agents. Build them as cron jobs, Kafka consumers, or direct service calls:

- Notification dispatch
- Email/SMS sending
- Report generation (if deterministic)
- Nightly data refreshes (FRED, RentCast, etc.)
- Audit log writes
- Credit balance updates
- Stripe webhook handling

Rule of thumb: if the logic can be written as deterministic `if X then Y`, it is not an agent. Claude-in-a-loop is for reasoning under uncertainty with tool use.

### Auth middleware

Agents authenticate via scoped JWTs issued from a new `/auth/agent-token` internal endpoint (not publicly exposed). Token includes `user_id`, `user_type: 'agent'`, `agent_id`, `capabilities[]`, short TTL (1 hour, auto-refresh).

The existing `authMiddleware` checks `user_type` on the token. Add one line: if `user_type === 'agent'`, enforce the agent's declared `capabilities[]` against the endpoint's required capability. Reject if not permitted.

```typescript
// src/middleware/auth.ts — extension
export function requireCapability(required: string) {
  return (req: Request, res: Response, next: NextFunction) => {
    const { user_type, capabilities } = req.user;
    if (user_type === 'agent' && !capabilities.includes(required)) {
      return res.status(403).json({ error: `Agent lacks capability: ${required}` });
    }
    next();
  };
}

// Usage:
router.post('/deals/:id/zoning-analysis',
  authMiddleware,
  requireCapability('write:zoning_analysis'),
  zoningAnalysisController.create
);
```

## Provenance: `LayeredValue<T>` Extension

Extend the existing `source` enum to include agent-specific tags:

```typescript
// src/types/layered-value.ts
export type LayeredValueSource =
  | 'platform'         // default/fallback values
  | 't12'              // from uploaded T12
  | 'rent_roll'        // from uploaded rent roll
  | 'tax_bill'         // from uploaded tax bill
  | 'override'         // user edit
  | 'agent:research'   // NEW
  | 'agent:zoning'     // NEW
  | 'agent:supply'     // NEW
  | 'agent:cashflow';  // NEW

export interface LayeredValue<T> {
  value: T;
  source: LayeredValueSource;
  agent_run_id?: string;    // NEW — links to agent_runs table when source starts with 'agent:'
  set_at: string;           // ISO timestamp
  set_by?: string;          // user_id (human or agent)
  confidence?: number;      // 0-1, optional
}
```

**Merge order** (lowest to highest priority):
1. `platform` (defaults)
2. `agent:*` (any agent-written value)
3. `t12` / `rent_roll` / `tax_bill` (document-derived)
4. `override` (user edit — always wins)

User overrides still sit on top of agent-written values. A user can accept an agent's suggestion (no override needed), or override it (agent value remains in history but is not the active value).

## Audit Log Unification

Single `audit_log` table for both human and agent actions.

```sql
CREATE TABLE audit_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  actor_id TEXT NOT NULL,
  actor_type TEXT NOT NULL CHECK (actor_type IN ('human', 'agent', 'system')),
  action TEXT NOT NULL,              -- e.g. 'deal.create', 'zoning.analyze', 'financial.edit'
  resource_type TEXT NOT NULL,       -- e.g. 'deal', 'parcel', 'zoning_analysis'
  resource_id TEXT NOT NULL,
  metadata JSONB,                    -- action-specific payload
  agent_run_id UUID,                 -- NULL unless actor_type='agent'
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_audit_resource ON audit_log(resource_type, resource_id, created_at DESC);
CREATE INDEX idx_audit_actor ON audit_log(actor_type, actor_id, created_at DESC);
CREATE INDEX idx_audit_agent_run ON audit_log(agent_run_id) WHERE agent_run_id IS NOT NULL;
```

Every API write goes through an audit middleware that inserts into this table. Agent actions land here with `actor_type='agent'` and a non-null `agent_run_id` pointing to the run record.

The deal activity feed reads from `audit_log` filtered by `resource_type='deal' AND resource_id=:deal_id`. Agent and human actions appear in the same stream, visually distinguished only by an actor icon.

## Agent Runtime Tables

Three new tables for agent-specific accounting. These are the **only** agent-specific tables — everything else writes to existing domain tables.

### `agent_runs`

```sql
CREATE TABLE agent_runs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agent_id TEXT NOT NULL,                -- 'research', 'zoning', 'supply', 'cashflow'
  agent_version TEXT NOT NULL,           -- semver
  prompt_version TEXT NOT NULL,          -- e.g. 'research-v0.3.1'
  deal_id UUID,                          -- nullable (some runs are not deal-scoped)
  triggered_by TEXT NOT NULL CHECK (triggered_by IN ('user', 'event', 'cron')),
  trigger_context JSONB,                 -- e.g. { event: 'deal_created', user_id: '...' }
  user_id TEXT,                          -- user on whose behalf the run executes (null for cron)
  status TEXT NOT NULL CHECK (status IN ('pending','running','succeeded','failed','aborted','budget_exceeded')),
  input JSONB NOT NULL,
  output JSONB,
  error TEXT,
  tokens_in BIGINT DEFAULT 0,
  tokens_out BIGINT DEFAULT 0,
  cost_usd NUMERIC(10,4) DEFAULT 0,
  started_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  completed_at TIMESTAMPTZ,
  duration_ms INTEGER
);

CREATE INDEX idx_agent_runs_deal ON agent_runs(deal_id, started_at DESC);
CREATE INDEX idx_agent_runs_status ON agent_runs(status, started_at DESC);
CREATE INDEX idx_agent_runs_agent ON agent_runs(agent_id, started_at DESC);
```

### `agent_run_steps`

One row per tool call within a run. For observability and debugging.

```sql
CREATE TABLE agent_run_steps (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agent_run_id UUID NOT NULL REFERENCES agent_runs(id) ON DELETE CASCADE,
  step_index INTEGER NOT NULL,           -- 0-indexed
  step_type TEXT NOT NULL CHECK (step_type IN ('prompt','tool_call','tool_result','output')),
  tool_name TEXT,                        -- populated for tool_call/tool_result
  payload JSONB NOT NULL,
  tokens_in INTEGER,
  tokens_out INTEGER,
  duration_ms INTEGER,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_agent_steps_run ON agent_run_steps(agent_run_id, step_index);
```

### `prompt_versions`

```sql
CREATE TABLE prompt_versions (
  id TEXT PRIMARY KEY,                   -- e.g. 'research-v0.3.1'
  agent_id TEXT NOT NULL,
  version TEXT NOT NULL,
  system_prompt TEXT NOT NULL,
  output_schema JSONB NOT NULL,          -- Zod schema as JSON
  tools JSONB NOT NULL,                  -- tool manifest
  active BOOLEAN NOT NULL DEFAULT false,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_by TEXT
);

CREATE UNIQUE INDEX idx_prompt_versions_active
  ON prompt_versions(agent_id) WHERE active = true;
```

Only one prompt version per agent is active at a time. Rollbacks are a flag flip.

### Interface

```typescript
// src/agents/runtime/types.ts
export interface AgentConfig {
  agentId: string;
  agentVersion: string;
  promptVersion: string;           // e.g. 'research-v0.3.1'
  tools: ToolDefinition[];
  outputSchema: z.ZodSchema;
  budgetCaps: BudgetCaps;
  modelName: 'claude-opus-4-7' | 'claude-sonnet-4-6' | 'claude-haiku-4-5-20251001';
}

export interface BudgetCaps {
  maxTokensPerRun: number;         // e.g. 500_000
  maxCostUsdPerRun: number;        // e.g. 5.00
  maxStepsPerRun: number;          // e.g. 30
  maxCostUsdPerDealPerDay: number; // e.g. 20.00 across all agents
  maxCostUsdPerUserPerMonth: number; // enforced via user's credit balance
}

export interface RunContext {
  dealId?: string;
  userId?: string;
  triggeredBy: 'user' | 'event' | 'cron';
  triggerContext?: Record<string, unknown>;
  correlationId?: string;          // for tracing across services
}

export interface ToolDefinition<TInput = unknown, TOutput = unknown> {
  name: string;
  description: string;
  inputSchema: z.ZodSchema<TInput>;
  outputSchema: z.ZodSchema<TOutput>;
  requiresCapability: string;      // e.g. 'read:parcels'
  execute: (input: TInput, ctx: RunContext) => Promise<TOutput>;
}
```

### Core class (skeleton)

```typescript
// src/agents/runtime/AgentRuntime.ts
export class AgentRuntime {
  constructor(
    private config: AgentConfig,
    private platformClient: PlatformClient,
    private metering: MeteringAdapter,
    private budget: BudgetEnforcer,
    private db: Database
  ) {}

  async run(
    input: unknown,
    ctx: RunContext
  ): Promise<z.infer<typeof this.config.outputSchema>> {
    // 1. Pre-flight budget check
    await this.budget.check(ctx, this.config.budgetCaps);

    // 2. Create agent_runs row
    const run = await this.db.agentRuns.create({
      agent_id: this.config.agentId,
      agent_version: this.config.agentVersion,
      prompt_version: this.config.promptVersion,
      deal_id: ctx.dealId,
      user_id: ctx.userId,
      triggered_by: ctx.triggeredBy,
      trigger_context: ctx.triggerContext,
      status: 'running',
      input
    });

    try {
      // 3. Load prompt
      const { system_prompt } = await this.db.promptVersions.getActive(this.config.agentId);

      // 4. Tool-calling loop
      const result = await this.loop({
        run,
        systemPrompt: system_prompt,
        userMessage: JSON.stringify(input),
        ctx
      });

      // 5. Validate output against schema
      const validated = this.config.outputSchema.parse(result.content);

      // 6. Mark complete, return
      await this.db.agentRuns.complete(run.id, {
        status: 'succeeded',
        output: validated,
        tokens_in: result.totalTokensIn,
        tokens_out: result.totalTokensOut,
        cost_usd: result.totalCost
      });

      return validated;
    } catch (err) {
      await this.db.agentRuns.complete(run.id, {
        status: err instanceof BudgetExceededError ? 'budget_exceeded' : 'failed',
        error: err.message
      });
      throw err;
    }
  }

  private async loop({ run, systemPrompt, userMessage, ctx }): Promise<LoopResult> {
    const messages: Message[] = [{ role: 'user', content: userMessage }];
    let totalTokensIn = 0, totalTokensOut = 0, totalCost = 0, stepIndex = 0;

    for (let i = 0; i < this.config.budgetCaps.maxStepsPerRun; i++) {
      // Budget check per step
      await this.budget.checkRunCap(run.id, totalCost, this.config.budgetCaps);

      // Call Claude via metering adapter
      const response = await this.metering.createMessage({
        model: this.config.modelName,
        system: systemPrompt,
        messages,
        tools: this.config.tools.map(toAnthropicToolSchema),
        max_tokens: 4096,
        metadata: {  // attribution metadata
          actor_type: 'agent',
          actor_id: this.config.agentId,
          agent_run_id: run.id,
          deal_id: ctx.dealId,
          triggered_by: ctx.triggeredBy
        }
      });

      totalTokensIn += response.usage.input_tokens;
      totalTokensOut += response.usage.output_tokens;
      totalCost += response.usage.cost_usd;

      // Log prompt step
      await this.db.agentRunSteps.create({
        agent_run_id: run.id,
        step_index: stepIndex++,
        step_type: 'prompt',
        payload: response,
        tokens_in: response.usage.input_tokens,
        tokens_out: response.usage.output_tokens
      });

      // Handle tool calls
      const toolUses = response.content.filter(b => b.type === 'tool_use');
      if (toolUses.length === 0) {
        // Final output
        const textContent = response.content.find(b => b.type === 'text')?.text;
        return {
          content: JSON.parse(textContent),
          totalTokensIn, totalTokensOut, totalCost
        };
      }

      // Execute tools
      const toolResults = await Promise.all(
        toolUses.map(tu => this.executeTool(tu, run, ctx, stepIndex++))
      );

      messages.push({ role: 'assistant', content: response.content });
      messages.push({ role: 'user', content: toolResults });
    }

    throw new Error(`Agent exceeded max steps (${this.config.budgetCaps.maxStepsPerRun})`);
  }

  private async executeTool(toolUse, run, ctx, stepIndex): Promise<ToolResultBlock> {
    const tool = this.config.tools.find(t => t.name === toolUse.name);
    if (!tool) throw new Error(`Unknown tool: ${toolUse.name}`);

    // Validate input
    const input = tool.inputSchema.parse(toolUse.input);

    // Log tool call
    await this.db.agentRunSteps.create({
      agent_run_id: run.id,
      step_index: stepIndex,
      step_type: 'tool_call',
      tool_name: tool.name,
      payload: input
    });

    // Execute via platform client (dogfooding — goes through API)
    const start = Date.now();
    const output = await tool.execute(input, ctx);
    const duration = Date.now() - start;

    // Validate output
    const validated = tool.outputSchema.parse(output);

    // Log tool result
    await this.db.agentRunSteps.create({
      agent_run_id: run.id,
      step_index: stepIndex,
      step_type: 'tool_result',
      tool_name: tool.name,
      payload: validated,
      duration_ms: duration
    });

    return {
      type: 'tool_result',
      tool_use_id: toolUse.id,
      content: JSON.stringify(validated)
    };
  }
}
```

## Metering Attribution — Extending the Existing Wrapper

Your existing `@stripe/token-meter` wrapper around `@anthropic-ai/sdk` is the metering primitive. Extend it to accept attribution metadata and route costs to the right bucket.

```typescript
// src/lib/metering/MeteringAdapter.ts
export interface MeteringMetadata {
  actor_type: 'human' | 'agent';
  actor_id: string;
  agent_run_id?: string;
  deal_id?: string;
  user_id?: string;                // who pays (null for cron jobs charged to platform)
  triggered_by?: 'user' | 'event' | 'cron';
}

export class MeteringAdapter {
  constructor(
    private tokenMeter: TokenMeter,              // existing @stripe/token-meter instance
    private anthropic: Anthropic,                // existing @anthropic-ai/sdk instance
    private creditService: CreditService         // existing credit balance service
  ) {}

  async createMessage(params: MessageCreateParams & { metadata: MeteringMetadata }) {
    const { metadata, ...apiParams } = params;

    // Pre-flight: does the payer have credits?
    if (metadata.triggered_by === 'user' && metadata.user_id) {
      await this.creditService.reserveCredits(metadata.user_id, estimateCost(apiParams));
    }

    // Execute via metered SDK
    const response = await this.tokenMeter.wrap(this.anthropic).messages.create(apiParams);

    // Post-flight: record to agent_runs + charge appropriate bucket
    const cost = calculateCost(response.usage, apiParams.model);

    await this.recordUsage({
      ...metadata,
      tokens_in: response.usage.input_tokens,
      tokens_out: response.usage.output_tokens,
      cost_usd: cost,
      model: apiParams.model
    });

    // Charge per bucket rules
    await this.chargeByBucket(metadata, cost);

    return { ...response, usage: { ...response.usage, cost_usd: cost } };
  }

  private async chargeByBucket(metadata: MeteringMetadata, cost: number) {
    if (metadata.triggered_by === 'user' && metadata.user_id) {
      // User-initiated → user pays
      await this.creditService.debit(metadata.user_id, cost, {
        reason: 'agent_run',
        agent_id: metadata.actor_id,
        agent_run_id: metadata.agent_run_id
      });
    } else if (metadata.triggered_by === 'cron' || metadata.triggered_by === 'event') {
      // Scheduled/event-driven → platform absorbs cost (tier benefit, not metered to user)
      await this.creditService.platformExpense(cost, {
        reason: metadata.triggered_by,
        agent_id: metadata.actor_id,
        deal_id: metadata.deal_id
      });
    }
  }
}
```

**Three-bucket charging rule, implemented:**

| Trigger | Who Pays | Enforcement |
|---------|----------|-------------|
| `triggered_by: 'user'` | User's credit balance | Fails fast if insufficient |
| `triggered_by: 'event'` | Platform (tier benefit) | Tier-gated: Scout gets none, Principal gets all |
| `triggered_by: 'cron'` | Platform (tier benefit) | Tier-gated: Operator gets weekly, Principal gets daily |

Tier gating is enforced at the trigger level — the event subscription or cron schedule itself checks the user's tier before spawning the run.

## `BudgetEnforcer` — Circuit Breakers

```typescript
// src/agents/runtime/BudgetEnforcer.ts
export class BudgetEnforcer {
  constructor(private db: Database) {}

  async check(ctx: RunContext, caps: BudgetCaps): Promise<void> {
    // Per-deal daily cap
    if (ctx.dealId) {
      const spent = await this.db.agentRuns.sumCostUsdByDealSince(
        ctx.dealId,
        startOfDay()
      );
      if (spent >= caps.maxCostUsdPerDealPerDay) {
        throw new BudgetExceededError(
          `Deal ${ctx.dealId} hit daily agent cap ($${caps.maxCostUsdPerDealPerDay})`
        );
      }
    }
    // Per-user monthly cap handled by credit service (existing)
  }

  async checkRunCap(runId: string, currentCost: number, caps: BudgetCaps): Promise<void> {
    if (currentCost >= caps.maxCostUsdPerRun) {
      throw new BudgetExceededError(
        `Run ${runId} exceeded per-run cap ($${caps.maxCostUsdPerRun})`
      );
    }
  }
}
```

## Tool Pattern — Dogfooding the Platform API

Every tool an agent uses calls the platform API via `platformClient.ts`. This is the same client the frontend imports. Agents get no private backdoor.

```typescript
// src/agents/tools/fetch_parcel.ts
import { z } from 'zod';
import { ToolDefinition } from '../runtime/types';
import { platformClient } from '@/lib/platform-client';

const InputSchema = z.object({
  parcel_id: z.string().optional(),
  address: z.string().optional(),
  county: z.string().optional()
}).refine(
  d => d.parcel_id || (d.address && d.county),
  { message: 'Must provide parcel_id or (address + county)' }
);

const OutputSchema = z.object({
  parcel_id: z.string(),
  address: z.string(),
  acres: z.number(),
  zoning_code: z.string(),
  assessed_value: z.number(),
  // ... etc
});

export const fetchParcelTool: ToolDefinition<
  z.infer<typeof InputSchema>,
  z.infer<typeof OutputSchema>
> = {
  name: 'fetch_parcel',
  description: 'Fetch parcel data by parcel_id or by address+county. Returns zoning, acres, valuation.',
  inputSchema: InputSchema,
  outputSchema: OutputSchema,
  requiresCapability: 'read:parcels',
  execute: async (input, ctx) => {
    // Goes through platform API with agent's service-account token
    const response = await platformClient
      .as({ agentId: 'research', runId: ctx.correlationId })
      .parcels
      .lookup(input);
    return response;
  }
};
```

### The `write_*` tools and the one exception

Most writes go through the API:

```typescript
// src/agents/tools/write_zoning_analysis.ts
execute: async (input, ctx) => {
  return platformClient
    .as({ agentId: 'zoning', runId: ctx.correlationId })
    .deals(ctx.dealId!)
    .zoningAnalysis
    .upsert(input);
}
```

**Exception:** `write_dealcontext` writes directly to the DealContext service, not through the API. Reason: DealContext is the assembly target that user-editable module data sits on top of. It is not itself user-editable through the UI. The performance win of direct-service writes matters during assembly (many writes in rapid succession), and the dogfooding argument is weaker because there is no equivalent user-facing write path.

```typescript
// src/agents/tools/write_dealcontext.ts
execute: async (input, ctx) => {
  // Direct service call — the one documented exception
  return dealContextService.upsert(ctx.dealId!, input, {
    source: 'agent:research',
    agent_run_id: ctx.correlationId
  });
}
```

Document this exception in `CLAUDE.md`. Do not expand it to any other tool without explicit architectural review.

## Durable Runtime — Inngest

Install Inngest for durable workflow execution. Each agent run is an Inngest function.

```bash
npm install inngest
```

### Event-driven example

```typescript
// src/agents/inngest/research.function.ts
import { inngest } from '@/lib/inngest';
import { researchAgent } from '@/agents/research.agent';

export const researchOnDealCreated = inngest.createFunction(
  { id: 'research-on-deal-created', retries: 3 },
  { event: 'deal.created' },
  async ({ event, step }) => {
    const dealId = event.data.deal_id;
    const userId = event.data.user_id;

    // Tier gate: only run auto-research for tiers that include it
    const tier = await step.run('check-tier', () =>
      tierService.getTier(userId)
    );
    if (!['principal', 'institutional'].includes(tier)) {
      return { skipped: true, reason: 'tier_not_included' };
    }

    // Run the agent (durable — survives restarts)
    const result = await step.run('research-agent-run', () =>
      researchAgent.run(
        { deal_id: dealId },
        { dealId, userId, triggeredBy: 'event',
          triggerContext: { event_name: 'deal.created' } }
      )
    );

    // Publish completion event for downstream agents
    await step.sendEvent('research-done', {
      name: 'research.completed',
      data: { deal_id: dealId, dealcontext_id: result.dealcontext_id }
    });

    return result;
  }
);
```

### Cron example

```typescript
// src/agents/inngest/research.cron.ts
export const researchWeeklyRefresh = inngest.createFunction(
  { id: 'research-weekly-refresh' },
  { cron: '0 2 * * 1' },  // Mondays 2 AM UTC
  async ({ step }) => {
    const deals = await step.run('fetch-active-deals', () =>
      dealService.listActive({ tier_min: 'operator' })
    );

    for (const deal of deals) {
      await step.run(`refresh-deal-${deal.id}`, () =>
        researchAgent.run(
          { deal_id: deal.id },
          { dealId: deal.id, userId: deal.owner_id, triggeredBy: 'cron' }
        )
      );
    }
  }
);
```

Inngest handles retries, backoff, concurrency limits, and observability. Each `step.run()` call is durable — if the process crashes, Inngest replays from the last completed step.

## Observability

Add OpenTelemetry instrumentation. Each agent run is a trace. Each tool call is a span.

```typescript
// src/agents/runtime/AgentRuntime.ts — instrumented version (sketch)
import { trace } from '@opentelemetry/api';

async run(input, ctx) {
  const tracer = trace.getTracer('jedi-agents');
  return tracer.startActiveSpan(
    `agent.${this.config.agentId}.run`,
    { attributes: {
        'agent.id': this.config.agentId,
        'agent.version': this.config.agentVersion,
        'deal.id': ctx.dealId,
        'triggered_by': ctx.triggeredBy
    }},
    async (span) => {
      try {
        // ... existing run logic
        span.setAttribute('agent.tokens_total', totalTokens);
        span.setAttribute('agent.cost_usd', totalCost);
        return validated;
      } catch (err) {
        span.recordException(err);
        span.setStatus({ code: SpanStatusCode.ERROR });
        throw err;
      } finally {
        span.end();
      }
    }
  );
}
```

Ship traces to whatever backend you prefer (Honeycomb, Grafana Tempo, Datadog). The `agent_run_id` becomes a queryable correlation key.

## What NOT to Do

- Do not create an `agent_outputs` table or any parallel storage. Agent outputs land in existing domain tables via LayeredValue.
- Do not give agents direct DB access. All writes go through the API (except the documented DealContext exception).
- Do not add new agents beyond the four until usage data shows specialization is warranted.
- Do not put prompts as inline strings in agent code. Version them in `prompt_versions` and load at runtime.
- Do not skip the output schema validation. Unvalidated agent output poisons the DealContext cache.
- Do not build agent-to-agent direct communication. Agents hand off via events (Kafka / Inngest), never via function calls into each other.
- Do not let budget caps be optional. Every agent has them. Every run enforces them.

## Open Decisions

1. **Which durable workflow engine:** Inngest (recommended — best DX for this use case), Temporal (more industrial, steeper curve), or BullMQ (DIY, more work). Default to Inngest unless ops preference differs.
2. **Observability backend:** Honeycomb, Datadog, Grafana Tempo, or OSS self-host. Affects only the OTel exporter config.
3. **Agent token TTL:** 1 hour with auto-refresh is the proposal. Shorter if security-conscious; longer if refresh overhead becomes a concern.
4. **Per-agent model selection:** Default is Opus 4.7 for Research (reasoning-heavy), Sonnet for Zoning/Supply/Cashflow (structured extraction). Tune from telemetry.
