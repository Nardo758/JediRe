## Context

You are implementing the Tax Service — a deterministic, jurisdiction-aware tax math engine that powers the F9 Taxes tab (Sections A, B, C, D) across all four sections of the JEDI RE platform. The F9 UI already exists; you are populating it, not redesigning it.

This is **Task 6.5** in `AGENT_TASK_INVENTORY.md`, classified as **C** (deterministic code). It is not an agent. It is a service that agents call as a tool.

The launch jurisdictions are: **FL** (Miami-Dade, Broward, Palm Beach county overlays), **GA** (Fulton county overlay), **TX** (Harris county overlay), **NC** (Mecklenburg, Wake county overlays), and a **federal** ruleset that always applies. Plus a **default** state ruleset for jurisdictions not yet modeled.

The Tax Service computes year-by-year tax lines across the deal hold period, emitting **reassessment events** at the years they fire based on jurisdiction-declared triggers (sale, cyclical revaluation, new construction CO, substantial improvement, etc.). These events flow into the F9 ProForma timeline, render as red bars in the year-by-year tax row, and feed M14 Risk's reassessment-risk scoring.

## Ground Rules — Non-Negotiable

These are derived from the existing `REPLIT_AGENT_IMPLEMENTATION_PROMPT.md` ground rules and apply unchanged here.

1. **No `if (state === 'X')` outside `src/services/tax/rulesets/X/`.** This is verified by a CI grep check. If you find yourself wanting to write state-conditional logic anywhere else, stop and architect it as a ruleset method.

2. **Three-layer separation is sacred.** Rule mechanics → TypeScript code. Annual rates → JSON config. Per-deal data → runtime fetch. Never embed a 2026 millage rate in `fl.ruleset.ts`. Never put cap-formula math in a JSON file. **Never hardcode NC's 4-year cycle in `nc.ruleset.ts` — it lives in the county rate sheet.** Mixing layers silently corrupts future-year deals.

3. **Every output wraps in `LayeredValue<T>`.** No raw numbers escape the service. Provenance is always preserved.

4. **Composition is mandatory.** Every property resolves three stacked rulesets (federal + state + county). The composer walks the stack. Do not bypass it.

5. **Rate sheets are validated at boot.** Use a JSON schema validator (`zod`) on every rate sheet during service initialization. Invalid sheets fail loud, not silent.

6. **Deterministic outputs.** Same inputs ⇒ byte-identical outputs. Test this explicitly, including timeline event sequences.

7. **No LLM in the hot path.** Agents have a narrow role (spec §11) and operate out-of-band. The runtime computation is pure code.

8. **Cache aggressively, invalidate by fiscal year.** Jurisdiction cache TTL = end of fiscal year. Parcel cache TTL = end of fiscal year OR on tax bill upload, whichever comes first.

9. **All writes are audited.** Per platform convention; rate sheet promotions, cache invalidations, and override events all hit `audit_log`.

10. **No UI changes.** F9 tab structure already exists per the latest design. You populate cells; you do not restructure the tab.

11. **Audit before building.** Before Phase 1, run a codebase audit per the "audit before building" principle in CLAUDE.md. Document findings in `SPEC_QUESTIONS.md`. Do not duplicate existing types or services.

12. **JSX rendering rules apply** for any UI integration code. Space after `return` before JSX; no `React.Fragment` with keys inside CSS grids; no IIFE patterns inside JSX; named function components with explicit returns.

13. **Triggers are declared, evaluated, and emitted — never assumed.** No code anywhere should hardcode "reassessment fires in year 1" or "sale always triggers reset" outside the trigger evaluator. The pattern is: ruleset declares triggers via `reassessmentTriggers()`, the evaluator walks the timeline, events get emitted with full provenance. If you find yourself writing `if (yearOfHold === 1) reassess()`, stop and route through the trigger evaluator instead.

## What NOT to Do

- ❌ Do NOT redesign the F9 Taxes tab UI — it exists, you populate it
- ❌ Do NOT make tax math agent-driven; this is a deterministic service
- ❌ Do NOT embed rates (millage, doc stamp, bonus dep %) inside ruleset TypeScript files — rates go in JSON
- ❌ Do NOT embed rule logic (10% cap formula, bonus dep schedule structure) in JSON — rules go in TypeScript
- ❌ Do NOT add `if (state === 'FL')` anywhere outside `rulesets/fl/`
- ❌ **Do NOT hardcode "reassessment fires in year 1" or "sale always triggers reset" anywhere outside the trigger evaluator. Triggers are declared and evaluated, never assumed.**
- ❌ **Do NOT bypass trigger evaluation for "simple" cases (e.g., "I know FL always reassesses on sale, let me just hardcode it").** The simple case must go through the same code path as the complex case.
- ❌ **Do NOT hardcode NC's revaluation cycle in `nc.ruleset.ts`.** Cycle length and anchor year live in county rate sheets via `trigger_parameters.cyclical`.
- ❌ Do NOT skip the Phase 0 audit — duplicating existing types/services is the highest-cost failure mode
- ❌ Do NOT bypass `LayeredValue` wrapping — every output field carries provenance
- ❌ Do NOT introduce new dependencies (Redis, alternate HTTP clients, etc.) without flagging
- ❌ Do NOT write a Miami-Dade-specific scraper in v1 — use ATTOM. County adapters are deferred until cost/staleness justifies them
- ❌ Do NOT model TX franchise tax in v1 — it's flagged out of scope
- ❌ Do NOT model 1031 exchanges, opportunity zones, or AMT — also out of scope
- ❌ Do NOT model ownership_restructure triggers — declared in taxonomy but not evaluated in v1
- ❌ Do NOT silently work around ground rules — flag conflicts in writing
