TASK — Phase 1 Correctness Fixes: Corpus Write Path

Per the Phase 1 review findings (C1, C2, H1, H2, H3, H4). The
corpus-write path has two critical bugs that make every T12 and
rent roll ingestion functionally inert despite the surrounding
infrastructure being in place. This ticket addresses the
correctness gap; the Phase 1 build's other deliverables (schema,
reminders, lifecycle service, query service) stay as they are.

CONTEXT

Phase 1 of the Historical Observations corpus build shipped per
docs/architecture/HISTORICAL_OBSERVATIONS_SPEC.md §11 Phase 1.
Review of the merged code surfaced two critical bugs in
backend/src/services/historical-observations/document-to-corpus.ts
that mean no T12 or rent roll data is correctly landing in
historical_observations:

  C1 — writeT12ToCorpus (lines 156-176) and writeRentRollToCorpus
       (lines 183-203) pass parcelId: '' (empty string) to
       ingestPropertyPerformance. The ingestor uses doc.parcelId
       directly, so every insert lands with parcel_id = '',
       making rows unreachable by geography queries and
       realized-output backfill.

  C2 — ingestPropertyPerformance (property-performance-ingestor.ts
       lines 201-208) silently returns when an existing row is
       found at (parcel_id × observation_date). This violates
       spec §7.6: when T12 and rent roll for the same month
       both arrive, the second one is silently discarded
       instead of merging signals and promoting the tier from
       S2 to S1.

The OM and TaxBill transformer paths in the same file (lines
220 and 300) handle this correctly: they call resolveParcelId
to look up the parcel from the deal, then upsertCorpusRow to
merge with any existing row. The T12 and RentRoll paths need
to follow the same pattern.

Additional findings to address in the same dispatch:

  H1 — deal_lifecycle_events table is referenced by Spec §7.9
       Invariant 2 and §4.1 but does not exist; no migration
       creates it, no service writes to it.

  H2 — dataCorpusReminderCron.ts implements Triggers 1, 2, 3
       but Triggers 1a (CoStar staleness) and 1b (comp performance
       gap) per spec §9.2 are completely absent.

  H3 — query.service.ts insertRow() keyToCol dictionary is
       missing 14 Phase 1 columns: property_asking_rent,
       property_signing_velocity, capital_event_*, data_quality_tier,
       redistribution_restricted, costar_submarket_*, market_survey_*.
       Currently latent (T12/RentRoll transformers don't set these),
       but any future writer using insertRow() will silently drop
       these columns.

  H4 — rentRollToCorpusRow (property-performance-ingestor.ts)
       does not compute property_signing_velocity from lease_start
       dates in the rent roll. Spec §7.6 requires this; the
       RentRollUnit type already carries leaseStart fields.

SCOPE

In scope:
  1. (C1, C2) Refactor writeT12ToCorpus and writeRentRollToCorpus
     to follow the OM/TaxBill pattern: resolveParcelId →
     transformer-to-fields-dict → upsertCorpusRow. Eliminate the
     calls to ingestPropertyPerformance from these two entry
     points.
  2. (H1) Add deal_lifecycle_events table migration; update
     lifecycle-transition.service.ts to write to it on every
     observed status transition.
  3. (H2) Add Trigger 1a (CoStar staleness) and Trigger 1b
     (comp performance gap) blocks to dataCorpusReminderCron.ts.
  4. (H3) Expand the keyToCol dictionary in query.service.ts
     insertRow() to include all 14 missing Phase 1 columns.
  5. (H4) Add property_signing_velocity computation to
     rentRollToCorpusRow.

Out of scope:
  - Retiring PropertyPerformanceIngestor entirely. After C1/C2 fix,
    nothing in the codebase calls ingestPropertyPerformance with
    valid arguments. Mark the function @deprecated with a comment
    pointing to the working OM/TaxBill pattern. Removing it is a
    follow-up cleanup ticket.
  - The M-series UI items (PropertyPerformanceUploadZone,
    AddExistingPropertyForm, DataCorpusNotification, Trigger 4
    inline banner). These are Phase 2-3 deliverables per spec §14

  - The L-series Phase 4+ items (CoStar parsers, REIT 10-K parsers,
    external signal ingestion). These are scheduled for Phase 4+
    per spec §11 phased plan.

DETAILED FIXES

C1 + C2 — T12 and RentRoll transformers route through upsertCorpusRow

File: backend/src/services/historical-observations/document-to-corpus.ts

Lines 148-203 (writeT12ToCorpus + writeRentRollToCorpus) get
rewritten to follow the OM/TaxBill pattern. The transformer
functions in property-performance-ingestor.ts (t12ToCorpusRow,
rentRollToCorpusRow) are reused for field extraction but
applied to the resolved parcel ID, then passed to
upsertCorpusRow rather than ingestPropertyPerformance.

Expected shape:

  export async function writeT12ToCorpus(
    pool: Pool,
    dealId: string,
    parsedT12: T12Data,
    reportPeriod: Date,
  ): Promise<void> {
    const resolved = await resolveParcelId(pool, dealId);
    if (!resolved) {
      logger.warn('[DocumentToCorpus] writeT12ToCorpus: no parcel resolved', { dealId });
      return;
    }

    // Normalize observation date to 1st of month
    const obsDate = new Date(Date.UTC(reportPeriod.getUTCFullYear(),
                                       reportPeriod.getUTCMonth(), 1));

    // Build field dict using existing transformer helper
    const row = t12ToCorpusRow(parsedT12, resolved.parcelId, obsDate, resolved.propertyId);
    const fields = stripMetadataKeys(row);  // remove parcel_id, observation_date,
                                            // geography_level, source_signals from
                                            // the fields dict (upsertCorpusRow
                                            // sets these)

    await upsertCorpusRow(pool, resolved.parcelId, obsDate, fields, ['t12']);
  }

  export async function writeRentRollToCorpus(
    pool: Pool,
    dealId: string,
    parsedRentRoll: RentRollData,
    snapshotDate: Date,
  ): Promise<void> {
    const resolved = await resolveParcelId(pool, dealId);
    if (!resolved) {
      logger.warn('[DocumentToCorpus] writeRentRollToCorpus: no parcel resolved', { dealId });
      return;
    }

    const obsDate = new Date(Date.UTC(snapshotDate.getUTCFullYear(),
                                       snapshotDate.getUTCMonth(), 1));

    const row = rentRollToCorpusRow(parsedRentRoll, resolved.parcelId,
                                    obsDate, resolved.propertyId);
    const fields = stripMetadataKeys(row);

    await upsertCorpusRow(pool, resolved.parcelId, obsDate, fields, ['rent_roll']);
  }

stripMetadataKeys helper:

  function stripMetadataKeys(row: PartialHistoricalObservationRow):
    Record<string, unknown> {
    const { parcel_id, observation_date, geography_level,
            observation_window, source_signals, data_quality_tier,
            is_subject_property, ...fields } = row as any;
    return fields;
  }

Note: pool argument is now USED (was _pool with underscore prefix
indicating "intentionally unused"). The transaction context flows
through to upsertCorpusRow's pool.query calls. If the calling
context wraps these in a transaction via a client (as opposed to
a pool), upsertCorpusRow may need to accept PoolClient instead;
verify against the actual caller in data-router.ts and adjust the
signature if needed.

After this refactor:
  - Two documents at the same parcel × month MERGE source_signals
    and PROMOTE tier from S2 to S1 (via computeTier()).
  - Empty parcel_id rows can no longer be inserted.
  - The existing OM/TaxBill paths continue to work unchanged.

H1 — deal_lifecycle_events table + writes

Migration:
backend/src/db/migrations/YYYYMMDD_deal_lifecycle_events.sql

  CREATE TABLE deal_lifecycle_events (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    deal_id         UUID NOT NULL REFERENCES deals(id) ON DELETE CASCADE,
    from_status     VARCHAR(40),
    to_status       VARCHAR(40) NOT NULL,
    transitioned_at TIMESTAMP NOT NULL DEFAULT NOW(),
    transitioned_by UUID,
    note            TEXT,
    metadata        JSONB,
    created_at      TIMESTAMP NOT NULL DEFAULT NOW()
  );

  CREATE INDEX idx_deal_lifecycle_events_deal_id
    ON deal_lifecycle_events(deal_id, transitioned_at DESC);

  CREATE INDEX idx_deal_lifecycle_events_to_status
    ON deal_lifecycle_events(to_status, transitioned_at DESC);

  COMMENT ON TABLE deal_lifecycle_events IS
    'Append-only log of deal status transitions. Used by the
     historical_observations corpus to distinguish pre-decision
     (broker-supplied) vs post-decision (operator-supplied) months
     per HISTORICAL_OBSERVATIONS_SPEC.md §7.9 Invariant 2.';

Write hook in backend/src/services/portfolio/lifecycle-transition.service.ts:

When the service receives a status-changed event (Kafka or direct
invocation), insert a row into deal_lifecycle_events BEFORE
running the corpus bootstrap. Include the from_status, to_status,
the transitioning user, and any optional note.

Also wire into backend/src/deals/deals.service.ts wherever
deals.status is updated — that's the canonical write point and
should emit the deal_lifecycle_events row regardless of whether

H4 — property_signing_velocity from rent roll

File: backend/src/services/historical-observations/property-performance-ingestor.ts
in rentRollToCorpusRow.

Compute lease-up velocity from the rent roll's lease_start dates:

  const observationMonthStart = obsDate;
  const observationMonthEnd = new Date(Date.UTC(
    obsDate.getUTCFullYear(), obsDate.getUTCMonth() + 1, 0
  ));

  // Count units with lease_start in a trailing 3-month window
  const trailingWindowStart = new Date(Date.UTC(
    obsDate.getUTCFullYear(), obsDate.getUTCMonth() - 2, 1
  ));

  const recentSignings = (rentRollData.units || []).filter(unit => {
    if (!unit.leaseStart) return false;
    const startDate = new Date(unit.leaseStart);
    return startDate >= trailingWindowStart && startDate <= observationMonthEnd;
  }).length;

  // Velocity = signings per month, averaged over 3 months
  const propertySigningVelocity = recentSignings / 3;

  // ... set on the row:
