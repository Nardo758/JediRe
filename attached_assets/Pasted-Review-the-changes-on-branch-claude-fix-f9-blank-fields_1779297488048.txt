## What was being fixed

## Commit e5963f1b9 — F9 blank-fields fix

Files: `backend/src/services/financial-model-engine.service.ts`, `frontend/src/pages/development/FinancialEnginePage.tsx`.

1. The LLM result from `POST /financial-model/build` had no LP/GP partition fields (lpIrr, lpEm, lpProfit, gpIrr, gpEm, gpTotalDistributions, gpPromoteEarned, totalProfit) because the LLM schema doesn't emit them. The deterministic runner computes them but was only used for cross-check then discarded. Fix: after the deterministic run, copy partition + scalar summary fields into result.summary. The pattern mirrors how result.evidence / result.reasoning are already injected.

2. `normalizeBuildResponse` in FinancialEnginePage.tsx read deterministic-shaped keys (cashOnCashByYear, noiByYear, dscrByYear) but the LLM emits cashOnCash, noiYear1, dscr — with cashOnCash and dscr as `number[]`, not scalars. The DSCR path returned the array, and OverviewTab's `summary.dscr.toFixed(2)` threw TypeError, tripping the error boundary. Fix: handle both shapes safely, average arrays to scalars.

3. `handleBuildModel` swallowed build errors to console only — users saw blank fields with no explanation. Fix: added buildError state, red banner near the BUILD MODEL button, button flips to RETRY BUILD when an error is pending.

## Commit 7063d5875 — hold-period profile refactor

Files: new `backend/src/services/hold-period-profiles.ts`; 8 call sites updated.

Context: a prior intermediate commit (since superseded) bumped six downstream caps from 30 to 36 to match a route-level cap bump. That was the wrong shape — replacing one hardcoded number with another. The user's directive: "These are period profiles that are the typical hold periods based on industry standards and debt."

The refactor introduces:
- STRATEGY_HOLD_PROFILES keyed by development_type (value-add / core / core-plus / opportunistic) with {typical, min, max}
- DEBT_HOLD_PROFILES keyed by loanType (agency / cmbs / bank / bridge / life_co / debt_fund), mirroring fetch_debt_assumptions.ts
- ABSOLUTE_MAX_HOLD_YEARS — computed at module load from `Math.max(...profile.max)`, NOT a literal
- resolveTypicalHold(ctx) and resolveMaxHold(ctx) — resolvers preferring strategy over debt, falling back to 5yr / absolute max

Eight sites refactored:
- 4 Zod schemas (run_joint_goal_seek, fetch_jurisdiction_tax_forecast, generate_roadmap, optimize_capital_structure) → .max(ABSOLUTE_MAX_HOLD_YEARS)
- cashflow.postprocess.ts:769 → Math.min(ABSOLUTE_MAX_HOLD_YEARS, …)
- trafficToProFormaService.ts:1108 → projection-loop cap uses constant
- deals.service.ts:967 → the legacy `strategyType === 'value-add' ? 5 : 7` now calls resolveTypicalHold({ strategy })
- inline-deals.routes.ts /financials handler → default hold (was hardcoded 10) now resolves from deal's development_type via STRATEGY_HOLD_PROFILES; cap uses ABSOLUTE_MAX_HOLD_YEARS

Behavior intent: existing deals should see ZERO number change. Profile `typical` values were calibrated to match the legacy 5/7 split.
