# JEDI RE — Extraction Pipeline Patch Set

**Date:** 2026-04-12
**Validated against:** Alta Porter on Peachtree (deal `c85c5ff5-49d1-42e7-92a2-a82f790587de`)

## What this fixes

The current ingestion pipeline silently produces wrong data and never bridges to the modules. After this patch:

1. **T12 parser** captures Yardi GL-coded charts of accounts correctly (was returning $0 on every value)
2. **Rent roll parser** handles Yardi RRwLC stacked-charge-code layout (was overstating units 10%, missing all sqft, missing `avgMarketRent`)
3. **Tax bill parser** correctly parses DeKalb County GA PDFs and captures appeal scenarios (was returning "Could not extract tax data")
4. **deal_properties join row** is created when extraction provisions a property (was orphaning every new property)
5. **Pipeline status flags** reflect what actually happened (was lying — every successful row insert claimed capsule + library updated)
6. **ProForma is auto-seeded** from extracted actuals using the LayeredValue pattern — platform baselines exist, real data fine-tunes, user overrides win
7. **Cross-document validation** flags variances between docs (e.g., T12 ad valorem vs current tax bill) with scenario payloads
8. **financialModelEngine.buildModel()** auto-fires after extraction, so the financial dashboard has data to read without manual triggering

## How the seeded LayeredValue works

After uploading T12 + rent roll + tax bill for Alta Porter, `deal_assumptions.year1` looks like:

```jsonc
{
  "real_estate_tax": {
    "platform": 1200000,                     // submarket baseline (when wired)
    "t12": 1390005,                           // accrual estimate
    "tax_bill": 1734481,                      // current bill (appeal pending)
    "override": null,                         // no user edit yet
    "resolved": 1734481,
    "resolution": "tax_bill",                 // tax_bill wins per priority rules
    "scenarios": {
      "appeal_settled_at_current": 1734481,
      "appeal_lost_unappealed": 2040293       // captured for sensitivity analysis
    },
    "warning": "Appeal pending — base case uses appealed assessment ($1,734,481)..."
  },
  "insurance": {
    "platform": 101850,                       // 291 units × $350/unit baseline
    "t12": null,                              // not in T12
    "override": null,
    "resolved": 101850,
    "resolution": "platform_fallback",
    "warning": "T12 has no insurance line — using platform baseline of $350/unit/yr"
  },
  "vacancy_pct": {
    "platform": 0.06,
    "t12": 0.069,
    "rent_roll": 0.0962,
    "override": null,
    "resolved": 0.0962,                       // rent roll snapshot wins (freshest)
    "resolution": "rent_roll"
  },
  "noi": {
    "resolved": 4063208,                      // derived from all the above
    "resolution": "platform_fallback"
  }
  // ... 25+ more LayeredValue fields
}
```

When a user edits a Year-1 cell in the proforma UI, call:
```
PATCH /api/v1/inline-deals/:dealId/proforma/year1/override
{ "fieldPath": "real_estate_tax", "value": 1500000 }
```
The override layer is set, `resolved` updates to the override, derived fields recompute, and the audit trail is preserved. Setting `value: null` clears the override and re-resolves from the priority chain.

## What still needs work (not in this patch set)

- **Aged AR, Box Score, Concession Burnoff, T-30 LTO parsers** — exist already; I haven't validated against real samples. Send those docs and I'll mirror the parser fixes if needed.
- **Frontend changes** — the new `/proforma/year1` and `/proforma/year1/override` endpoints exist, but the FinancialDashboard component still reads from `/financial-dashboard/:dealId/summary`. Two options:
  1. Update FinancialDashboard to call `/proforma/year1` instead → cleaner long-term
  2. Update `/financial-dashboard/summary` to read seeded values from `deal_assumptions.year1` when present → backwards-compat preferred
- **Frontend LayeredValue UI** — when a user clicks a Year-1 cell, the UI should show a popover with all source layers and let them override or accept the resolution. Sub-day UI build, not started.
- **Platform baseline service** — the seeder currently passes `platform: null` for all baselines. The location-derived baseline service that subscribes to `dealStore` updates needs to be wired. Until then, seeder uses extraction values 1:1 (which is correct fallback behavior).
- **Tax bill: prior year balance / total payoff** — pdf-parse linearizes the two-column installment coupon. Fix would require positional PDF parsing (e.g., `pdf2json`). Not blocking proforma seeding.
