SECTION 0  —  WHAT THIS DOCUMENT IS
================================================================================

The math spec answers "how does the agent compute each field."
This document answers "what flows to what, in what order, and what
re-fires when something changes."

These are independent contracts. Both must be right.

This document covers:
  1. The static dependency graph (producer → consumer edges)
  2. The temporal execution order (phases, gates, parallelism)
  3. The invalidation propagation rules (what re-fires on change)
  4. The tab-level wiring (which engine outputs hydrate which UI surface)
  5. The cross-module integration (M-prefixed module wiring)
  6. The data store contracts (single source of truth, subscriptions)

The orchestrator code reads this document as its operational instruction
set. The agent reads this only for the phases that surround its own
invocation (it doesn't need to know about UI rendering).

CODE ANCHORS — this document is built on three existing pieces of code.
If they change, this document goes stale:

  1. backend/src/services/module-wiring/module-registry.ts
     The canonical declarative wiring. Every module declares its outputs,
     what it feeds, what it receives, with strength qualifiers.

  2. backend/src/services/module-wiring/module-event-bus.ts
     The runtime propagation layer. Modules emit DATA_UPDATED events;
     downstream modules subscribe and re-fire.

  3. backend/src/services/module-wiring/module-wiring-orchestrator.ts
     The sequencer. Topologically orders modules per dependency graph.

If you change any wiring, update both this document AND the registry.
Drift between the two is a documentation bug.

SECTION 1  —  ARCHITECTURE OVERVIEW
================================================================================

WHERE THE WIRING LAYER SITS

    Frontend (FinancialEnginePage)
           ↓ POST /api/v1/financial-model/build  { dealId, assumptions }
    Backend route handler
           ↓
    Module Wiring Orchestrator       ← reads MODULE_REGISTRY for dep graph
           ↓                          ← uses event bus for propagation
    Phase 1: Document extraction        (parallel)
           ↓
    Phase 2: Source resolution          (LayeredValue assembly)
           ↓
    Phase 3: F9 agent invocation        (Cashflow Agent runs derivations)
           ↓
    Phase 4: Cross-module enrichment    (M11 debt sizing loop, M14 risk)
           ↓
    Phase 5: Reconciliation             (math spec §6 invariants verified)
           ↓
    Persistence (financial_model_results table)
           ↓
    Response to frontend
           ↓
    mergeModelIntoFinancials()          ← folds into f9Financials
           ↓
    13 F9 tabs render from canonical state

ROLES AND RESPONSIBILITIES

  - The MODULE_REGISTRY declares the WHAT (which modules exist, what they
    consume and produce).

  - The orchestrator decides the WHEN (topological order, when each runs).

  - The event bus handles the HOW (data updates propagate to subscribers
    asynchronously).

  - Each module service implements the COMPUTATION (the actual math in
    the math spec).

  - The agent (Cashflow Agent specifically) is invoked DURING phase 3
    of the orchestration. It executes the math spec. It does NOT decide
    what to call next — the orchestrator does that.

  - The frontend (dealStore + tab components) is the CONSUMER. It reads
    from canonical state and renders. It does NOT compute.

SECTION 2  —  STATIC DEPENDENCY GRAPH
================================================================================

This section maps every producer → consumer edge that touches F9.

────────────────────────────────────────────────────────────────────────────────
2.1  WITHIN-F9 DATA FLOWS  (the engine's internal wiring)
────────────────────────────────────────────────────────────────────────────────

Within F9 itself (M09 in the registry), data flows in this graph:

  Document Extractions (T-12, Rent Roll, Tax Bill, OM)
                ↓
        Source Resolver (LayeredValue<T>)
                ↓
       ┌────────┼─────────┐
       ↓        ↓         ↓
    Y1 Pro    Y1 Unit    Y1 Tax
     Forma     Mix       Schedule
       ↓        ↓         ↓
       └────────┼─────────┘
                ↓
       Multi-Year Projections (Y2..Y10)
                ↓
       ┌────────┼─────────┐
       ↓        ↓         ↓
    Returns  Waterfall  Sensitivity
       ↓        ↓         ↓
       └────────┼─────────┘
                ↓
            Decision Tab (synthesis + verdict)

KEY EDGES INSIDE F9:

  Document Extractions → Source Resolver:
      T-12 → year1.{gpr, payroll, vacancy_loss, …}
      Rent Roll → year1.{gpr (alt), unit_mix, vacancy_pct (current)}
      Tax Bill → year1.real_estate_tax
      OM → year1.* (Tier 4 fallback)
      Per-field priority defined in math spec §4.2 (FIELD_PRIORITIES map
      in proforma-seeder.service.ts)

  Source Resolver → Year-1 Pro Forma:
      Resolved values populate proforma.year1[*].resolved
      Sources tagged in proforma.year1[*].source

  Year-1 Pro Forma → Year-1 Unit Mix Tab:
      gpr (when use_unit_mix_for_gpr flag is OFF)
      OR Unit Mix → Year-1 GPR (when flag is ON, gprFromUnitMix overrides)
      This is the only bidirectional flow inside F9; everything else is
      strictly forward.

  Year-1 Pro Forma → Multi-Year Projections:
      Year-1 NOI is the baseline (Year 1 of projections == Year-1 of pro forma)
      HARD INVARIANT: projections[1].noi == proforma.year1.noi
      If they diverge, orchestrator halts.

  User Assumptions → Multi-Year Projections:
      rentGrowth[year] (drives revenue line growth)
      expenseGrowth (drives expense line growth)
      vacancySchedule[year] (drives vacancy assumption per year)
      holdYears (sets cashflow vector length)

  Year-1 Pro Forma + User Assumptions + M07 Traffic →
                                        Multi-Year Projections:
      M07.absorption_rate → vacancy schedule (lease-up mode only)
      M07.predicted_leases_week → revenue ramp (lease-up only)
      M07.rent_growth_adjustment → rentGrowth[year] modifier
      In stabilized mode, M07 is optional. In development mode, required.

  Multi-Year Projections → Returns:
      Equity CF vector → Newton-Raphson IRR
      Stabilized NOI (Year holdYears+1) / exitCap → grossSalePrice
      All §3.8 calcs in math spec.

  Multi-Year Projections → Waterfall:
      cfads(y) → distribution to tiers per §3.12

  Multi-Year Projections → Sensitivity:
      Re-run projections at shocked rent growth × exit cap combinations
      Per math spec §3.9

  All output → Decision Tab:
      Verdict + risk register + walkthrough narrative + integrity checks

────────────────────────────────────────────────────────────────────────────────
2.4  BIDIRECTIONAL CYCLES  (formal two-pass loops)
────────────────────────────────────────────────────────────────────────────────

Two cycles are declared in MODULE_REGISTRY. The orchestrator's
data-flow router knows about these and does NOT treat them as errors.

CYCLE 1: M09 ⇄ M11 (Pro Forma ⇄ Capital Structure)

  Description: Debt sizing requires NOI; final NOI requires final debt service.
  Pass 1: M09 emits stub NOI with placeholder debt service
  Pass 2: M11 sizes debt against stub NOI (LTV / DSCR / debt yield constraints)
  Pass 3: M11 returns final loan terms to M09
  Pass 4: M09 finalises debt service and recomputes cash-on-cash & DSCR

  Implementation: orchestrator runs M09 in "dry mode" first, hands NOI to M11,
  then re-runs M09 with M11's debt output as input.

  Worth noting: this is a stable cycle. It converges in one round-trip
  because the dependence is one-way after the loop closes. If M11 wanted
  to re-size debt based on M09's final DSCR, the loop would not
  converge — but in practice M11 sizes once based on stub NOI, and
  M09's final NOI is downstream-only.

CYCLE 2: M09 ⇄ M14 (Pro Forma ⇄ Risk Dashboard)

  Description: M14 wants to risk-adjust IRR; F9 wants risk-adjusted reserves.
  Pass 1: M09 emits unweighted IRR
  Pass 2: M14 computes composite_risk_score and reserve_overrides
  Pass 3: M14 emits cap_rate_adjustment_bps
  Pass 4: M09 re-runs with adjusted exit cap (only Returns recomputes)

  This cycle is OPTIONAL — runs only when M14 is enabled and configured.
  If skipped, F9 uses user-input exit cap directly.


================================================================================
SECTION 3  —  EXECUTION PHASES & SEQUENCING
================================================================================

The orchestrator runs F9 in 6 phases. Each phase has explicit gates;
phase N+1 cannot start until phase N's gate passes.

────────────────────────────────────────────────────────────────────────────────
PHASE 1  —  DOCUMENT EXTRACTION  (parallel, async)
────────────────────────────────────────────────────────────────────────────────

INPUT:    Uploaded files in deal_data (T-12 PDF, Rent Roll Excel, Tax Bill PDF, OM PDF)

EXECUTION:
  parallel:
    T-12 Parser           (Yardi-aware, fallback to generic)
    Rent Roll Parser       (RRwLC, fallback to generic)
    Tax Bill Parser
    OM Parser              (Claude-extraction)

  Each parser writes its output to deal_data.extraction_<type>.

GATE:
  At least one of {extraction_t12, extraction_rent_roll,
  extraction_tax_bill} must exist. If none present:
    - Surface integrity check 'extraction_data_missing' (status: error)
    - Halt. Do not proceed.
  (Per the patch in §13 GAP-1 of the math spec.)

OUTPUT:   deal_data.extraction_* JSONB blobs

TIME BUDGET: 15-60 seconds per parser; runs in parallel.
PARALLELISM: high (each parser is independent)
IDEMPOTENCY: idempotent (re-parsing same file produces same output;
                          parsers ARE deterministic enough for caching)

────────────────────────────────────────────────────────────────────────────────
PHASE 2  —  SOURCE RESOLUTION  (LayeredValue assembly)
────────────────────────────────────────────────────────────────────────────────

INPUT:    All extraction capsules from Phase 1 + user overrides + assumptions

EXECUTION:
  for each canonical field in proforma.year1:
    sources = collect from extractions per FIELD_PRIORITIES (math spec §4.2)
    resolved = highest-priority non-null source
    detect collisions (math spec §4.3)
    write LayeredValue<T> with provenance

  Source: backend/src/services/proforma-seeder.service.ts
  Function: seedProFormaYear1()

GATE:
  - Every required field has either:
      a) A non-null resolved value, OR
      b) An explicit null with a reason in metadata
  - No silent zeros. No silent missing fields.

OUTPUT:   deal_assumptions.year1 (JSONB column)
          One row per LayeredValue<T> field with source attribution

TIME BUDGET: <2 seconds (pure data manipulation, no LLM)
PARALLELISM: serial (resolution is not parallelizable; each field depends
                     on the FIELD_PRIORITIES lookup)
IDEMPOTENCY: idempotent given same extraction inputs

────────────────────────────────────────────────────────────────────────────────
PHASE 3  —  F9 AGENT INVOCATION  (Cashflow Agent runs the model)
────────────────────────────────────────────────────────────────────────────────

INPUT:    DealContext (Phase 2 output + assumptions + cache)
          User overrides (per-field, per-year)
          dealType (existing | development | redevelopment)
          Math spec document

EXECUTION:
  1. Cache check: if DealContext key matches recent cache (< 24h),
     start from cached state.
  2. Mode dispatch: branch on dealType per math spec §3.14.
  3. Year-1 build: §3.6 of math spec.
  4. Multi-year projection: §3.6 forward.
  5. Disposition: §3.7.
  6. Returns: §3.8.
  7. Reasoning + collision report + walkthrough: §7.

  The agent runs the math spec end-to-end, then returns ModelResults.

GATE:
  - Math spec §6 hard invariants (INV-1 through INV-10) all hold.
  - All required fields are populated (or explicitly null with reason).
  - Reasoning metadata attached to every value.

OUTPUT:   ModelResults JSON (per math spec §11)

TIME BUDGET: 30-90 seconds (LLM-bound)
PARALLELISM: serial (the agent is one process)
IDEMPOTENCY: NOT idempotent — agent calls vary slightly run-to-run.
              Cache layer mitigates this; cache hit is idempotent.

────────────────────────────────────────────────────────────────────────────────
PHASE 4  —  CROSS-MODULE ENRICHMENT  (cycle resolution)
────────────────────────────────────────────────────────────────────────────────

INPUT:    Phase 3 output (raw ModelResults)
          Other modules' outputs available in DealContext

EXECUTION (in order):

  Step 4a. M11 cycle (debt sizing):
    if (M11 enabled and assumptions.financing.constraint == 'dscr_min'):
      Run M11 with stub NOI from Phase 3
      Receive final loan terms
      Re-run F9 §3.5 (debt schedule) and §3.6 (Y1 NOI) with new debt
      ModelResults.debtMetrics updated

  Step 4b. M14 cycle (risk-adjusted exit cap):
    if (M14 enabled):
      Run M14 with Phase 3 IRR + cash flows
      Receive cap_rate_adjustment_bps
      Adjust exitCap by adjustment
      Re-run F9 §3.7 (disposition) and §3.8 (Returns) only
      ModelResults.summary.irr/em/exitCap updated

  Step 4c. M22 push (deal-level baseline):
    Emit DATA_UPDATED event with NOI and cash_flow_projections.
    M22 subscribes; updates deal_monthly_actuals baseline for variance
    tracking. (Does not block F9 completion.)

GATE:
  - Both cycles converge in single round-trip.
  - If cycle does not converge (which shouldn't happen but safeguard
    is required), halt and report.

OUTPUT:   Enriched ModelResults

TIME BUDGET: 5-15 seconds (M11 is fast; M14 is fast)
PARALLELISM: 4a and 4b can run in parallel after Phase 3.
              4c is fire-and-forget (event emission only).
IDEMPOTENCY: idempotent given same inputs

────────────────────────────────────────────────────────────────────────────────
PHASE 5  —  RECONCILIATION  (server-side belt-and-suspenders)
────────────────────────────────────────────────────────────────────────────────

INPUT:    Final ModelResults from Phase 4

EXECUTION:
  Re-verify math spec §6 hard invariants on the agent's output:
    INV-1 through INV-10
    All soft checks (SOFT-1 through SOFT-10)

  This is duplicate to what the agent already did in Phase 3, but
  belt-and-suspenders is correct. The agent could have a bug; the
  server-side check catches it before persistence.

GATE:
  Hard invariant failure → halt, return 500 with diagnostic detail.
  Soft check failure → flag in integrityChecks but proceed.

OUTPUT:   Verified ModelResults + integrityChecks array

TIME BUDGET: <1 second (pure arithmetic)
PARALLELISM: not applicable
IDEMPOTENCY: idempotent

────────────────────────────────────────────────────────────────────────────────
PHASE 6  —  PERSISTENCE & RESPONSE
────────────────────────────────────────────────────────────────────────────────

INPUT:    Verified ModelResults

EXECUTION:
  1. Write to financial_model_results table with version tag.
  2. Update deal_assumptions row (assumptions_used).
  3. Emit DATA_UPDATED event for downstream subscribers (M22, M14 if pull).
  4. Cache result in DealContext cache with TTL = 24h.
  5. Return ModelResults to frontend.

GATE:    None.

OUTPUT:  HTTP 200 response with ModelResults envelope

TIME BUDGET: 1-2 seconds
PARALLELISM: write and emit can be parallel
IDEMPOTENCY: idempotent (same input → same persistence)


================================================================================
SECTION 4  —  INVALIDATION RULES
================================================================================

When something changes, only specific fields invalidate. The orchestrator
identifies the dirty fields and re-runs the minimum subset of phases
needed to refresh them.

────────────────────────────────────────────────────────────────────────────────
4.1  USER-DRIVEN INVALIDATION
────────────────────────────────────────────────────────────────────────────────

CHANGE: User uploads new T-12

  INVALIDATES: All Phase 1 outputs (T-12 capsule)
  RE-RUN: Phases 1-6 (full)
  EXCEPTION: If new T-12 is identical to prior, cache short-circuits.

CHANGE: User uploads new rent roll

  Same as T-12.

CHANGE: User uploads new tax bill

  INVALIDATES: extraction_tax_bill capsule, real_estate_tax field,
              taxes.* domain
  RE-RUN: Phase 2 (re-resolve real_estate_tax),
          Phase 3 (only §3.13 tax sub-derivation),
          Phase 5, Phase 6.
  Optimization: full agent re-run not necessary; only tax module re-fires.

CHANGE: User edits purchasePrice in Assumptions tab

  INVALIDATES: closingCosts, docStamps, intangibleTax, titleInsurance,
               totalAcqCost, totalEquity,
               LTV-derived loanAmount (if LTV is canonical),
               propertyTax (reassessment base),
               going-in cap, exit-implied prices, ALL returns metrics
  RE-RUN: Phase 3 (most fields recompute), Phases 4-6.

CHANGE: User edits marketRent

  INVALIDATES: GPR(y) for all y, lossToLease$(y) for all y,
               baseRevenue, EGR, mgmtFee, NOI, CF, DSCR, returns
  RE-RUN: Phase 3, Phase 4, Phase 5, Phase 6.
  PRESERVED: tax schedule, sources & uses, debt schedule (rate-driven, not
             revenue-driven)

CHANGE: User edits vacancy_y1 only (year 1 only)

  INVALIDATES: vacancy$(1), baseRevenue(1), EGR(1), NOI(1), CF(1), DSCR(1),
               Y1 returns; Y2+ unchanged.
  RE-RUN: Phase 3 with cached Y2+ projections.
  This is rare — most users don't override single years.

CHANGE: User edits rentGrowth[Y1] only

  INVALIDATES: cumulativeGrowth(y) for y >= 1, GPR(y) for y >= 1,
               ALL operating fields y >= 1, returns
  RE-RUN: Phase 3 (full multi-year), Phases 4-6.

CHANGE: User edits rate (debt rate)

  INVALIDATES: monthlyPmt, debtService(y) for y > ioPeriod,
               principal(y) for y > ioPeriod, DSCR(y) for affected years,
               Y2+ CF, returns
  RE-RUN: M11 cycle if enabled (Phase 4a), then Phase 4b, 5, 6.
  Note: rate change does NOT change NOI; only debt service.

CHANGE: User edits exitCap only

  INVALIDATES: grossSalePrice, dispositionDocStamps, equityProceeds,
               exitYear cashflow, IRR, equityMultiple
  RE-RUN: Math spec §3.7 (disposition) and §3.8 (Returns) only.
          Operating model (Phase 3 §3.6) unchanged.
  This is the cheapest re-run path; sensitivity matrix uses this.

CHANGE: User edits holdYears

  INVALIDATES: stabilizedNOI (different forward year),
               cashflow vector length, IRR, EM, sensitivity matrix
  RE-RUN: Phase 3 with extended/truncated projection horizon, then 4-6.

CHANGE: User edits LP/GP equity split or promote tiers

  INVALIDATES: waterfall distributions only
  RE-RUN: Math spec §3.12 only. Leaves all other ModelResults intact.

CHANGE: User edits dealType (existing → development)

  INVALIDATES: EVERYTHING. Mode dispatch in §3.14 changes. Different
               formulas apply. Different lease-up curves.
  RE-RUN: Full Phases 3-6.

────────────────────────────────────────────────────────────────────────────────
4.2  EXTERNAL-MODULE-DRIVEN INVALIDATION
────────────────────────────────────────────────────────────────────────────────

CHANGE: M07 traffic engine recalibrates

  TRIGGER: M07 emits DATA_UPDATED event
  INVALIDATES: For development/lease-up deals only:
                 vacancy schedule, lease-up curve, Y1-Y2 revenue
               For stabilized deals:
                 nothing (M07 is optional)
  RE-RUN: Phase 3 (if dev/lease-up), Phases 4-6.
  Frontend behavior: F9 shows "stale data" badge until re-run completes.

CHANGE: M14 risk model updates (new event in M35 fires through M14)

  TRIGGER: M14 emits DATA_UPDATED event
  INVALIDATES: cap_rate_adjustment_bps, reserve_overrides
  RE-RUN: Phase 4b only (M14 cycle re-fire), then 5, 6.
  Operating model unchanged.

CHANGE: M05 publishes new submarket rent trend

  TRIGGER: M05 emits DATA_UPDATED event
  INVALIDATES: marketRent (if Tier-3 derivation), rentGrowth[year]
                derivation
  RE-RUN: Phase 3 (all multi-year), Phases 4-6.
  COST: full agent run. Should debounce — many M05 updates per day.

CHANGE: M27 publishes new comp valuation data

  TRIGGER: M27 emits DATA_UPDATED event
  INVALIDATES: exit cap rate corroboration
  RE-RUN: NOTHING automatically. M27 is a corroboration source, not a
          driver. The user sees a "comp data updated" notification and
          can choose to re-run.

CHANGE: M11 changes available debt products

  TRIGGER: M11 emits DATA_UPDATED event
  INVALIDATES: only if user has selected "use M11 recommendation"
  RE-RUN: M11 cycle re-fires (Phase 4a), then 5, 6.

CHANGE: M26 (Real Estate Tax Engine) updates

  TRIGGER: M26 emits DATA_UPDATED event
  INVALIDATES: propertyTax(y) for all y
  RE-RUN: Phase 3 §3.13 only, then 5, 6.

────────────────────────────────────────────────────────────────────────────────
4.3  CACHE INVALIDATION
────────────────────────────────────────────────────────────────────────────────

DealContext cache TTL: 24 hours.

Cache key:    sha256(dealId + assumptionsHash + dealDataHash + specVersion)

Cache hit conditions:
  - Cache key exact match
  - Spec version matches running spec
  - No upstream module DATA_UPDATED event since cache write

Cache invalidation triggers (all force a full Phase 1-6 re-run):
  - Any of the user-driven invalidations in §4.1 above
  - DATA_UPDATED event from any upstream module
  - Math spec version bump
  - Manual cache flush by admin

The orchestrator never silently serves stale data. If cache integrity
is uncertain, treat as miss.


================================================================================
SECTION 5  —  TAB-LEVEL WIRING
================================================================================

This section maps each F9 tab to the canonical state it consumes.

GLOBAL CONVENTION: All tabs read from f9Financials (the canonical state
shape returned by GET /api/v1/deals/:id/financials), augmented by
ModelResults from Build via mergeModelIntoFinancials.

Tabs do NOT compute. Tabs render.

────────────────────────────────────────────────────────────────────────────────
TAB: OVERVIEW (after Move 1 trim)
────────────────────────────────────────────────────────────────────────────────

READS:
  f9Financials.summary.{irr, equityMultiple, avgCoC, noi, dscr, goingInCap}
  f9Financials.dealName, dealAddress
  f9Financials.proforma.integrityChecks (preview)
  f9Financials.reasoning.walkthroughOneLiner

DOES NOT READ:
  detailed Y1 operating statement (lives in Pro Forma tab)

SECTION 6  —  CONCURRENCY RULES
================================================================================

────────────────────────────────────────────────────────────────────────────────
6.1  WHAT CAN RUN IN PARALLEL
────────────────────────────────────────────────────────────────────────────────

PARALLELIZABLE:
  - Document parsers in Phase 1 (T-12, RR, Tax Bill, OM)
  - Sensitivity matrix cells (each cell is independent rerun of §3.6-§3.8)
  - Multiple deals being modeled simultaneously (cross-deal concurrency)
  - Phase 4a (M11 cycle) and Phase 4b (M14 cycle) — independent

NOT PARALLELIZABLE:
  - Phase 1 → Phase 2 → Phase 3 sequence (data dependencies)
  - Source resolution within a single deal
  - Multi-year projection within a single deal (year N depends on year N-1
    via cumulative growth + ending balances)
  - Waterfall tier resolution within a single distribution year

────────────────────────────────────────────────────────────────────────────────
6.2  RACE CONDITIONS TO BE AWARE OF
────────────────────────────────────────────────────────────────────────────────

RACE: Two users edit the same deal's assumptions simultaneously.
  RESOLUTION: Last-write-wins on the assumptions row. The orchestrator
              uses the assumptions present at Build invocation. No
              optimistic locking currently. (Acceptable risk; deal-team
              size is small.)

RACE: M07 republishes mid-Build.
  RESOLUTION: Build uses M07 data snapshot at invocation time. Stale
              data warning surfaces post-Build if M07 has updated since.

RACE: User clicks Build twice rapidly.
  RESOLUTION: Frontend disables Build button while building (state
              `building` in FinancialEnginePage). Backend should also
              dedupe via cache key.

────────────────────────────────────────────────────────────────────────────────
6.3  IDEMPOTENCY GUARANTEES
────────────────────────────────────────────────────────────────────────────────

  Phase 1 parsers:        Idempotent (same file → same extraction)
  Phase 2 resolver:        Idempotent (same extractions → same resolution)
  Phase 3 agent:           NOT idempotent (LLM stochasticity); cache mitigates
  Phase 4a (M11 cycle):    Idempotent given same inputs
  Phase 4b (M14 cycle):    Idempotent given same inputs
  Phase 5 reconciliation:  Idempotent
  Phase 6 persistence:     Idempotent (same input → same row written)

SECTION 7  —  FAILURE MODES
================================================================================

────────────────────────────────────────────────────────────────────────────────
7.1  FAILURE: NO EXTRACTION DATA  (Phase 1 produces nothing useful)
────────────────────────────────────────────────────────────────────────────────

SYMPTOM: All four extraction capsule keys absent on deal_data.

DETECTION: Phase 1 gate fails.

RESPONSE:
  - Halt orchestration before Phase 2.
  - Surface integrityChecks: 'extraction_data_missing' (status: error)
  - Frontend displays a hard banner: "No T-12, rent roll, or tax bill
    found for this deal. Upload documents to populate the model."
  - Save what was attempted; allow user to retry after upload.

────────────────────────────────────────────────────────────────────────────────
7.2  FAILURE: PARTIAL EXTRACTION  (some capsules present, others absent)
────────────────────────────────────────────────────────────────────────────────

SYMPTOM: Some of {extraction_t12, extraction_rent_roll,
                  extraction_tax_bill, extraction_om} present; others absent.

DETECTION: Source resolver in Phase 2 finds null for some required fields.

RESPONSE:
  - Proceed with available data.
  - Resolved value is null where source was absent.
  - integrityCheck: 'partial_extraction' (status: warn) listing missing fields.
  - Confidence on derived fields is reduced (math spec §5).
  - Frontend displays partial state with caveats.

────────────────────────────────────────────────────────────────────────────────
7.3  FAILURE: INVARIANT VIOLATION  (Phase 5 reconciliation fails)
────────────────────────────────────────────────────────────────────────────────

SYMPTOM: Math spec INV-1 through INV-10 not all satisfied.

DETECTION: Phase 5 server-side check.

RESPONSE:
  - Halt. Do not persist or return to frontend.
  - Log full diagnostic: invariant ID, year, expected, actual.
  - Return HTTP 500 with diagnostic detail.
  - Alert on engineering channel.

This is a hard failure indicating an agent bug. It should not happen
in production; if it does, halt and investigate.

────────────────────────────────────────────────────────────────────────────────
7.4  FAILURE: AGENT TIMEOUT  (Phase 3 exceeds time budget)
────────────────────────────────────────────────────────────────────────────────

SYMPTOM: Cashflow Agent runs past 120-second timeout.

DETECTION: Frontend's apiClient timeout fires.

RESPONSE:
  - Frontend shows "Build timed out" with retry button.
  - Backend logs timeout for debugging.
  - Cache may be partially written; cache key validation prevents serving.

────────────────────────────────────────────────────────────────────────────────
7.5  FAILURE: M11 CYCLE NON-CONVERGENCE  (Phase 4a)
────────────────────────────────────────────────────────────────────────────────

SYMPTOM: M11 wants more debt than DSCR allows; loop oscillates.

DETECTION: After 3 round-trips, no convergence.

RESPONSE:
  - Halt cycle. Use M11's last computed loan terms.
  - integrityCheck: 'capital_stack_unconverged' (status: warn)
  - Frontend renders with warning badge.

────────────────────────────────────────────────────────────────────────────────
7.6  FAILURE: USER OVERRIDE INVALIDATES CACHE BUT NEW BUILD FAILS
────────────────────────────────────────────────────────────────────────────────

SYMPTOM: User edits a field, Build fires, fails. Old cached state still
exists but assumptions don't match it anymore.

DETECTION: Cache hit fails the assumptionsHash check.

RESPONSE:
  - Show user the prior valid model with a "stale" badge.
  - Allow user to revert override or retry Build.
  - Never silently roll back override.

SECTION 9  —  DATA STORE CONTRACTS
================================================================================

────────────────────────────────────────────────────────────────────────────────
9.1  CANONICAL STATE  (single source of truth)
────────────────────────────────────────────────────────────────────────────────

The frontend treats f9Financials (post-merge) as the single source of
truth for all F9 tabs. No tab maintains a local copy of model results.

Per project memory: dealStore.ts is the message bus — no direct imports
between module components. All module-to-module communication routes
through the store.

f9Financials shape: F9DealFinancials (defined in
frontend/src/pages/development/financial-engine/types.ts).

This shape is augmented by mergeModelIntoFinancials() in
FinancialEnginePage.tsx to fold ModelResults into f9Financials, producing
mergedFinancials. All tabs receive mergedFinancials.

────────────────────────────────────────────────────────────────────────────────
9.2  STORE SUBSCRIPTIONS
────────────────────────────────────────────────────────────────────────────────

Per dealStore.ts, hooks expose specific slices:

  useDealOverview()             → summary metrics
  useUnitMixIntelligence()      → unit mix data (M29)
  useDevCapacity()              → development envelope (M03)
  useStrategyArbitrage()        → strategy data (M08)
  useProForma()                 → proforma data
  useDealBasics()               → core deal info
  useDealType()                 → existing | development | redevelopment
  useDealTypeConfig()           → type-specific config
  useStrategyAvailability()     → which strategies apply
  useAssumptions()              → user-editable assumptions

Tabs MUST use these hooks rather than direct store reads. The hooks
provide stable subscription patterns and prevent unnecessary re-renders.

────────────────────────────────────────────────────────────────────────────────
9.3  COMPONENT-LOCAL STATE
────────────────────────────────────────────────────────────────────────────────

Components may hold local state for:
  - Edit mode toggles (Compact ↔ Detailed)
  - Modal open/close state
  - Form draft values before save
  - Sort/filter state for tables
  - Hover / focus state
  - Active tab within a tab

Components must NOT hold local state for:
  - Model results
  - Assumptions
  - Source documents
  - Deal context

────────────────────────────────────────────────────────────────────────────────
9.4  WRITE PATHS
────────────────────────────────────────────────────────────────────────────────

User edits flow through these endpoints:

  PATCH /api/v1/deals/:dealId/financials/override
    Per-field override on deal_assumptions.year1
    Writes LayeredValue<T> with source='override'
    Returns updated f9Financials

  POST /api/v1/financial-model/build
    Triggers full Build (Phases 1-6)
    Returns ModelResults

  POST /api/v1/deals/:dealId/proforma/seed
    Manually triggers Phase 2 (re-resolve from extractions)
    Useful when extraction capsules update without F9 invalidation

  POST /api/v1/deals/:dealId/financials/reparse
    Triggers Phase 1 (re-extract documents)

  PATCH /api/v1/financial-model/:modelId
    Saves named version
    Enables version comparison (Move 3)

  STATUS: Cache key construction is in place; subscriber loop is not.

APPENDIX B  —  TERMINOLOGY GLOSSARY (WIRING-SPECIFIC)
================================================================================

  CANONICAL STATE     The single source of truth for F9 data.
                       For frontend: f9Financials (post-merge).
                       For backend: deal_assumptions.year1 + ModelResults.

  CYCLE               A bidirectional flow between two modules requiring
                       multi-pass resolution. Two declared: M09⇄M11, M09⇄M14.

  DATA_UPDATED        Event type emitted when a module's output changes.
                       Other modules subscribe to invalidate downstream state.

  GATE                A phase-completion check. Phase N+1 cannot start
                       until Phase N's gate passes.

  IDEMPOTENT          Re-running with same inputs produces same output.
                       Most phases are; agent invocation is not.

  INVALIDATION        Marking a downstream field as stale because an
                       upstream input changed. Triggers re-derivation
                       of just that field plus its downstream chain.

  LAYEREDVALUE        A typed wrapper around any value carrying source,
                       confidence, and reasoning metadata. Defined in
                       proforma-seeder.service.ts. Math spec §7.1.

  MERGE FUNCTION      mergeModelIntoFinancials() in FinancialEnginePage.tsx.
                       Folds ModelResults from Build into f9Financials.
                       Critical wiring point; bug surface area.

  MODULE_REGISTRY     The canonical declarative wiring map. Source of truth
                       for who-feeds-whom. Math spec §4.2 references it
                       indirectly via FIELD_PRIORITIES.

  PHASE               A discrete sequenced step in the orchestrator
                       (Phases 1-6 in §3).

  PRODUCER            A module that emits data via outputs.

  CONSUMER            A module that subscribes via receivesFrom.

  STORE               dealStore.ts — the Zustand store + hook system that
                       is the single source of truth for frontend state.
                       "store as message bus" — no direct cross-component
                       imports; everything routes through hooks.

  STRENGTH            ModuleDependency.strength: 'required' or 'optional'.
                       Required dependencies must resolve before consumer runs.
                       Optional dependencies fall back to defaults.

  SUBSCRIBE           A module registers a handler for an event type via
                       moduleEventBus.subscribe(eventType, handler).
