Why the inline-recompute on stamp mismatch is the right pattern
The naive cache invalidation approach would have been: when STANCE toggles, fire a recompose. That works but has problems:

Recompose is expensive (DB round-trips, full y1 reassembly)
Failure modes (network, timeout) leave the cache in an unknown state
Race conditions between multiple toggle events
The recompose itself has to be triggered from the right place, which is more code

What you shipped instead — stamp the treatment into the cache payload, validate on read, inline-recompute from a pure function on mismatch — has none of those problems:

Pure function (amortizeConcessions from deal_data.concession_records)
No DB calls on mismatch
Stale cache becomes self-correcting on next read
Negligible hot-path cost
Race-free (the stamp is always self-consistent with the value)

This is the architectural pattern worth keeping for similar cases: cache validity carried in the cache payload itself, with a cheap pure-function fallback on mismatch. Apply it next time a derived value depends on a toggleable input — there will be more of these as OperatorStance Phase 1 lands and stance fields start cascading through computations.
