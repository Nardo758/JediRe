DATA LIBRARY CONSOLIDATION + ROLLUPS STUB

CONTEXT
Three problems converge in this dispatch:

1. Parallel "Data Library" surfaces exist. F10 Settings → Data → 
   Data Library shows 299 assets. /archive/library shows 1,695 
   files. Both call themselves "Data Library." Same underlying 
   data, two competing UI surfaces with the same name.

2. The AR ARCHIVE button on the Terminal F-key bar shouldn't 
   exist. Data Library is a Settings/Data area, not a Terminal 
   page. F-key bar should be F1-F10 only.

3. A Rollups view is needed (cohort analytics — expense and 
   trend rollups by asset type, with curation). Full design 
   specced separately. This dispatch ships a placeholder tab 
   that shows asset-type groupings using already-populated 
   fields; full Rollups implementation comes after M02 zoning 
   + Atlanta enrichment populate the cohort dimensions.

After this dispatch:
- F10 Settings → Data → Data Library is the single canonical home
- Four tabs: Assets | Files | Inbox | Rollups
- AR ARCHIVE button removed from Terminal F-key bar
- Old /archive/library and /archive/intake routes either redirect 
  to the new structure or render the same component as the 
  corresponding tab
- Cross-linking between Assets and Files tabs is interactive

═══════════════════════════════════════════════════════════════════
PRINCIPLES
═══════════════════════════════════════════════════════════════════

1. INFORMATION ARCHITECTURE AUDIT IS MANDATORY. Before any code 
   changes, grep the codebase for every place "Data Library" is 
   referenced. Report findings. The audit that should have happened 
   when Phase 4 was dispatched. Catch any third or fourth parallel 
   surface I haven't seen.

2. ONE CANONICAL HOME. F10 Settings → Data → Data Library is the 
   single Data Library page. Every other entry point either 
   redirects here or is removed. No exceptions.

3. PRESERVE EXISTING FUNCTIONALITY. Both the Asset view (currently 
   on Data Library page) and the File browser (currently at 
   /archive/library) must continue to work. This dispatch 
   relocates and connects them, doesn't redesign either.

4. THE ROLLUPS TAB IS A STUB, NOT A FULL IMPLEMENTATION. Show 
   asset-type grouping with currently-populated fields. Reserve 
   the navigation slot. Make it clear that full cohort analytics 
   ship later. Don't fake depth.

5. CROSS-LINKING IS THE INTEGRATION. Tabs don't just coexist; 
   they reference each other. Clicking a file count on an asset 
   row navigates to Files tab filtered to that asset. Clicking 
   a property name in Files navigates to the Asset row.

6. PAIRED-READ VERIFICATION. After consolidation, both old 
   workflows still produce the same data: a user can find any 
   file they found before, see any asset they saw before, with 
   no data loss.

STEP 8 — CROSS-LINKING INTEGRATION
═══════════════════════════════════════════════════════════════════

Verify the following user journeys work end-to-end:

A. Assets → Files journey:
   1. User on Assets tab clicks the FILES count for a specific asset

   1. User on per-property page (/archive/properties/:parcelId)
