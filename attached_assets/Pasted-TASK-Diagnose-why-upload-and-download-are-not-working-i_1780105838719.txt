═══════════════════════════════════════════════════════════════════════════
STEP 2 — REPRODUCE UPLOAD (10 min)
═══════════════════════════════════════════════════════════════════════════

Do NOT use the browser. Use curl to exactly replicate what the frontend POSTs.
This bypasses every layer of frontend confusion and tests the actual contract.

2A. Read the FilesSection.tsx upload handler. Find the exact fetch/api call.
    Note:
       - HTTP method
       - Full URL path
       - Headers (Content-Type, Authorization)
       - Body shape (FormData? JSON? multipart field name?)
    Cite the lines.

2B. Reproduce exactly with curl. Use a real file:
       echo "test content" > /tmp/test-upload.txt
       curl -v -X <METHOD> \
         -H "Authorization: Bearer <TOKEN>" \
         <HEADERS_FROM_2A> \
         <BODY_SHAPE_FROM_2A> \
         http://localhost:<PORT>/<PATH>
    Capture the full response — status code, headers, body.

2C. Same moment, capture the backend log line(s) for this request. Look for:
       - Did the request hit the route? (express logs the path)
       - Did it reach the controller? (any console.log or req-id traces)
       - Did it reach the service?
       - Did it write to storage?
       - Did it INSERT into data_library_files?
       - Did anything throw?

2D. After the request: query the DB.
       SELECT id, file_name, deal_id, submarket_id, storage_path, deleted_at,
              created_at FROM data_library_files ORDER BY created_at DESC LIMIT 3;
    Did a row appear? With what columns populated?

2E. Did the file actually land in storage? If S3: list the bucket prefix. If
    local disk: ls the upload directory. Compare to storage_path from the row.

CATEGORIZE the failure:
   [ ] Route 404 — frontend path doesn't match backend mount
   [ ] Route 401/403 — auth middleware rejects
   [ ] Route 400 — body shape wrong, multer field name mismatch, etc.
   [ ] Route 500 — backend exception, capture the stack trace
   [ ] Route 200 but no DB row — service swallows error or commits to wrong table
   [ ] Route 200 + DB row but no storage file — storage write failed silently
   [ ] Route 200 + DB row + file — backend works, problem is purely frontend rendering

STEP 3 — REPRODUCE DOWNLOAD (10 min)
═══════════════════════════════════════════════════════════════════════════

3A. Pick the file ID from STEP 2D (or any existing row in data_library_files).

3B. Read the download handler in FilesSection.tsx (or shared DocumentList if
    refactored). Note the exact request shape — same fields as 2A.

3C. Curl it:
       curl -v -X <METHOD> \
         -H "Authorization: Bearer <TOKEN>" \
         http://localhost:<PORT>/<PATH>/<FILE_ID>/<download_path>
    Full response — status, headers, body.

3D. Tail backend logs for this request. Same questions as 2C: which layer did
    it reach, did it throw?

3E. If the response was supposed to be a signed URL — was the URL well-formed?
    Did curl that URL too:
       curl -v "<SIGNED_URL>"
    What did it return? S3 SignatureDoesNotMatch? AccessDenied? Or the file?

3F. If the response was supposed to be raw bytes — did Content-Type and
    Content-Disposition headers come back correctly?

CATEGORIZE the failure (same buckets as STEP 2, plus):
   [ ] Endpoint returns URL but URL is invalid/expired/wrong bucket
   [ ] Endpoint returns bytes but Content-Disposition missing — browser opens
       inline instead of downloading
   [ ] Endpoint 404 — file_id route never registered
   [ ] Endpoint expects different param than frontend sends
