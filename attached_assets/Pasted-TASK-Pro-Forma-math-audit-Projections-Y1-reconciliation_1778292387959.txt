PHASE 0a — PRO FORMA MATH AUDIT

For every line item rendered on the Pro Forma tab, document:

  1. Display value          (UI render)
  2. DB value               (deals.assumptions / deal_data)
  3. Resolution path        (which LV layer wins; fallback chain)
  4. Formula                (if computed, e.g. NOI = EGI − Total OpEx)
  5. Input sources          (per-input source citation)
  6. Unit convention        (decimal? whole-percent? dollars per unit?)
  7. First-principles check (recompute from authoritative inputs;
                             does it match display?)

Group by section:
  Section 1 — Revenue (GPR, Loss to Lease, Vacancy, Concessions,
              Bad Debt, Net Rental Income, Other Income, EGI)
  Section 2 — Operating Expenses (each named category + custom_opex)
  Section 3 — NOI bridge (Total OpEx, NOI, Replacement Reserves,
              NOI After Reserves)
  Section 4 — Capital metrics (Debt Service, CF, DSCR, Cap Rate,
              Yield on Cost, Cash on Cash where rendered)

Per-line classification:

  ✓ MATH_OK      — formula correct, inputs traced, units consistent,
                   first-principles recompute matches display
  ⚠ UNIT_DRIFT   — formula correct but stored unit differs from
                   platform convention (Bishop's opexGrowthPct = 2.8
                   instead of 0.028 is the canonical example)
  ⚠ STALE_CACHE  — display value differs from first-principles
                   recompute, but matches a prior seed; forceReseed
                   would close the gap
  ✗ MATH_BUG     — formula wrong, subtotal doesn't add, or input
                   pulled from wrong source
  ✗ GROWTH_LEAK  — Pro Forma value has forward growth applied where
                   it shouldn't (Pro Forma is AS-IS, not Y1-projected)
  ? UNCLEAR      — couldn't resolve without further investigation

Each finding cites:
  - ProFormaSummaryTab.tsx (display location)
  - proforma-adjustment.service.ts (resolution + toDollarRow)
  - proforma-seeder.service.ts (seed compute)
  - DB column where value is persisted

PHASE 0b — Y1 RECONCILIATION

For every line item appearing on BOTH Pro Forma AND Projections Y1,
build a comparison table:

  | Line | Pro Forma | Projections Y1 | Match | Path |

Where Match = Y, document the resolution path producing the
identical value (e.g., "both resolve via gpr LV broker layer").

Where Match = N, classify the divergence:

  INTENTIONAL    — explicit Y1 user override or M07-derived value
                   patches the Y1 cell; provenance traceable
  COMPUTE_DRIFT  — same inputs, different code paths producing
                   different values (bug)
  SOURCE_DRIFT   — Pro Forma reads from one LV layer, Projections
                   Y1 reads from another (architecture question:
                   which is source of truth?)
  GROWTH_BUG     — Projections Y1 has growth applied that shouldn't
                   apply ("Y1 = this year," growth begins Y2)
  ROUNDING       — values differ by < 0.5%, likely order-of-
                   operations rounding

For every divergence, cite the code path that computes Projections
Y1 and the path that computes Pro Forma; identify the divergence
point.

PHASE 0c — UNIT NORMALIZATION SWEEP

This audit doubles as the catalog for the open
ASSUMPTION_UNIT_NORMALIZATION concern.

Scan deals.assumptions on both deals for every percent-typed field
(any key ending in _pct, _percent, _rate, or matching known growth /
yield / vacancy fields). For each field, classify:

  DECIMAL       — stored value < 1 (e.g., 0.03 for 3%)
  WHOLE_PERCENT — stored value > 1 (e.g., 2.8 for 2.8%)
  ZERO_OR_NULL  — not stored
  AMBIGUOUS     — can't tell (e.g., a value of 1.0 could be either
                  "1%" or "100%")

Known instances on Bishop: opexGrowthPct = 2.8, rentGrowthYr1 = 3.
Catalog every other instance on either deal.

Output a single table:

  | Deal | Field | Stored | Convention | Consumers |

Where Consumers = list of code paths that read this field and how
they interpret it (multiplied by 100? divided by 100? raw?).

This output feeds the future ASSUMPTION_UNIT_NORMALIZATION
migration; the audit catalogs but does not fix.

PHASE 0d — REGRESSION CHECKS

Targeted checks for known-issue patterns that should be CLOSED but

     in custom_opex_* on both deals (S1-01 closure check).
  2. Reserves Y1 propagation: confirm Y1 reserves on Projections

  3. Purchase price dual-write: confirm deals.budget and
     deals.deal_data.purchase_price match on both deals (#623

Each check returns PASS / FAIL / N/A with evidence.
