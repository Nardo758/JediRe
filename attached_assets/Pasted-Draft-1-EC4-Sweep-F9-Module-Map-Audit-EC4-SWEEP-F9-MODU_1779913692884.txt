Draft 2 — BUG-UTIL-01 Service-Layer Enforcement
BUG-UTIL-01 — SERVICE-LAYER ENFORCEMENT

CONTEXT
Per Batch 1 implementation verification, BUG-UTIL-01 is a latent 
double-count risk in proforma-adjustment.service.ts. The structural 
issue:

deal_assumptions.year1 can contain both combined utilities AND 
sub-line decomposition simultaneously:
  year1.utilities    = 187,094  (combined, from T12 parser)
  year1.water_sewer  =  78,579  (split ratio, from agent)
  year1.electric     =  59,870  (split ratio, from agent)
  year1.gas_fuel     =  33,677  (split ratio, from agent)

If proforma-adjustment.service.ts sums all expense fields without 
distinguishing combined-vs-sub-line, it double-counts utilities 
(~$172K overstated on the example above).

Current protection: Batch 1 implementation added an agent-prompt 
guard instructing the agent to suppress the combined utilities 
row when writing sub-lines. This protects write-side cases where 
the agent is the writer.

Current gap: Service-layer (read-side) is unprotected. Any other 
writer to deal_assumptions.year1 (manual operator input, future 
tooling, broker package import, data migration) could produce 
the double-count, and the service layer would sum it incorrectly.

This dispatch adds service-layer enforcement: detect the 
combined+sub-line conflict during summation, and apply the right 
behavior.

STATE VERIFICATION (per P8 corollary):
1. proforma-adjustment.service.ts exists at expected path
2. The current summation logic actually sums all expense fields 
   without filtering (verify by reading the code)
3. deal_assumptions.year1 schema supports both utilities AND 
   water_sewer/electric/gas_fuel fields (per types.ts:479-484)
4. No prior BUG-UTIL-01 fix has been applied

LESSON 1 — VERIFICATION CATCHES DATA-LEVEL ERRORS IN COMPLETED WORK
═══════════════════════════════════════════════════════════════════

Context: Tax Gap Remediation dispatch landed all 5 gaps as "fixed" 
with high confidence. The Build-mode bot's state verification 
confirmed each gap had a corresponding code/prompt change. The 
dispatch's testing section presumably passed.

The verification dispatch caught a numeric error: TN 
assessmentRatio was set to 0.25 (residential rate) instead of 
0.40 (commercial/multifamily rate). This was hidden inside a data 
addition that looked structurally complete.

Without verification, TN tax estimates would have produced values 
at 62.5% of correct indefinitely. Operators would have underwritten 
TN deals with materially lower tax burdens than reality. The error 
would not have surfaced until a TN deal closed and the actual 
property tax bill arrived months later.

Standing lesson: P8 verification doesn't just confirm document 
integrity. It catches data-level errors hidden inside implementations 
that look structurally correct. Spot-checks must include sample-level 
data validation, not just "the change was made."

For future verification dispatches involving data additions (new 
jurisdictions, new line items, new ratios, new defaults): always 
include at least one spot-check that confirms the numeric values 
against authoritative external reference, not just internal 
consistency.

LESSON 2 — STATE VERIFICATION CATCHES DISPATCH AUTHOR ASSUMPTIONS
═══════════════════════════════════════════════════════════════════

Context: The f9-financial-export.service.ts:172 cleanup dispatch 
was drafted with the assumption that the $350 hardcode was dead 
code. The reasoning: Batch 1 implementation wired the three-tier 
rule into the derivation path; therefore the export-path fallback 
would never fire.

The bot's state verification caught the assumption error: 
y1('replacement_reserves') can still return null for deals 
underwritten before Batch 1 was wired. The hardcode protects 
those legacy deals from null output during F9 export.

The correct fix was not removal but observability: retain the 
fallback, add console.warn so the fallback firing is visible in 
logs. Operators can then identify which legacy deals need a 
Batch 1 re-run.

Standing lesson: Dispatch authors (Claude, in this case) can be 
wrong about what's "dead code" or "safe to remove." State 
verification catches these assumption errors before they produce 
regressions. The corollary: dispatches that authorize removal of 
"dead" code should explicitly include "verify dead-ness" as the 
first state verification step, with a fallback plan if dead-ness 
isn't confirmed.

For future cleanup dispatches: always frame as "verify dead-ness 
THEN remove" rather than "remove" with verification as implicit 
precondition.

═══════════════════════════════════════════════════════════════════
META-OBSERVATION ON BOTH LESSONS
═══════════════════════════════════════════════════════════════════

Both lessons share a common thread: the bot's state verification 
discipline catches errors that look invisible from above. Dispatch 
authors operate on conceptual models of "what should be true" or 
"what's done." The bot operates on what's actually in the code 
right now.

When these diverge, the bot's view is authoritative. The dispatch 
gets corrected, not the codebase.

This is exactly the discipline P8's state verification corollary 
was added to produce. Both lessons confirm the corollary is doing 
its work.
