1. Flag the deals.updated_at proxy limitation explicitly. Step 5 says: "The caller computes these by comparing deal_assumptions.updated_at (seed time) to deals.updated_at (last deal edit)." Acceptable for Phase 1 but the proxy is coarse — deals.updated_at fires on any edit to the deal record, not specifically on changes to the source field. If an underwriter renames the deal at 14:00 but broker_claims.proforma.stabilizedGpr was last written at 13:43:25, the proxy says "source written at 14:00" when the relevant write was 30 minutes earlier.
For Bishop's specific case the proxy worked because the diagnostic reasoning didn't depend on field-level write times — contract services was null on both sides. But for cases where source IS populated and we need true field-level write semantics, this proxy will misclassify some fraction of findings.
Add to the task as an explicit known limitation with a forward note: "Phase 2 may introduce field-level write tracking via an extraction events audit table. Until then, false STALE_SEED classifications are possible when an unrelated deal edit happens after seeding. Misclassification rate should be monitored once Phase 1 has 4+ weeks of production data."
This is the load-bearing assumption that determines whether timestamp-based classification works at all. Worth surfacing rather than burying.
