TASK — FIX-1: Schedule TrafficCalibrationJob via Inngest (weekly)

Per docs/architecture/TRAFFIC_ENGINE_STATE_AUDIT.md §11 Fix #1
and findings TE-02 + TE-08. Activates the existing three-layer
Bayesian calibration hierarchy by giving it a weekly schedule.

CONTEXT

The TrafficCalibrationJob class in backend/src/jobs/
trafficCalibrationJob.ts is fully implemented (see audit §3 and §8)
with Baseline (cold-start priors), Platform (traffic_calibration_
factors Bayesian hierarchy), and Deal/Subject-First layers all
wired into trafficPredictionEngine. The job's run() method:

  1. Pulls lease events from rent roll snapshots (TTM window)
  2. Computes evidence per property
  3. Rolls up to scope buckets (msa × submarket × class × vintage_band)
  4. Applies Bayesian update: posterior = (prior × n_prior +
     evidence × n_evidence) / (n_prior + n_evidence)
  5. Writes versioned rows to traffic_calibration_factors
  6. Snapshots old rows to traffic_calibration_history
  7. Computes absorption benchmarks
  8. Publishes Kafka event traffic.calibration.updated

Today, this job has no automatic trigger — manual admin invocation
only. Result: platform posteriors are frozen at initial seed
values. All downstream M07 consumers (M09 ProForma, M25 JEDI,
M14 Risk) operate on stale coefficients indefinitely.

This fix adds a weekly Inngest cron on Monday 02:00 UTC. Weekly
(not nightly) because: (a) rent-roll snapshots arrive episodically,
not daily; (b) Bayesian posteriors stabilize over multiple weeks
of evidence, not single days; (c) weekly cadence reduces compute
load while still keeping calibration meaningfully fresh.
