TASK — FIX-3: Emit Asymmetric Confidence Bands

Per docs/architecture/TRAFFIC_ENGINE_STATE_AUDIT.md §11 Fix #3
and findings TE-01 + TE-12. Replaces the current 3-point ±1σ
confidence band with a 5-point asymmetric percentile band,
backed by per-evidence JSONB storage so M38 can recompute
arbitrary percentiles later without re-running calibration.

CONTEXT

Per audit §4.3, M07 currently emits THREE different confidence
representations, none of which is the spec's required
{p10, p25, median, p75, p90}:

  1. confidence (top-level): { score: number 0-100; tier:
     'High'|'Medium'|'Low'; breakdown }
  2. confidence_band: { low, mid, high } — ±1σ around posterior
  3. confidence_v2: { score, tier, data_weeks, per_metric }

The TRAFFIC_ENGINE_CALIBRATION_SPEC.md §1.5 requires asymmetric
percentile bands because the response distributions for lease
absorption and rent growth are not symmetric. The existing ±1σ
representation assumes symmetry and loses information.

This fix:
  1. Adds per-evidence values storage as JSONB array (so M38
     can recompute different percentiles later)
  2. Updates confidence_band shape to {low, p25, p50, median,
     p75, high} where p50 and median are aliases for the same
     value
  3. Wires the new shape through trafficToProFormaService and
     downstream consumers

CORE IMPLEMENTATION

backend/src/jobs/trafficCalibrationJob.ts lines 398-400:

REPLACE the existing ±1σ computation:

  // OLD (lines 398-400):
  const mean = evidenceValues.reduce((a,b) => a+b, 0) / n;
  const stdDev = Math.sqrt(evidenceValues.reduce(
    (acc, v) => acc + (v - mean)**2, 0) / n);
  const confidenceBand = {
    low: mean - stdDev,
    mid: mean,
    high: mean + stdDev,
  };

WITH percentile computation + per-evidence storage:

  // NEW:
  // Store raw per-evidence values
  const evidenceRecords = evidenceArray.map(e => ({
    property_id: e.propertyId,
    value: e.value,
    recorded_at: e.recordedAt.toISOString(),
  }));

  // Compute percentile band
  const sorted = [...evidenceValues].sort((a, b) => a - b);
  const percentile = (p: number) =>
    sorted[Math.min(Math.floor(sorted.length * p),
                    sorted.length - 1)];

  const median = percentile(0.5);
  const confidenceBand = {
    low: percentile(0.10),
    p25: percentile(0.25),
    p50: median,
    median: median,
    p75: percentile(0.75),
    high: percentile(0.90),
  };

  // Persist both
  const factorRow = {
    // ... existing fields ...
    confidence_band: confidenceBand,
    evidence_values: evidenceRecords,
  };

This preserves the per-evidence array for M38's eventual
percentile recomputation (FIX-3 Q1 option A) and emits the
asymmetric percentile band immediately.

backend/src/services/trafficPredictionEngine.ts:

read from the calibration factor's confidence_band (now
percentile-based) rather than computing ±1σ.

If legacy LegacyConfidenceBand is encountered (pre-FIX-3 row),
convert it on the fly: low → low, mid → median+p50, high → high,
p25/p75 derived by linear interpolation between low/mid and
mid/high respectively. This handles the migration period
where some rows have new shape and others have legacy.

BACKFILL OF EXISTING ROWS

backend/src/scripts/backfill-traffic-calibration-percentiles.ts (NEW):

For all existing traffic_calibration_factors rows where
evidence_values IS NULL:
  IF the original evidence is still available in
  traffic_calibration_history or related sources:
    Reconstruct evidence_values and recompute percentile band
  ELSE:
    Approximate band from existing {low, mid, high}:
      low → low, mid → p50 + median, high → high,
      p25 = low + 0.25 * (mid - low),
      p75 = mid + 0.25 * (high - mid)
    Flag as approximate via a new column.

  Reports: N rows fully reconstructed, M rows approximated.

This is a one-time script, not a continuous cron.

NOTES

  - This fix is more consequential than FIX-1 because it
    changes the shape of M07's emitted contract. Downstream
    consumers (Pro Forma, JEDI Score, M14 Risk) gain
    access to risk-adjusted bands that were impossible to
    emit before.
  - The per-evidence JSONB storage is the foundation for
    M38's eventual reliability work — when M38 ships, it
    reads evidence_values to compute per-stratum reliability
    against realized outcomes from the historical_observations
    corpus.
  - The legacy LegacyConfidenceBand conversion is intentional
    technical debt — it allows FIX-3 to ship without forcing
    a hard cutover. After 2-3 calibration cycles, all rows
    will have new shape and the legacy code path can be
    removed in a follow-up.
