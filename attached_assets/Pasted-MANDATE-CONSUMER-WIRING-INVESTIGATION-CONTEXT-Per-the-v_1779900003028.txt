MANDATE CONSUMER WIRING — INVESTIGATION

CONTEXT
Per the verification verdict, the Single-Value Mandate is fully 
lifted in v1.3. All 9 instances confirmed removed. regimeDataByField

This raises a real question: what happens when those sub-fields

This investigation maps all three surfaces and produces the 
implementation roadmap. No code changes. Output is a structured 
plan for what needs to ship to make the Mandate lift visibly 
useful to operators.

STEP 4 — GAP 2 (CONFIDENCE PROPAGATION) DESIGN
═══════════════════════════════════════════════════════════════════

The Mandate lift verification flagged Gap 2 as the most consequential 
remaining design question: when sub-fields have asymmetric confidence 
(pre 'high', post 'medium'), how does the parent LayeredValue's 
confidence surface?

This investigation should address Gap 2 because the answer affects 
RegimeExpand rendering AND agent self-validation AND F9 KPI 
displays. Without a decision, each consumer might handle it 
differently, producing UI inconsistency.

A. Document the three options for parent confidence:
   - Inherit lower of pre/post (conservative; surfaces uncertainty)
   - Inherit higher of pre/post (optimistic; rare in financial 
     modeling)
   - Independent (parent has its own confidence; sub-fields 
     are detail)
   - Weighted by strategy (value-add weights post; stabilized 
     weights pre)

B. Recommend an option with reasoning

C. Identify implementation impact:
   - Which files need to know the rule?
   - Where does the rule live (helper function, type definition, 
     per-consumer)?
