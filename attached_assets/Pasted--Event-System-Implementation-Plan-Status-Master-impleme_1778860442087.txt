## 0. Purpose and Scope

The platform tracks 16 event subtypes in `EVENT_SUBTYPE_REGISTRY`. Three independent Phase-0 audits — Event Propagation, Capital/Exit Subsystem, Traffic Engine State — converged on the same architectural debt: **events are tracked but not consumed.** The Event Propagation Audit's event×module matrix found 66 NOT_WIRED cells and 16 MISCLASSIFIED cells. The modules that are *declared* responsible for propagating each event subtype's effect largely don't query for those events.

This document is the master implementation plan for closing that debt and extending the system. It covers three workstreams:

- **WS-1 — Event-wiring consolidation.** The Event Propagation Audit's 7-fix sequence, the Capital/Exit audit's D4 findings, and the event-touching subset of the Traffic Engine fixes, merged into one consolidated backlog. They overlap at the file-and-line level; dispatching them separately would put multiple agents in the same functions.
- **WS-2 — Registry expansion.** Two new event subtypes (`development_moratorium`, `eviction_moratorium`) and one new cause metric (`business_formation_trend`).
- **WS-3 — Forward Supply.** A new M03 feature: a forward developable-supply projection per submarket, the supply-side sibling of what M07 does for demand.

**What this document is not:** a re-audit. The three Phase-0 audits stand. This plan reads them, consolidates them, and extends them.

**The discipline this plan enforces:** no code dispatch in WS-1, WS-2, or WS-3 begins until Phase 0 (the synthesis read-pass, §3) completes. The merge map and file-collision matrix that Phase 0 produces are what prevent the parallel-collision problem. This is non-negotiable and is the reason Phase 0 is Phase 0.

## 2. The Canonical Consumption Pattern

The Event Propagation Audit's Section 11 established the canonical event-consumption pattern. **Every wiring change in this plan conforms to it.** It is not re-litigated here; it is adopted as the standard.

```sql
-- The canonical pattern: direct SQL against key_events,
-- LEFT JOIN event_forecasts when a projected magnitude is needed.

SELECT ...
FROM key_events ke
LEFT JOIN event_forecasts ef
  ON ef.event_id = ke.id
  AND ef.status = 'active'
  AND ef.window_months = $windowMonths
WHERE ke.subtype = $subtype
  AND ke.status IN ('announced', 'in_progress', 'materialized')   -- per consumer need
  AND ST_DWithin(                                                  -- or: ke.msa_id = $msaId
    ST_SetSRID(ST_Point(ke.lng, ke.lat), 4326)::geography,
    ST_SetSRID(ST_Point($lng, $lat), 4326)::geography,
    $radiusMiles * 1609.34
  )

-- Column mapping (from EP Fix 1):
--   impact_type      ← ke.subtype
--   impact_magnitude ← ef.point_estimate
--   confidence       ← ef.confidence
```

**Rules:**

- `key_events` is the **event of record** for all consumption — filtered by `subtype`, `status`, and a geospatial or `msa_id` predicate.
- `event_forecasts` is **joined on** when the consumer needs the forecast magnitude / confidence of a *projected* impact, keyed by `window_months`.
- `demand_events` is the **wrong (legacy) table.** EP Fix 1 is explicitly "repoint *off* `demand_events`." Anything still reading it is on dead schema and fails silently when the legacy table is empty.
- The `event.classified` Kafka topic is **not** in the canonical consumption path. No Section 11 fix reads from it. (It may exist for other purposes; consumers do not subscribe to it.)

Conformance to this pattern is an acceptance criterion for every wiring fix in WS-1, WS-2, and WS-3.

## 5. WS-2 — Registry Expansion

Three additions to `causal-discipline-engine.ts`. **These do not get committed ahead of Phase 0.** They land as part of their consumer-wiring W-items, so each new signal arrives with a consumer attached — the Event Propagation Audit's central lesson is that tracked-but-not-consumed is the dominant failure mode, and adding signals without consumers makes the matrix worse.

### 5.1 `development_moratorium` — new event subtype

```typescript
{ eventSubtype: 'development_moratorium', primaryChannel: 'M03_devcap',
  rationale: 'Halts new permitting; removes future developable supply for the
              moratorium duration. The negative-direction sibling of
              zoning_upzoning / entitlement_approval — M03_devcap handles
              development-capacity changes in either direction.' }
```

- **Channel:** M03_devcap (single channel).
- **Lands with:** W6 (M03 devcap wiring). The M03 consumer that EP Fix 6 builds must read `development_moratorium` alongside `entitlement_approval` / `zoning_upzoning` / `multifamily_permit`.
- **Downstream:** a direct input to WS-3 Layer 3 — a development moratorium in the radius is a hard cap on forward supply for its duration.

### 5.2 `eviction_moratorium` — new event subtype

```typescript
{ eventSubtype: 'eviction_moratorium', primaryChannel: 'M09_proforma_direct',
  rationale: 'Discontinuous policy constraint on collections and turnover;
              a direct proforma path change, not market-clearing-mediated.
              Sibling to rent_control_passage.' }
```

- **Channel:** M09_proforma_direct (single channel).
- **Lands with:** W4 (M09 proforma mutation). EP Fix 4 wires `rent_control_passage` and `tax_abatement`; `eviction_moratorium` is the same wiring pattern in the same file.
- **Note:** `eviction_moratorium` and `rent_control_passage` are both tenant-protection policies. A future refactor could consolidate them under a `tenant_protection_policy` subtype with variants. Not blocking — two clean subtypes for now; revisit only if the data sources make the distinction noisy.

### 5.3 `business_formation_trend` — new cause metric (not an event subtype)

```typescript
{ metricId: 'business_formation_trend', name: 'Business Formation Trend',
  category: 'demand_cause' },
```

- **Why a cause metric, not an event:** business formation data (Census Business Formation Statistics, published weekly/monthly by state and MSA) is *continuous* — a number that moves over time — not a discrete thing-that-happened-on-a-date. It belongs with `wage_growth_trend` and `in_migration_trend` in `CAUSE_METRICS`, not in `EVENT_SUBTYPE_REGISTRY`.
- **Causal logic:** new business formations are a *leading indicator* of employment growth. They precede `employer_expansion` rather than duplicating it — formations signal future demand-pool growth before it shows up as realized headcount.
- **Channel implication:** feeds M07_traffic the same way `in_migration_trend` does — a demand-pool driver.
- **Lands with:** the M07 demand-cause consumption path. Phase 0 places it; it is lighter than the subtype additions because it extends an existing metric category rather than adding a new event to the matrix.

### 5.4 Explicitly NOT added — demand surge

Demand surge is **not** a registry addition. It is already present as `C_SURGE_INDEX` in `SYMPTOM_METRICS` — and it is on the correct side of the cause/symptom line. Demand surge is the *observable downstream consequence* of demand events (`employer_expansion`, `in_migration_driver`, `business_formation_trend`), not an independent cause. Adding it as an event would create a causal loop: the thing that is supposed to be the *measured output* of demand events would also become an *input*, enabling double-counting.

If demand surge should *drive* a model effect, that is a wiring item — wire the existing `C_SURGE_INDEX` symptom into a consumer — not a new event. If it should *trigger an alert or re-underwrite*, that is an M22 / monitoring concern (the Monitor tab, Capital/Exit CE-05 territory). **Open decision (§9):** which of these is intended.

## 6. WS-3 — Forward Supply Feature

A new M03 feature: a forward developable-supply projection per submarket. The supply-side sibling of what M07's traffic engine does for demand.

### 6.1 The formula

```
forward_supply(submarket, radius) =
    static_capacity_under_current_zoning
  + (probable_rezone_capacity × trend_weight)

where:

  static_capacity_under_current_zoning =
    Σ over swept undeveloped/underbuilt MF-zoned parcels of
      ( parcel_acreage × binding_units_per_acre )

  binding_units_per_acre =
    the zoning module's feasibility-resolved constraint
    (setbacks / FAR / parking minimums / height / lot coverage),
    NOT the nominal zoning designation number

  probable_rezone_capacity =
    Σ over swept non-MF parcels of theoretical MF capacity
    if rezoned

  trend_weight =
    a function of M03_devcap event density in the submarket
    (zoning_upzoning + entitlement_approval event history,
     net of development_moratorium)
```

### 6.2 The three layers

The feature decomposes into three layers. Only Layer 3 is genuinely new modeling.

**Layer 1 — Radius parcel sweep.** A GIS query: every parcel zoned for multifamily within a 3-mile and 5-mile radius of the deal, with geometry, acreage, and current improvements. This is mostly *existing data access* — the platform has Florida county GIS integrations (3 verified) and ArcGIS GeoEnrichment. If the existing zoning module pulls only deal-parcel zoning rather than parcel-level zoning across a radius, the sweep is a small extension, not a build.

**Layer 2 — Batch feasibility capacity.** For each swept parcel, run the zoning module's *existing* feasibility-resolution logic — the M02→M03 handoff that already resolves setbacks, FAR, parking minimums, and height into a binding units-per-acre for the *subject* parcel. The Forward Supply feature applies that same logic to *every parcel in the radius*. **This is the piggyback** — vectorize the existing zoning-module logic over the radius sweep; do not reimplement zoning interpretation. The userMemory records the governing principle: *zoning constraints must be binding gates, not advisory — parking is frequently the actual binding feasibility constraint.* Layer 2 also applies the **undeveloped/underbuilt filter**: distinguishing a vacant lot from a parcel with an aging 12-unit building on land zoned for 60 units. The latter is *underbuilt* — latent supply — and arguably matters more than raw vacant land. This is a current-use-vs-allowed-use delta, joining current-improvement data (ATTOM, county assessor records) to the zoning capacity.

**Layer 3 — Rezone/upzone trend weighting.** The genuinely new probabilistic model. Layers 1 and 2 produce a *static* picture: theoretical capacity under *current* zoning. But the feature's goal is *future* supply — which includes parcels not multifamily-zoned today but sitting where the rezone/upzone trend says they likely will be. Layer 3 consumes the M03_devcap event stream — `zoning_upzoning`, `entitlement_approval`, and `development_moratorium` events accumulated per submarket *are* the rezone/upzone trend — and weights the static capacity by the probability that currently-non-MF parcels get rezoned.

### 6.3 Build status and gating

| Layer | What it is | Build status | Gate |
|---|---|---|---|
| **1 — Radius parcel sweep** | GIS query for MF-zoned parcels in 3mi/5mi rings | Mostly existing data access; small extension if zoning module is deal-parcel-only | **Buildable now** |
| **2 — Batch feasibility capacity** | Vectorize existing M02/M03 feasibility logic over the sweep; undeveloped/underbuilt filter | Piggyback — no new feasibility engine | **Buildable now** (event-independent) |
| **3 — Rezone/upzone trend weighting** | Probabilistic model consuming M03_devcap events | Genuinely new modeling | **Gated on WS-1 W6 (EP Fix 6)** — Layer 3 is the M03 event channel's most important consumer |

**Layers 1 and 2 are buildable now** — GIS plus existing feasibility logic, no event dependency. The static forward-supply-capacity picture is genuinely useful on its own: "here is the theoretical multifamily capacity in the radius under current zoning" is a real answer without the trend layer.

**Layer 3 is gated on W6.** It is the consumer of the M03_devcap events that EP Fix 6 wires. You cannot wire a consumer to events that are not flowing. And **Layer 3's trend-model calibration is corpus-gated (Phase B)** — how much a `zoning_upzoning` event density predicts an actual rezone is an empirical question the `historical_observations` corpus answers, same as every other magnitude question on the platform.

### 6.4 Required new code

**Backend:**
- `backend/src/services/forward-supply/radius-sweep.service.ts` — Layer 1: the GIS query for MF-zoned parcels in 3mi/5mi rings
- `backend/src/services/forward-supply/batch-feasibility.service.ts` — Layer 2: vectorized call into the existing zoning-module feasibility resolution, plus the undeveloped/underbuilt filter
- `backend/src/services/forward-supply/rezone-trend.service.ts` — Layer 3: the probabilistic trend model (gated on W6; calibration corpus-gated)
- `backend/src/services/forward-supply/forward-supply.service.ts` — the orchestrator composing the three layers into the `forward_supply` projection
- Edits to the zoning module to expose its feasibility-resolution logic as a callable unit if it is currently inlined into the subject-parcel path
- `backend/src/api/rest/forward-supply.routes.ts` — the endpoint

**Frontend:**
- A Forward Supply panel — likely on the M03 / development surface, showing the 3mi and 5mi ring projections side by side, with the static-vs-trend-weighted split visible (so the user can see Layer 1+2 capacity vs. the Layer 3 probabilistic addition)

### 6.5 Open spec decision — fixed rings vs. adaptive shed

The "3 and 5-mile radius" — fixed choice or parameter? M07's traffic engine uses an *adaptive* commute-shed rather than a fixed radius. For a dense urban submarket, 3mi might capture the relevant competitive supply shed; for a Florida exurban deal, 5mi might be too tight.

**Recommendation:** fixed 3mi and 5mi rings for v1 — two standard rings the user can compare, simple and predictable. Adaptive shed is a later refinement. But this is a real spec decision (§9).

1. **Demand surge — drive or flag?** `C_SURGE_INDEX` already exists as a symptom metric. Does it need to *drive* a model effect (a wiring item — wire the existing symptom into a consumer) or *trigger an alert / re-underwrite* (an M22 monitoring item)? This determines whether it is a WS-1 wiring line or a monitoring-track item. It is **not** a registry addition either way.

2. **Forward Supply — fixed rings or adaptive shed?** Recommendation is fixed 3mi/5mi rings for v1, adaptive shed as a later refinement. Needs confirmation before WS-3 Layer 1 specs the sweep query.

3. **`eviction_moratorium` — own subtype or `tenant_protection_policy` variant?** Recommendation is two clean subtypes (`eviction_moratorium`, `rent_control_passage`) now, with a possible future consolidation under `tenant_protection_policy`. Confirm, or decide to do the consolidation up front.

4. **Speculative macro subtypes — hold or add?** `credit_spread_move` and `capital_flow_shift` were raised as possible M14 additions (the M14 channel is thin relative to its importance for the Capital/Exit and Layer 2 work). Recommendation: **hold** — do not add a macro subtype without a confirmed data source and a consumer that will read it. The Event Propagation Audit's lesson is that tracked-but-unconsumed is the dominant failure mode. Revisit when there is a concrete data source.

5. **`transit_infrastructure` subtype — add?** A new rail station / highway interchange / transit line changes demand geography. The platform has `transportation_infra` as a *cause metric* but no discrete event for "transit project announced/opened." Worth a subtype if the data sources carry it; same hold-until-there-is-a-source discipline as #4.

6. **WS-3 Layer 2 — is the zoning module's feasibility logic already callable?** If the M02/M03 feasibility resolution is inlined into the subject-parcel path rather than exposed as a callable unit, Layer 2 needs a small refactor to extract it before it can be vectorized. Phase 0 or an early WS-3 scoping step should confirm.

## 10. Summary

The platform tracks 16 event subtypes; three audits found that most are not consumed. This plan consolidates the event-wiring debt into one backlog (WS-1), extends the registry with two moratorium subtypes and one business-formation cause metric (WS-2), and specs the Forward Supply feature (WS-3) — the supply-side sibling of the M07 demand model.

The discipline that holds it together: **Phase 0 (the synthesis read-pass) runs first, read-only, and produces the merge map that prevents parallel-collision.** Nothing in WS-1/WS-2/WS-3 dispatches as code until Phase 0 completes. The registry additions land *with* their consumer wiring, never ahead of it. Forward Supply Layers 1+2 are buildable now; Layer 3 is gated on the M03 event wiring (W6); all Phase B magnitude calibration is gated on the `historical_observations` corpus.

The critical-path item not on the implementation track: **D2's PR must land.** It removes the hardcoded exit-cap trajectory that EP-03, FA-02, and EP Fix 2 all depend on. It is done work sitting in a branch, and it gates W3 and part of W7.
