## Ground Rules (Non-Negotiable)

These are architectural constraints. Do not violate them without explicit written approval:

1. **Agents are service-account users, not privileged services.** Every agent has a row in the `users` table with `user_type='agent'` and authenticates through the same auth middleware as human users. No god-mode DB access.

2. **Dogfooding.** Agents write through the platform API via `platformClient.ts` — the same client the frontend uses. The only documented exception is DealContext assembly writes, which may go direct-to-service. Do not add other exceptions without written approval.

3. **One source of truth.** Agent outputs land in existing domain tables (parcels, zoning, comps, deal_financials, etc.) wrapped in `LayeredValue<T>` with `source: 'agent:*'`. Do not create parallel "agent_outputs" tables or shadow storage. The only agent-specific tables are: `agent_runs`, `agent_run_steps`, `prompt_versions`, and one new table `market_commentary` for the Commentary Agent.

4. **Five agents only.** Research, Zoning, Supply, CashFlow, Commentary. Do not create new agents. The 10 routing specialists are intent labels and dispatch to either one of the 5 agents or to a general LLM handler. The 16 analyst personas are system prompt variants on the Coordinator — not agents.

5. **Budget caps are mandatory.** Every agent run has per-run, per-deal-per-day, and per-user-per-month cost ceilings enforced at the metering layer before any Claude call fires. Runs that exceed caps abort with status `budget_exceeded`.

6. **All writes are audited.** Every API write (human or agent) flows through audit middleware that inserts into the unified `audit_log` table. Agent writes have `actor_type='agent'` and a non-null `agent_run_id`.

7. **Prompts are versioned and stored in DB, not inline in code.** The `prompt_versions` table holds system prompts and output schemas. Code loads the active version at runtime.

8. **If a task can be written as deterministic `if X then Y` logic, it is NOT an agent.** It is a service, cron job, DB trigger, or Inngest function. Claude-in-a-loop is reserved for reasoning under uncertainty with tool use. Per the task inventory, 56% of tasks are deterministic code — do not overbuild those as agents.

9. **Web search is a fallback, not a primary data path.** Only Research and Commentary agents get the `web_search` tool. Zoning may optionally get it with strict gov-only allowlist. Supply and CashFlow do NOT get web search. System prompts must instruct agents to prefer structured data tools first. Every search-derived fact requires a citation in the output.

10. **Nothing is hardcoded to Florida outside the FL ruleset files.** Tax math, insurance mechanics, DOT data sources, regulator monitoring — all use jurisdiction-aware pluggable rulesets. If you find yourself writing `if (state === 'FL')` outside a ruleset file, stop and architect it as a ruleset instead. Florida is the first market, not the only market.

## Build Order — Mandatory Sequencing

Execute phases in order. Do not start a phase until the previous phase is fully merged, tested, and reviewed. Each phase ends with a review checkpoint.

### Phase 1 — Foundation (Infrastructure Only, No Agent Prompts Yet)

**Goal:** All infrastructure for agents to exist and authenticate, without any agents actually running yet. This phase is pure plumbing. No LLM calls should be made in this phase.

**Deliverables:**

1. DB migrations (in this exact order):
   - `users.user_type` column + `agent_metadata` jsonb column + check constraint
   - Seed migration inserting 5 agent user rows: `agent_research`, `agent_zoning`, `agent_supply`, `agent_cashflow`, `agent_commentary` (idempotent via ON CONFLICT)
   - `agent_runs` table with all indexes specified in spec
   - `agent_run_steps` table with FK cascade to agent_runs
   - `prompt_versions` table with partial unique index on `(agent_id) WHERE active = true`
   - `audit_log` table with three indexes specified in spec
   - `market_commentary` table with lookup index

2. Extend `LayeredValue<T>` type in `src/types/layered-value.ts`:
   - Add 5 new source values: `agent:research`, `agent:zoning`, `agent:supply`, `agent:cashflow`, `agent:commentary`
   - Add optional `agent_run_id?: string` field
   - Update merge-order logic: `platform` → `agent:*` → `t12/rent_roll/tax_bill` → `override`

3. Auth middleware extension:
   - New internal endpoint `POST /api/auth/agent-token` — issues scoped JWTs for agent service accounts (not publicly exposed, requires internal service credential)
   - `authMiddleware` enhancement to decode `user_type` from token
   - New `requireCapability(capability: string)` middleware factory
   - Token TTL of 1 hour with auto-refresh; capability list from `agent_metadata`

4. `SourceBadge` frontend component in `frontend/src/components/primitives/SourceBadge.tsx`:
   - Reads `source` from LayeredValue wrapper
   - Renders small badge with tooltip (agent name + timestamp when applicable)
   - Add `T.accent.agent` to the design token system (match existing T-token conventions — Bloomberg dark, max 2px radius, no shadows)

**Phase 1 Acceptance Criteria:**
- All 7 migrations run clean up and down
- Seed migration creates exactly 5 agent rows; rerunning is idempotent
- `POST /api/auth/agent-token` returns a valid JWT for a known agent ID; rejects unknown agent IDs
- A dummy endpoint protected by `requireCapability('read:parcels')` accepts a token for `agent_research` (which has the capability) and rejects a token for `agent_zoning` (which doesn't)
- LayeredValue merge order is unit-tested: `platform < agent:* < t12 < override`
- `SourceBadge` renders correctly for all 9 source values in Storybook or equivalent

### Phase 2 — Runtime (Still No Agent Prompts)

**Goal:** The `AgentRuntime` class and all supporting primitives exist and are unit-tested. Still no actual agent prompts or end-to-end runs — this phase proves the runtime works with a test fixture.

**Deliverables:**

1. `MeteringAdapter` class in `src/lib/metering/MeteringAdapter.ts`:
   - Wraps existing `@stripe/token-meter` + `@anthropic-ai/sdk` integration
   - Accepts `MeteringMetadata` object on every call (actor_type, actor_id, agent_run_id, deal_id, user_id, triggered_by)
   - Routes cost to correct bucket per three-bucket rule:
     - `triggered_by: 'user'` → debits user's credit balance via existing `creditService`
     - `triggered_by: 'event' | 'cron'` → platform expense, not charged to user
   - Fails fast if user has insufficient credits (pre-flight reserve)

2. `BudgetEnforcer` class in `src/agents/runtime/BudgetEnforcer.ts`:
   - Per-deal daily cap check via `agent_runs` aggregation
   - Per-run cap check called between each tool-calling iteration
   - Throws typed `BudgetExceededError` that the runtime catches and marks run as `budget_exceeded`
   - Default caps configured in `src/agents/config/budget.ts` per the table in the spec

3. `ToolDefinition` interface and `ToolRegistry` pattern in `src/agents/runtime/`:
   - Zod-validated input and output schemas on every tool
   - `requiresCapability` field enforced before tool execution
   - Helper function `toAnthropicToolSchema(tool)` that converts to the SDK's tool format

4. `AgentRuntime` class in `src/agents/runtime/AgentRuntime.ts`:
   - Constructor signature per spec
   - `run(input, ctx)` method implementing the full loop:
     - Pre-flight budget check
     - Create `agent_runs` row
     - Load active prompt from `prompt_versions`
     - Tool-calling loop (up to `maxStepsPerRun` iterations)
     - Per-step budget check
     - Tool input/output validation against Zod schemas
     - Log every step to `agent_run_steps`
     - Final output schema validation
     - Mark run complete (succeeded / failed / budget_exceeded)
     - Return validated output or throw typed error

5. OpenTelemetry instrumentation:
   - Each agent run is a trace; `agent_run_id` is the correlation key
   - Each tool call is a span
   - Standard attributes: `agent.id`, `agent.version`, `deal.id`, `triggered_by`, `agent.cost_usd`
   - Exporter configured for project's existing observability backend (or OSS default if none yet)

6. `platformClient.ts` extension:
   - New method `platformClient.as({ agentId, runId })` returns a client scoped to that agent's service-account token
   - Attaches correlation headers for tracing

7. One test-fixture tool (`echo_tool`) + test-fixture agent (`echo_agent`) for runtime validation:
   - `echo_tool` takes a string, returns the string uppercased
   - `echo_agent` uses `echo_tool`, has a trivial system prompt ("uppercase the input"), writes no side effects
   - Full end-to-end test: create run → loop executes → tool validates → output validates → run marked succeeded → all steps logged → budget enforced

**Phase 2 Acceptance Criteria:**
- `echo_agent` successfully runs end-to-end against live Claude API with full metering attribution
- Budget cap test: setting `maxCostUsdPerRun: 0.001` causes the run to abort with `budget_exceeded` status
- Tool input validation test: passing invalid input to `echo_tool` causes the loop to feed the Zod error back to Claude for self-correction (not crash)
- OTel traces appear in the observability backend with full parent-child span relationships
- Credits test: user-triggered `echo_agent` run debits user's credit balance; event-triggered run does not
- Run records in `agent_runs` and `agent_run_steps` are complete and queryable

**STOP. Review checkpoint with Leon before proceeding to Phase 3.**

### Phase 3 — First Real Agent (Research Agent)

**Goal:** Research Agent is the pattern every other agent follows. Get it right before replicating.

**Deliverables:**

1. Research Agent tools (in `src/agents/tools/`):
   - `fetch_parcel` — lookup by parcel_id OR address+county
   - `fetch_costar_metrics` — submarket metric time-series fetch
   - `fetch_county_records` — ownership history + tax records
   - `fetch_fred_indicators` — FRED macro series
   - `fetch_bls_employment` — QCEW data by county/MSA
   - `fetch_rentcast_comps` — rent comp pull
   - `fetch_google_places_reviews` — reviews + NLP sentiment
   - `fetch_fdot_traffic` — traffic count data (note: Florida-specific tool; add parallel tools for other states' DOT sources post-launch)
   - `fetch_municode_zoning` — zoning code text (shared with Zoning Agent)
   - `web_search` — Tavily-backed fallback search (scoped per `AGENT_SEARCH_CONFIG` in inventory Category 19). System prompt must instruct: "Use structured data tools first. Use web_search only when structured tools cannot answer."
   - `fetch_webpage` — deep read of a specific URL returned by search
   - `write_dealcontext` — the ONE exception: goes direct-to-service, not through API (document this in the tool file)

   Every tool has: Zod input + output schemas, `requiresCapability` field, unit tests with mocked external APIs, integration tests against real APIs (opt-in, expensive to run).

   **Web search specifics:**
   - Per-run search cap of 10 (enforced by `BudgetEnforcer` as a new cap type)
   - 24h result cache keyed on normalized query string
   - Domain blocklist: `*.pinterest.com`, `*.quora.com`, `*.reddit.com` for Research Agent
   - Output schema requires citation fields (source URL + retrieval timestamp) when search results influenced any returned value
   - `LayeredValue<T>` fields derived from web search get metadata flag `derived_from_search: true`

2. Research Agent system prompt in `src/agents/prompts/research/system.ts` + output schema in `src/agents/prompts/research/output-schema.ts`. Seed both into `prompt_versions` as the active version for `agent_id='research'`.

3. Research Agent wiring in `src/agents/research.agent.ts`:

   - Exports a singleton

4. Inngest function in `src/inngest/functions/research.function.ts`:

   - Tier-gated (Principal+ for auto-trigger; Operator gets manual-only)
   - Calls `researchAgent.run(...)` inside `step.run(...)` for durability
   - Publishes `research.completed` event on success

5. API endpoints (all follow existing route conventions):
   - `POST /api/agents/research/run` — user-initiated trigger
   - `GET /api/agents/runs/:runId` — run status + output
   - `GET /api/agents/runs/:runId/steps` — detailed step trace

6. Frontend integration:
   - Deal activity feed reads from `audit_log`; agent activities show with a distinct "AGENT" actor badge

   - SourceBadge component now actually appears on DealContext fields flowing into the UI

**Phase 3 Acceptance Criteria:**

## What NOT to Do

- Do not create agents beyond the 5 specified. If you think one is needed, write up the case and get explicit approval before coding.
- Do not create parallel storage for agent outputs. Everything lands in existing domain tables via LayeredValue.
- Do not inline system prompts as string literals in agent code. Use `prompt_versions`.
- Do not bypass the platform API for writes (except the one documented DealContext exception).
- Do not skip output schema validation. Unvalidated output will corrupt downstream caches.
- Do not implement agent-to-agent direct function calls. Agents communicate via Kafka/Inngest events only.
- Do not skip budget caps. Every agent has caps. Every run enforces them.
- Do not expand the "DealContext direct-write" exception to other tools without written approval.
- Do not build MetricRecommendation or AgentOrchestrator as agents — they are a service and the runtime layer respectively (see taxonomy addendum).
- Do not implement anything from the 113-task inventory until Phase 5 foundation is approved.
- Do not give web search to Supply or CashFlow agents. They operate on structured data only.
- Do not let agents use web search as a primary research path. It is a fallback. If a structured data tool (FRED, CoStar, RentCast, etc.) covers the question, use that instead. Enforce in system prompts.
- Do not hardcode Florida-specific logic outside ruleset files. If you're writing `if (state === 'FL')` or `// TODO: handle other states later`, you are building the wrong abstraction. Use the ruleset pattern from day one.
- Do not let search results get written to the DB without citations. Every field derived from web search must carry source URL + retrieval date in its LayeredValue metadata.
