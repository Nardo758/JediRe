THE CANONICAL CONSUMPTION PATTERN — already established

The Event Propagation Audit's Section 11 establishes the
canonical event-consumption pattern, and the synthesis must
treat it as the standard all wiring conforms to:

  Direct SQL query against key_events (LEFT JOIN
  event_forecasts when a projected magnitude/confidence is
  needed), filtered by subtype, status, and a geospatial or
  msa_id predicate.

  - key_events is the event-of-record for ALL consumption.
  - event_forecasts is joined on when the consumer needs the
    forecast magnitude (ef.point_estimate / ef.confidence),
    keyed by window_months.
  - demand_events is the WRONG (legacy) table — Fix 1 is
    "repoint OFF demand_events." Anything still reading it is
    on dead schema.
  - The event.classified Kafka topic is NOT in the canonical
    consumption path — no Section 11 fix reads from it.

The synthesis does not re-litigate this. It adopts it as the
standard and checks every fix and every Capital/Exit finding
for conformance.
