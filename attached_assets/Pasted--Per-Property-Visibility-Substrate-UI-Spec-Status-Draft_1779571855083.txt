**Status:** Draft architecture spec — implementation contract for sequenced build sessions
**Owner:** Leon / JEDI RE
**Pairs with:** `HISTORICAL_OBSERVATIONS_SPEC.md`, `M09_PROFORMA_SPEC.md`, `DEAL_CAPSULE_ROADMAP.md`
**Related:** Comp ProForma Reconstruction (downstream consumer), F9 source pills (parallel surface)

## 1. Purpose

The platform has 1,034 corpus rows ingested across 296 archived properties and ~40 OM PDFs parsed for year_built. The data substrate exists. **What's missing is the user-facing surface that makes it visible, manageable, and usable — and the plumbing that turns inbound property data into complete profiles.** Five independent product needs converge on the same architecture:

1. **Property Description fill** — agents aren't populating the property description modal because no agent is responsible for the archive properties; the rich extraction from parse-om isn't routed to the modal's read path.
2. **Data Library file visibility** — users can't see the source files agents are reasoning against. Source pills on F9 are designed to drill to a source but there's no surface to drill to.
3. **Asset profile time-series** — users need to see trends per asset, not just point-in-time facts.
4. **Active intake pipeline** — uploads shouldn't end at storage. They should trigger automatic property identification, classification, and enrichment. When the agent can't figure it out from documents + municipal records + web research, it should prompt the user for the minimum identifier needed to unblock.
5. **Apartment Locator API enrichment** — the platform's scraped rental data feed is dumping property records daily but has no plumbing to build profiles on them. Same enrichment pipeline as Data Library uploads applies; just a different upstream source.

These five are not separate features. They share the **same substrate**: property attributes (Stream A), source documents (Stream B), time-keyed observations (Stream C), and **one Intake Orchestrator** that processes inbound property data regardless of upstream source.

**Architectural principle: one enrichment pipe, many upstream sources.** The Apartment Locator scrape, Data Library uploads, parse-om endpoint, and future Pipeline-deal pageloads all funnel into the same orchestrator. Per-source ingest adapters normalize input shape; the orchestrator does the universal work of identify → resolve → enrich → resolve-or-block.

**Strategic principle: Atlanta-first as municipality blueprint.** Rather than build speculative infrastructure for all markets simultaneously, this spec ships the complete enrichment pipeline for Atlanta (Fulton, DeKalb, Cobb, Gwinnett counties), validates profile quality end-to-end, then documents a per-metro expansion template. FL/TX/AZ counties are not blockers to shipping; they're each ~1-2 sessions of wrapper-and-test work against the proven blueprint.

This spec defines that substrate, the streams, the orchestrator, the Atlanta blueprint, and the user surfaces.

Every displayed number on the per-property page has a source pill. The pill resolves through the streams to either a property attribute (with `LayeredValue` provenance), a source document (linked file), or a time-series observation (with the underlying parser run).

**The Intake Orchestrator is the active intelligence layer, source-agnostic.** Whether a property arrives from the daily Apartment Locator scrape, a user uploading documents, the parse-om endpoint, or a future Pipeline-deal pageload, it enters the same state machine: classify → parse → enrich → resolve-or-block. When the agent can't resolve a property identifier, it surfaces a specific contextual question in the Intake Inbox.

**Source adapters** normalize upstream input into a common `IntakeJob` payload. Each adapter is small (~50-150 LOC) and stateless. Adding a new source = adding an adapter, not modifying the orchestrator.

## 4. Shared Substrate

### 4.1 Property Descriptions Table

`property_descriptions` — one row per parcel_id. Each mutable attribute is a `LayeredValue` with source provenance.

```sql
CREATE TABLE property_descriptions (
  parcel_id              text PRIMARY KEY,
  -- Identity
  property_name          jsonb,  -- LayeredValue<string>
  address                jsonb,  -- LayeredValue<{ street, city, state, zip }>
  msa                    jsonb,  -- LayeredValue<string>
  county                 jsonb,  -- LayeredValue<string>
  -- Physical
  year_built             jsonb,  -- LayeredValue<number>
  year_renovated         jsonb,  -- LayeredValue<number>
  unit_count             jsonb,  -- LayeredValue<number>
  building_count         jsonb,  -- LayeredValue<number>
  stories                jsonb,  -- LayeredValue<number>
  stories_band           jsonb,  -- LayeredValue<'walkup'|'midrise'|'highrise'|'super_highrise'>
  total_sqft             jsonb,  -- LayeredValue<number>
  rentable_sqft          jsonb,  -- LayeredValue<number>
  lot_size_acres         jsonb,  -- LayeredValue<number>
  construction_type      jsonb,  -- LayeredValue<'wood_frame'|'concrete'|'steel'|'mixed'>
  parking_type           jsonb,  -- LayeredValue<'surface'|'structured'|'wrap'|'podium'>
  parking_spaces         jsonb,
  parking_ratio          jsonb,
  -- Asset class
  asset_class            jsonb,  -- LayeredValue<'A'|'B'|'C'>
  property_type          jsonb,  -- LayeredValue<'multifamily'|'office'|'retail'|...>
  -- Amenities (boolean flags)
  has_pool               jsonb,  -- LayeredValue<boolean>
  has_fitness            jsonb,
  has_clubhouse          jsonb,
  has_concierge          jsonb,
  has_business_center    jsonb,
  has_dog_park           jsonb,
  is_master_metered      jsonb,
  is_individual_metered  jsonb,
  -- Regulatory
  zoning_code            jsonb,  -- LayeredValue<string>
  flood_zone             jsonb,  -- LayeredValue<string> (FEMA NFHL)
  in_opportunity_zone    jsonb,  -- LayeredValue<boolean>
  -- Description text
  narrative              jsonb,  -- LayeredValue<string> (Research Agent generated)
  -- Metadata
  created_at             timestamptz DEFAULT now(),
  updated_at             timestamptz DEFAULT now()
);
```

**LayeredValue schema per field:**

```typescript
type LayeredValue<T> = {
  resolved: T | null;           // The final value the UI displays
  layers: {
    om?:        { value: T; source_file_id: uuid; confidence: number; extracted_at: timestamp };
    municipal?: { value: T; source: string; fetched_at: timestamp; api_endpoint: string };
    web?:       { value: T; source_url: string; fetched_at: timestamp };
    manual?:    { value: T; user: string; entered_at: timestamp; note?: string };
  };
  resolution_rule: 'manual_override' | 'municipal_canonical' | 'om_canonical' | 'highest_confidence';
};
```

**Resolution rules** (which layer wins when sources disagree):

| Field class | Default winner | Rationale |
|---|---|---|
| Legal attributes (year_built, sqft, parcel #, zoning) | Municipal canonical | County appraiser is authoritative |
| Asset class, narrative, amenity tier | OM canonical | Marketing context not in public records |
| Current state (photos, reviews) | Web | Most recent |
| All fields | Manual override (when present) | User explicit input always wins |

### 4.2 Data Library Files Table

`data_library_files` — one row per ingested file.

```sql
CREATE TABLE data_library_files (
  id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  -- Linkage
  parcel_id           text,
  deal_id             uuid,  -- NULL for archive-only files
  -- File identity
  original_filename   text NOT NULL,
  sha256              text NOT NULL UNIQUE,
  mime_type           text,
  size_bytes          bigint,
  -- Storage
  storage_provider    text NOT NULL,  -- 'r2' | 's3' | 'replit' | 'local_pending'
  storage_bucket      text,
  storage_key         text,           -- path within bucket
  cdn_url             text,           -- public signed URL or CDN endpoint, nullable
  -- Classification + parsing
  document_type       text,           -- T12 | RENT_ROLL | TAX_BILL | LEASING_STATS | OM | OTHER
  parser_used         text,
  parser_version      text,
  parser_status       text,           -- 'success' | 'partial' | 'failed' | 'unparsed'
  parser_run_id       uuid,
  parser_error        text,
  -- Provenance
  uploaded_at         timestamptz DEFAULT now(),
  uploaded_by         text,
  source_signal       text,           -- 'archive_seed' | 'pipeline_upload' | 'portfolio_monthly'
  -- License
  license_restricted  boolean DEFAULT false,
  license_source      text,           -- 'costar' | 'yardi_matrix' | etc.
  -- Constraints
  CONSTRAINT parcel_or_deal CHECK (parcel_id IS NOT NULL OR deal_id IS NOT NULL),
  CONSTRAINT valid_status CHECK (parser_status IN ('success','partial','failed','unparsed'))

CREATE INDEX idx_dlf_parcel ON data_library_files(parcel_id);
CREATE INDEX idx_dlf_deal ON data_library_files(deal_id);
```

### 4.3 Corpus-to-File Linkage

Extend `historical_observations`:

```sql
ALTER TABLE historical_observations 
  ADD COLUMN source_file_ids uuid[];

CREATE INDEX idx_ho_source_files ON historical_observations USING GIN (source_file_ids);
```

When the parser writes a corpus row, it records the file_ids that contributed. For the 1,034 already-ingested rows, a one-time backfill SQL reconstructs the linkage from parser logs and folder convention (parcel_id + document_type + observation_date).

### 4.4 Time-Series Query Layer

```typescript
interface TimeSeriesQuery {
  parcel_id: string;
  metrics: TimeSeriesMetric[];   // ['gpr', 'noi', 'occupancy', 'signing_velocity', ...]
  range: { start: Date; end: Date };
  granularity?: 'monthly' | 'quarterly' | 'annual';  // default: monthly
  include_provenance?: boolean;  // default: true
}

interface TimeSeriesResult {
  parcel_id: string;
  series: {
    [metric: string]: Array<{
      observation_date: Date;
      value: number | null;
      tier: 'S1' | 'S2' | 'C1' | 'C2' | 'M1' | 'M2';
      source_file_ids: uuid[];  // when include_provenance
    }>;
  };
  coverage: {
    [metric: string]: {
      observations_count: number;
      date_range: { start: Date; end: Date };
      gaps: Array<{ start: Date; end: Date }>;  // periods with no data
    };
  };
}

async function getAssetTimeSeries(query: TimeSeriesQuery): Promise<TimeSeriesResult>
```

Implementation: standard SQL against `historical_observations`, grouped by metric, ordered by observation_date. Pre-aggregated where reasonable (avoid per-request computation of stable values). Coverage diagnostics surface gaps so UI can render them as visible discontinuities rather than misleading interpolations.


Single endpoint, single page. The UI calls this once and renders everything.

### 5.3 OM extraction routing fix

Per the project memory and prior diagnosis, parse-om already extracts: unit_mix, proforma, investment_thesis, asset_class, units, address, year_built. **This extraction is currently being persisted only for year_built (and only after the just-shipped backfill).** All other fields are returned in the response and discarded.

The fix:
1. Extend the parse-om persistence to write all extracted fields to `property_descriptions` with `layers.om` provenance
2. UPSERT semantics: if `property_descriptions` row exists for parcel_id, MERGE (don't overwrite manual overrides or higher-priority municipal data)
3. Mirror to `historical_observations` denormalized columns where present (year_built, unit_count) for backward-compatible reads

## 6. Stream B — Data Library (Vault + Workbench)

The Data Library is the platform's two-way data store. Files come in, files come out. The user can dump bulk data in, browse what's there, download what they uploaded, and export the corpus to other tools. It is not a one-way ingest sink.

### 6.1 Storage architecture decision

3,744 files, estimated 5-20GB total, requiring:
- Inline viewing (high egress for PDF rendering)
- Scalable to multi-user later
- S3-compatible API (broad SDK support)
- Cost-efficient at small scale

**Decided: Cloudflare R2.** Zero egress fees is the deciding factor for a platform that will display many PDFs inline. S3-compatible API, $0.015/GB/month storage.

**Configuration:**
- Bucket: `jedire-archive` (private by default)
- Access: signed URLs with short expiration (1 hour) for inline viewing
- Optional: Cloudflare CDN in front for cached delivery to repeat viewers
- Lifecycle: no automatic deletion (manual cleanup only)

### 6.2 Upload Modalities

The Data Library supports four upload paths. Each routes through the Intake Orchestrator (§7) for classification and enrichment.

**6.2.1 In-app drag-drop folder upload (primary)**

- Browser File System Access API for folder upload with structure preservation
- Drops entire property folders containing T-12 + rent roll + tax bill + leasing stats + OM
- Folder name → tentative `parcel_id` (orchestrator confirms or prompts)
- Drag-drop zone on `/archive/upload` and on each per-property page
- Progress UI per file with classification result and enrichment status
- File System Access API fallback to multi-file selection when browser doesn't support directory access (Safari, older browsers)

**6.2.2 Drag-drop multi-file (fallback)**

When browser doesn't support folder access, accept multi-file selection. Loses folder structure but preserves bulk-upload capability. Orchestrator handles each file independently and uses filename + content classification.

**6.2.3 Standalone CLI uploader (initial seeding)**

For the 3,744 local archive files, a standalone Node.js script bypasses the browser entirely. Direct R2 upload from the local machine, then registers via the file registration API. See §6.3.

**6.2.4 Programmatic API upload**

`POST /api/v1/archive/files/upload` for integrations. Multipart form upload, returns file_id + intake_job_id. Same orchestration chain applies.

### 6.3 Standalone CLI uploader (for initial 3,744-file seed)

Standalone Node.js script that runs on Leon's local machine, uploads files directly to R2:

```typescript
// scripts/archive-bulk-file-upload.ts

interface UploadConfig {
  rootFolder: string;              // path to 296-property folder root
  parcelIdResolver: (folderPath: string) => string;  // folder → parcel_id
  documentTypeResolver: (filename: string) => DocumentType;
  concurrency: number;             // default: 8 parallel uploads
  resumeFromManifest?: string;     // path to manifest of prior run
}

async function bulkUpload(config: UploadConfig): Promise<UploadReport>
```

**Pipeline per file:**
1. Compute sha256
2. Check `data_library_files` via API for existing row with same sha256 → skip if present (idempotent)
3. Identify parcel_id from folder structure
4. Classify document_type via filename + `classifyDocument()` logic
5. Multipart upload to R2 (chunks of 5MB for files >10MB)
6. POST to backend: `POST /api/v1/archive/files/register` with file metadata + storage_key
7. Backend writes `data_library_files` row, creates `intake_jobs` record, orchestrator picks up
8. Log progress

**Manifest output** (resumable):
```json
{
  "started_at": "2026-05-21T...",
  "completed_at": null,
  "total_files": 3744,
  "uploaded": 1847,
  "skipped_dup": 23,
  "failed": 5,
  "intake_jobs_created": 1847,
  "failures": [
    { "file": "...", "error": "...", "retry_count": 3 }
  ]
}
```

Run can be interrupted and resumed without re-uploading completed files. **Estimated runtime:** 8 parallel uploads × ~2-3MB average × 3,744 files ≈ 30-90 minutes.

### 6.4 File registration endpoint

```
POST /api/v1/archive/files/register

Body:
{
  parcel_id: string | null,    // null when CLI uploader can't resolve from folder
  sha256: string,
  original_filename: string,
  mime_type: string,
  size_bytes: number,
  storage_key: string,
  document_type: string | null   // null when classifier didn't trigger yet
}

Response:
{
  file_id: uuid,
  status: 'registered' | 'duplicate',
  intake_job_id: uuid,           // orchestrator job created/linked
  linked_observations: number    // count of historical_observations rows now linked
}
```

Backend:
1. INSERT into `data_library_files` (handles `ON CONFLICT (sha256) DO NOTHING`)
2. Create `intake_jobs` row in `pending` state
3. Backfill `historical_observations.source_file_ids` for matching rows (if parcel_id present)
4. Return file_id, intake_job_id, and link count

### 6.5 File operations

User-facing operations on individual files or in bulk.

**Per-file operations (file row in browse view or per-property page):**

| Operation | Endpoint | Notes |
|---|---|---|
| View inline | `GET /api/v1/archive/files/:fileId/url` (signed URL) | PDF/XLSX/CSV/TXT inline |
| Download original | `GET /api/v1/archive/files/:fileId/url?download=1` | Browser save dialog, original file as uploaded |
| Download parsed JSON | `GET /api/v1/archive/files/:fileId/parsed` | The parser's extraction output as JSON |
| Replace | `POST /api/v1/archive/files/:fileId/replace` (multipart) | Uploads new version, supersedes old (old kept, marked as superseded) |
| Re-parse | `POST /api/v1/archive/files/:fileId/reparse` | Triggers parser + orchestrator re-run; parser_version updated |
| Delete | `DELETE /api/v1/archive/files/:fileId` | Soft-delete (marks as deleted; file remains in R2 with 30-day TTL for recovery) |
| Move to different property | `POST /api/v1/archive/files/:fileId/move` (body: `{ new_parcel_id }`) | Re-runs enrichment with new parcel context |

**Bulk operations:**

| Operation | Endpoint | Notes |

## 7. Intake Orchestrator + Intake Inbox

The active intelligence layer. When files arrive (via any upload path), the orchestrator coordinates classification → parsing → enrichment, and surfaces user prompts when blocked. This is the difference between a passive vault and an agent that helps you onboard data.

### 7.1 Orchestrator state machine

```
                        ┌─────────────┐
                        │   pending   │  file uploaded, awaiting pickup
                        └──────┬──────┘
                               │
                        ┌──────▼──────┐

                        └──────┬──────┘
                               │

                  │            │            │
            (parse OK)   (parse fail)  (no parser fit)
                  │            │            │
                  │            ▼            ▼
                  │      ┌─────────────────────────┐
                  │      │  blocked_needs_user     │
                  │      │  blocked_reason:        │
                  │      │  parser_failed OR       │
                  │      │  unrecognized_format    │
                  │      └─────────────────────────┘
                  │
            ┌─────▼─────┐

            │           │  1. Other files for same parcel
            │           │  2. Municipal API by address
            │           │  3. Web search by name + address
            │           │  4. Google Places by name + address
            └─────┬─────┘
                  │
        ┌─────────┼─────────┐

   (resolved)  (ambig)  (no_match)
        │         │         │

```

### 7.2 Prompting behavior

**Moderate prompting policy (decided):** Agent tries all four resolution sources (other docs → municipal API → web search → Google Places) before prompting the user. Does not give up on first failure; does not retry indefinitely. When it has exhausted its capability surface and is genuinely stuck, surfaces a specific contextual question.

**Block reasons and corresponding prompts:**

| Block reason | When it fires | Prompt to user |
|---|---|---|
| `parser_failed` | Parser threw error or returned no data | "I couldn't parse this file. It looks like a [classified type] but the format isn't matching. Can you confirm what type of document this is?" |
| `unrecognized_format` | Classifier couldn't determine type | "What kind of document is this? [T-12 / Rent Roll / Tax Bill / OM / Leasing Stats / Other]" |
| `no_property_identifier` | Parsed but no address/name found in document | "I found [summary of extracted data]. Can you tell me the property name or address?" |
| `ambiguous_match` | Resolver found multiple candidates | "There are N properties matching this. Which one? [list]" |
| `municipal_lookup_failed` | Resolved name+address but county API returned no record | "I resolved this to [property], but I couldn't find it in [county] records. Is the address correct?" |
| `multiple_municipal_matches` | County API returned >1 record for the address | "[N] municipal records match this address. Which parcel? [list]" |
| `manual_review_required` | Post-user-input attempts still failed | "I'm marking this file as unresolved. You can retry or mark it as ignored." |

The prompt is **specific to what's known and what's missing**. Not a generic "agent failed" message. Each prompt has actionable user options.

### 7.3 Re-trigger logic

When user submits input via Intake Inbox, the orchestrator does NOT re-run from scratch. It resumes from the blocked point with the new input merged into the job's `user_input` field. This:
- Saves API calls (no redundant municipal lookups)
- Saves time (no re-parsing)
- Preserves the enrichment chain in `enrichment_log` for audit

If the resumed attempt still fails, the job blocks again with a different reason or escalates to `manual_review_required`.

### 7.5 Enrichment chain detail

When orchestrator enters `enriching` state, the chain runs in order:

**Step 1 — Other documents for same parcel.** If parcel_id is already known from another file in the same folder, use it directly. Cheapest resolution path.

**Step 2 — Municipal API by address.** If address was extracted, look up in the appropriate county property appraiser API. Returns canonical year_built, sqft, lot_size, owner, parcel #. Stream A gets populated.

**Step 3 — Web search by name + address.** If name and address (or partial) are known, web-search for the property. Looks for: property website, ApartmentRatings, ApartmentList, news articles, developer announcements. Web findings inform the narrative field and supplement amenity / asset class detection.

**Step 4 — Google Places by name + address.** Returns lat/long, business hours, photos, reviews, place_id. Reviews are an important secondary signal — concession trajectory often shows up in reviews before it shows up in market reports.

After each step, the orchestrator updates `property_descriptions` with appropriate `LayeredValue` provenance and logs the step in `enrichment_log`. The user sees the full chain in the Intake Inbox when the job is in any blocked state.

### 7.6 Bulk seed backfill

For the 3,744 archive files already uploaded via CLI (pre-orchestrator), a one-time backfill creates retroactive IntakeJob records. Each job runs through the orchestrator. Many will resolve cleanly (parcel_id already in folder structure, classification already correct). Some will block in the Intake Inbox — which is the right outcome, those are files with bad/missing identification anyway.

## 14. Addendum — Cohort Rollup View

### 14.1 Purpose

Per-Property Visibility surfaces single-property data. Cohort Rollup surfaces aggregate data across multiple properties grouped by user-selected dimensions. The two views are complementary: drill into a single property (per-property page) or zoom out to see how a segment is trending (Cohort Rollup).

Bloomberg has a sector index for every category and lets users slice equities by any combination of (industry × geography × market cap × style). This is the same idea applied to RE — cohort indices selectable by any combination of (product type × asset class × MSA × vintage), with T-12 financial rollups normalized per-unit for comparability.

The decision per Section 14 is **multi-dimensional grid as the primary cohort dimension** — not fixed cuts. Users define their cohort by selecting which dimensions to facet by and which to filter on; the system aggregates accordingly with cohort-size confidence indicators.

### 14.2 Cohort Dimensions

Available dimensions for cohort definition:

| Dimension | Values | Source field |
|---|---|---|
| **Product type** | Garden, Mid-rise, High-rise, Townhome, SFR | `property_descriptions.stories_band` + structural inference |
| **Asset class** | A, A-, B+, B, B-, C+, C | `property_descriptions.asset_class` |
| **MSA** | All resolved MSAs in the corpus | `property_descriptions.msa` |
| **Submarket** | Submarkets within MSA (deeper drill) | `property_descriptions.submarket` |
| **Vintage band** | pre-1980, 1980-2000, 2000-2015, 2015+ | derived from `property_descriptions.year_built` |
| **Unit count band** | <50, 50-200, 200+ | derived from `property_descriptions.unit_count` |
| **Construction type** | Wood frame, Concrete, Steel, Mixed | `property_descriptions.construction_type` |
| **Parking type** | Surface, Structured, Wrap, Podium | `property_descriptions.parking_type` |

Each dimension can be used as:
- **Facet** — splits the rollup grid (e.g., "facet by MSA" produces one row per MSA)
- **Filter** — constrains cohort membership (e.g., "filter to Class B only")
- **Drill-through** — click into a cohort cell to navigate into deeper detail

A single cohort selection might be: `facet=[product_type, msa], filter={asset_class: 'B', vintage_band: ['2000-2015','2015+']}`. The result: one rollup row per (product_type, msa) cell, only including Class B properties built 2000+.

### 14.3 Metrics Displayed

Per cohort × year cell, the rollup shows:

**Revenue (per-unit, annual):**
- GPR/unit
- Loss-to-lease/unit
- Vacancy/unit
- Concessions/unit
- Other income/unit
- EGI/unit (Effective Gross Income)

**Expenses (per-unit, annual):**
- Real estate tax/unit (often the dominant trajectory)
- Insurance/unit (the FL story right now — material in any FL cohort)
- Repairs & Maintenance/unit
- Management fee/unit
- Utilities/unit (split by metering type when known)
- Marketing & advertising/unit
- Payroll/unit
- Replacement reserve/unit
- Total OpEx/unit

**Performance:**
- NOI/unit
- Operating margin %
- Occupancy %
- CAGR per line across the displayed period

**Sale events (when applicable):**
- Cap rate at sale (median + IQR across cohort sales in period)
- Price/unit at sale
- N of sales in cohort during period

**Cohort metadata:**
- N (number of contributing properties)
- Confidence indicator (based on N + time-series completeness)
- Coverage (% of contributing properties with data in the displayed year)

### 14.4 Aggregation Method

Default: **median** (resistant to outliers; appropriate for skewed financial distributions where one large property could dominate a mean).

User-selectable aggregation modes:
- Median (default)
- Mean
- Unit-count-weighted mean (for per-unit metrics where unit-weighted is more representative)
- Distribution view (render P25 / P50 / P75 instead of single number)

### 14.5 Cohort Size Guardrails

Statistical confidence depends on N. The view enforces visibility of cohort size so users don't over-interpret thin cuts:

| N | Treatment |
|---|---|
| ≥ 12 | Full aggregation, all metrics shown, high-confidence indicator (green) |
| 6 ≤ N < 12 | Aggregation shown with medium-confidence indicator (yellow) |
| 3 ≤ N < 6 | Aggregation shown with low-confidence indicator (red), individual properties listed below the aggregate |
| < 3 | Aggregation suppressed; individual properties listed instead with a "cohort too small to aggregate" message |

This prevents the failure mode of users drilling into Garden × Class B+ × Tampa × pre-1980 × Wood-frame, getting N=2, and treating the resulting average as a market signal.

### 14.6 Time Dimension

The rollup is yearly by default:
- Annual T-12 rollups (typically 5-year window: 2020-2024)
- Optional T-12 trailing window (current date - 12 months)
- Optional custom range (user-picked start/end)

### 14.7 Comparison Mode

Side-by-side comparison of cohorts:

```
Garden × Class B × Atlanta        │   Mid-rise × Class B × Atlanta
N = 84 · Confidence: HIGH         │   N = 23 · Confidence: HIGH
                                  │
GPR/unit   $14,820 → $18,120      │   GPR/unit   $19,240 → $24,180
NOI/unit   $7,920 → $9,480        │   NOI/unit   $10,840 → $12,920
OpEx/unit  $5,720 → $7,330        │   OpEx/unit  $7,440 → $9,580
                                  │
Δ NOI/unit Garden:    +19.7%      │   Δ NOI/unit Mid-rise:   +19.2%
Δ OpEx Garden:        +28.1%      │   Δ OpEx Mid-rise:       +28.8%
Cap @ sale (N=12):    5.7%        │   Cap @ sale (N=4):      5.3%
```

At-a-glance: Garden vs Mid-rise per-unit deltas, which segment is widening/narrowing in spread, where OpEx growth is concentrated, cap rate differential between product types within the same submarket.

Also supports cohort vs MSA index ("how does Garden Class B Atlanta compare to all Atlanta multifamily?") and cohort vs national norm (where data permits).

### 14.8 Export

- Excel export of the rollup table (matches displayed shape)
- Underlying property list as a separate sheet (N contributing parcel_ids with names + links to per-property pages)

### 14.9 Multi-Dimensional Cohort Query Layer

New service at `src/services/cohort-rollup/`:

```typescript
type CohortDimension = 
  | 'product_type' | 'asset_class' | 'msa' | 'submarket' 
  | 'vintage_band' | 'unit_count_band' | 'construction_type' | 'parking_type';

type CohortMetric = 
  | 'gpr' | 'egi' | 'noi' | 'opex_total' 
  | 'opex_tax' | 'opex_insurance' | 'opex_rm' | 'opex_mgmt' 
  | 'opex_utilities' | 'opex_marketing' | 'opex_payroll' | 'opex_reserves'
  | 'occupancy' | 'cap_rate_at_sale';

interface CohortQuery {
  facets: CohortDimension[];
  filters: Partial<Record<CohortDimension, string[]>>;
  metrics: CohortMetric[];
  range: { start: Date; end: Date };
  granularity?: 'annual' | 't12_trailing';
  aggregation?: 'median' | 'mean' | 'weighted_mean' | 'distribution';
}

interface CohortResult {
  cells: Array<{
    facet_values: Record<CohortDimension, string>;
    n: number;
    confidence: 'high' | 'medium' | 'low' | 'suppressed';
    contributing_parcel_ids: string[];
    metrics: Record<CohortMetric, YearlyValue[]>;
  }>;
}

async function getCohortRollup(query: CohortQuery): Promise<CohortResult>
```

SQL-level aggregation: medians via `PERCENTILE_CONT(0.5) WITHIN GROUP`, IQR via `PERCENTILE_CONT(0.25)` and `PERCENTILE_CONT(0.75)`, distribution snapshots stored as pre-aggregated arrays where reasonable.

### 14.13 Strategic Value

This view delivers analytical capability competitors don't have:

- **CoStar** publishes submarket averages but doesn't expose product-type cross-cuts with full OpEx line-item visibility
- **RealPage / Yardi Matrix** have data but at proprietary aggregate level — not exposed as a user-driven analytical tool with arbitrary dimension selection
- **Bloomberg** does this beautifully for equities but doesn't exist for RE at this granularity

Combined with comp ProForma reconstruction and per-property visibility, Cohort Rollup is what makes the "Bloomberg Terminal for RE" positioning real. Multi-dimensional segment analysis with line-item granularity is exactly what equity traders use Bloomberg for. This is the surface that demos best — "show me how Garden Class B Atlanta OpEx has trended over 5 years, with insurance broken out" is a question every multifamily operator wants answered and no platform answers well today.
