# 2026-03-07 - Intelligence Context Engine Phase 0

## 🚀 Major Work: Intelligence Layer Foundation

### Context
Leon asked me to analyze the data architecture for agent training after I created the M28 (Google Workspace Integration) architecture doc. His concern: "the architecture we need to train the agents and modules with all the data sources."

### Problem Identified
Current agents are **stateless** - they get fresh context on each run but don't "learn" from data. With GWS adding 100-300 docs/day (Gmail OMs, Drive reports, Sheets comps), we'll have data explosion without intelligence.

**Current flow:**
- Broker email → Document Parser → stores in `deal_capsules.metadata` JSONB
- Agent runs → queries Data Library by city/tags (keyword match)
- **No connection between the OM that started this deal and the agent's context**

### Solution: Intelligence Context Engine (L2.5)

Created comprehensive architecture doc: `/home/leon/clawd/docs/gws-data-integration-analysis.md`

**Key insight:** Add unified data layer between API Gateway and Service Layer with:
1. **Unified Document Registry** - single source for all docs with vector embeddings
2. **Document Relationships** - track supersedes/supplements/contradicts
3. **Schema Normalization** - normalize fields across sources
4. **Agent Task Learnings** - historical memory for pattern recognition

### Implementation Started

Built Phase 0 foundation (committed + pushed to `jedire-agents-implementation`):

**Migrations created:**
- `081_intelligence_layer_foundation.sql` - pgvector + unified_documents + doc_relationships
- `082_schema_normalization.sql` - field_mappings table with 25+ default mappings
- `083_agent_task_learnings.sql` - agent_task_learnings + agent_patterns tables

**Service layer:**
- `backend/src/services/intelligence-context.service.ts` - Full IntelligenceContextService with:
  - `indexDocument()` - Add docs to unified registry
  - `semanticSearch()` - Vector similarity search across all docs
  - `getAgentContext()` - Comprehensive context for agent execution
  - `createDocumentRelationship()` - Link related documents
  - `logAgentLearning()` - Store task execution for pattern learning

**Key Features:**
- Semantic search using pgvector (OpenAI text-embedding-3-small, 1536 dim)
- Cross-source linking (Gmail OM + Drive T12 + Sheets comps)
- Field normalization (CBRE "NOI" vs MM "Net Operating Income" → `noi_annual`)
- Confidence scoring + validation status
- Data quality flags (outlier detection)
- Historical pattern learning

**Example agent upgrade:**
- **Before:** Gets 2 generic impact fee CSVs from Data Library
- **After:** Gets OM + market reports + 5 comparable deals with similarity scores + historical learnings ("midrise Atlanta urban core: avg 195d timeline, 35% variance rate, 92% success")

### Built:

**1. Admin Intelligence Dashboard** (`/admin/intelligence`)
- Document registry stats (total, embedded, validated, flagged)
- Source breakdown (Data Library, Gmail, Drive, Sheets)
- Agent learning metrics (30d approval rates, confidence, execution time)
- Agent breakdown by type
- Document types distribution
- Data quality alerts + embedding backlog
- Cost tracking (OpenAI tokens)
- API endpoints: `/api/v1/intelligence/stats`, `/documents/pending`, `/documents/flagged`, `/patterns`

**2. Document Categorization System** (Migration 084)
- **24 document categories** organized by purpose:
  - Financial: OM, T12, RENT_ROLL, PROFORMA
  - Property: APPRAISAL, SURVEY, INSPECTION, ENV_REPORT
  - Zoning: ZONING_CODE, ZONING_LETTER, SITE_PLAN, ENTITLEMENT_APP
  - Market: MARKET_REPORT, COMP_SHEET, RENT_SURVEY, DEMO_REPORT
  - Construction: COST_ESTIMATE, IMPACT_FEES, PERMIT_TIMELINE
  - Legal: PSA, TITLE_REPORT, LEASE_ABSTRACT
  - Asset Mgmt: BUDGET, VARIANCE_REPORT, CAPEX_PLAN, TENANT_LEDGER
- **Module routing**: Each category knows which modules consume it (M01, M09, M15, etc.)
- **Purpose tags**: financial, zoning, market, competitive, operational
- **Deal stages**: lead, analysis, underwriting, due_diligence, closing, asset_management
- **Agent priority**: 1=critical, 5=normal, 10=low
- **Module data requirements**: Defines what each module needs (required vs optional)

**3. Archive Deals System** (Migration 085)
- `archived_deals` table with outcome tracking
- Archive reasons: closed_purchased, closed_sold, passed_pricing, passed_zoning, lost_to_competition
- **Actual vs Projected** tracking: NOI, cap rate, IRR
- **Lessons learned** capture: what worked, what didn't, would we do again?
- **Historical outcomes** table for agent learning
- **Comparable finder**: Find similar past deals by city/type
- Document retention + storage paths
- Training data flags (use_for_training, comparable_to_future_deals)

**4. Document Router Service**
- **Auto-categorization**: Pattern matching on title/filename/content
- **Module routing**: Determines which modules can use each document
- **Data quality scoring**: 
  - Has all required documents?
  - Missing optional documents?
  - Overall score (0-1)
  - Ready vs blocked modules
- **getDealDataQuality()**: Shows which modules are ready to run
- Example: "M09 ProForma: 87% quality (has T12 + RENT_ROLL, missing BUDGET)"

### How It Works:

**Data Flow Example:**
```
1. Gmail: Broker emails OM
2. Parser: Extracts text + { noi_annual, unit_count, cap_rate }
3. Intelligence: Indexes in unified_documents + generates embedding
4. Router: Auto-categorizes as "OM" → routes to [M01, M09, M15, M22]
5. M09 Agent: Queries getDocumentsForModule('M09')
   - Gets: OM (helpful but not required)
   - Data quality: 60% (missing T12, RENT_ROLL)
   - Result: "Need more data"
6. User uploads T12 + RENT_ROLL
7. M09 Agent: Data quality: 100% → "87% confidence projection"
8. Deal closes → Archived with outcomes
9. Future deals: "3 similar Atlanta deals: avg 7.8% IRR, 195d timeline"
```

**Benefits:**
- Agents know what data they have vs need
- Modules show "ready" or "blocked" status
- Automatic routing (no manual wiring)
- Historical learning from outcomes
- Comparable deal discovery
