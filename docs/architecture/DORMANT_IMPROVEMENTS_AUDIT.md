# Dormant Seeder Improvements Audit

**Phase:** 0 (read-only audit)
**Date:** 2026-05-08
**Trigger:** Item 3 post-fix finding — proforma-seeder is write-once by design
(`ensureDealAssumptionsSeeded` is a no-op when `year1` already exists).
Improvements shipped after initial seeding are invisible to existing deals unless
a `forceReseed` or backfill script is explicitly run.

**Scope:** Every git commit that touched `proforma-seeder.service.ts` since project
start, plus every DB-persisted derived value with a TTL/cache pattern that could
serve stale outputs generated by pre-fix logic.

**Phase 1** (build backfills for BACKFILL_NEEDED findings) is a separate greenlight
after Leon reviews this doc.

### S1-01 — Custom opex inflation filter (`e0258abb`, Apr 20 2026)

**Commit:** `e0258abbd1df61b86a26f36f24e91c5941f5ae9a`
**Description:** `fix(proforma-seeder): filter out revenue/rollup/below-the-line items from custom opex`

**What changed:** The T12 parser captures unrecognized GL rows as `customLineItems`.
Pre-fix, the seeder summed ALL of them into `total_opex_resolved`, including revenue
lines (GPR, EGI, collected rent), rollup subtotals (Total Income, NOI), and
below-the-line items (debt service, depreciation). This caused massively inflated
opex — the commit message cites "$43M on a $6M GPR property" — and negative NOI.

`EXCLUDE_FROM_CUSTOM_OPEX` regex array added (30 patterns) to filter before
`custom_opex_*` accumulation.

**Backfill evidence:** None found. No backfill script, no reparse mention in commit.
Any deal seeded between initial creation (Apr 12) and Apr 20 with T12 custom line
items may carry inflated opex in `year1.total_opex_resolved`.

**Current DB state (both live deals):** `total_opex_resolved` is null at the top level
of `year1` JSONB — this field may be computed at query time rather than stored as a
top-level key, or both deals were reseeded after Apr 20. Task #435 (Apr 28) triggered
a reparse of 464 Bishop; Sentosa Epperson's reseed path is unconfirmed.

**Classification: BACKFILL_NEEDED (VERIFY)**

Qualifying criteria: any deal with T12 custom line items that was first seeded before
Apr 20 2026 and not explicitly reseeded since. Heuristic: look for `custom_opex_*`
keys in `year1` JSONB whose labels match the now-excluded patterns. Effort: **S**
(add `EXCLUDE_FROM_CUSTOM_OPEX` filter to forceReseed path; run
`reseed-deal.ts` per affected deal).

**S1-01 — Custom opex filter (e0258abb, Apr 20)**

Run this query to detect any live deal that may still carry pre-fix inflated opex:

```sql
SELECT d.name, da.deal_id,
       jsonb_object_keys(da.year1) AS year1_key
```

## Appendix — Seeder Guard Reference

`ensureDealAssumptionsSeeded` (`proforma-seeder.service.ts:853`):

```typescript
const hasExistingSeed = existing.rows[0]?.year1 != null;
if (hasExistingSeed && !opts.forceReseed) {
  return { seeded: false, skipped: true, reason: 'year1 already seeded' };
}
```

Any improvement to `buildSeed()` is invisible to existing deals unless the caller
passes `{ forceReseed: true }` or a backfill script calls `seedProFormaYear1`
directly. This is the systemic root cause of all BACKFILL_NEEDED findings above.

The `reseed-deal.ts` script introduced in `0af3c1c7` is the canonical one-deal
reseed tool. The `reseed-other-income.ts` script from Item 3 is the template for
multi-deal batch backfills.
