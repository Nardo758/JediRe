## Decision: DO NOT DROP

### Reasons to keep

1. **External share content store** — `capsule_external_shares` FKs to `deal_capsules.id`. Dropping `deal_capsules` would cascade-delete all external shares and break `/share/:shortcode` recipient links permanently.

2. **Agent context buffer** — The `deal_data`, `platform_intel`, `user_adjustments`, and `module_outputs` JSONB columns provide a stable, versioned snapshot that AI agents read without touching the live `deals` table.

3. **Decoupling boundary** — `deal_capsules` serves as the publishing boundary between the internal deal workspace (Drizzle-managed `deals` + `deal_assumptions`) and the external-facing share surface. This separation is intentional.

4. **No `deal_id` column** — The linkage via `deal_data->>'deal_id'` (JSON field) is unconventional but is the live production contract. Changing it requires a migration and audit of all consumers.

## `?token=` query-param limitation

All share URLs generated by the platform use **path params**, not query params:

- `/share/:shortcode` — shortcode-based recipient links (current standard)
- `/capsule-link/:accessToken` — legacy access-token links (still supported)

The URL pattern `?token=abc` (query param) is **never generated** by the system. `DealDetailPage` detects recipient mode via `useParams()` (`shortcode` or `token` path segments), not `useSearchParams()`. A URL like `/deals/:dealId/detail?token=abc` would NOT be recognized as a recipient session — it would render the owner view.

**Implication:** No code change needed. If a future share mechanism uses query params, add `useSearchParams()` detection in `DealDetailPage` alongside the existing `useParams()` check.
